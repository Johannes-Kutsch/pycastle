# OpenCode Timeout Exhaustion Uses Unknown-Reset Usage-Limit Path

OpenCode Go usage exhaustion can surface as exhausted idle-timeout retries rather than as a structured provider usage-limit event. Pycastle therefore preserves the existing in-invocation timeout retry budget, then treats final exhausted OpenCode idle-timeout retries as an unknown-reset usage-limit event with no raw provider message, so the existing service fallback and sleep decision path handles it and the final status follows the usage-limit path. The `<service>_minimum_unknown_reset_duration_hours` config fields are generic minimum wake-estimation inputs for resetless usage limits, but the idle-timeout-to-usage-limit transformation is OpenCode-specific; Codex and Claude idle-timeout retries remain ordinary timeouts.
