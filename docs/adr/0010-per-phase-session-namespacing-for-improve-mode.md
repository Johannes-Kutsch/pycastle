# Per-phase session namespacing for improve mode

The improve agent's four phases (`01-scan`, `02-prd`, `03-issues`, `04-no-candidate-report`) currently share a single Claude conversation: a single **derived session UUID** keyed only on `(role, worktree_path)` resumes phase N onto phase N-1's growing transcript, and the per-role session signal directory `.pycastle-session/improve/` is treated as one bucket. The shared transcript was deliberate — phase 02 was meant to reuse phase 01's codebase exploration without re-scanning, and phase 03 to reuse phase 02's PRD draft without re-fetching — but in practice the conversation grows monotonically across the run, the agent pays cache-warming cost on every turn against a transcript that already contains the full scan and the full PRD body, and phase 03's vertical-slice work touches none of phase 02's drafting prose. We split phase 03 onto its own Claude session by introducing a **session namespace** dimension — a small string folded into both `derived_session_uuid` and the per-role session signal directory path — and partition the four phases into two namespaces: `main` for phases 01/02 (and the alternative phase 04 path), `issues` for phase 03. Phase 03 starts fresh with no prior conversation; it receives the parent PRD's number, title, and body inlined into its prompt as `{{ISSUE_NUMBER}}` / `{{ISSUE_TITLE}}` / `{{ISSUE_BODY}}` / `{{ISSUE_COMMENTS}}` placeholders (mirroring the `# CONTEXT` block of `implement-prompt.md`), and re-explores `CONTEXT.md` and the area's ADRs at the start of the conversation.

The PRD issue number itself crosses the namespace boundary via the existing agent-output protocol rather than a new persistence file: phase 02's prompt now emits the parent issue as JSON-form `<issue>{"number": N, "labels": []}</issue>`, which the IMPROVE-role parser in `process_stream` recognises (alongside the bare-integer `<issue>N</issue>` tags phase 03 emits per filed sub-issue) and surfaces as `IssueOutput(number=N, labels=[])` — the same structured return type already used by the preflight-issue agent. `improve_phase` captures `output.number` into a local variable and uses it to drive a fresh `GithubService.get_issue` fetch when assembling phase 03's prompt args. No `_prd_issue` persistence file is added.

The choice not to persist trades a rare crash window for plumbing simplicity: if the orchestrator process dies after phase 02 writes `_phase_progress = "02-prd"` but before phase 03's `_phase_in_flight` is written, the next invocation has the progress file but no in-memory PRD number and no way to recover it. `improve_phase` detects this orphan condition (`last_id == "02-prd"` AND `in_flight_id != "03-issues"`) at the top of the loop, clears `_phase_progress`, and restarts the cycle from phase 01 — the parent PRD already on GitHub becomes a dead issue (no triage label, invisible to the implement-mode query, manual cleanup required). A crash *during* phase 03 (after the role prompt with PRD body has reached Claude) is recoverable with no orphan: the runner sees the `issues/` namespace dir is non-empty → Resume → continuation prompt only → Claude reattaches to its session with the PRD already in turn 1 of conversation history. Phase 02's prompt-level dedup check is concurrently dropped — its sole purpose was duplicate-protection on Resume of an already-completed phase 02, but with the new orphan-reset semantics every phase-02 invocation either runs to completion or is wiped and restarted from phase 01, so deduplication has no remaining caller.

## Considered Options

- **Status quo (single shared Claude conversation across all four phases).** Rejected: the cost is the conversation we're trying to avoid. Phase 02 is a one-shot PRD draft and never benefits from phase 01's transcript beyond the candidate identity; phase 03 actively wants to re-scan the area against fresh CONTEXT.md vocabulary rather than inheriting whatever shorthand emerged in phases 01–02. The only real saving was "don't re-fetch the PRD body" — replaced by host-side `GithubService.get_issue` plus prompt-args injection, which is bounded (PRD body + a few hundred lines of CONTEXT.md/ADRs) versus the unbounded shared transcript.
- **`force_fresh: bool` flag on `RunRequest`.** Rejected: the runner's existing `decide_agent_run_kind(role, session_dir_present)` is purely data-driven; an override flag short-circuits that contract and introduces a parallel control path. It also still requires moving `_phase_progress` and `_phase_in_flight` out of the role session dir (they would otherwise be wiped by `wipe-before-Fresh` on phase 03's forced-fresh entry), which the chosen design avoids by namespacing the wipe scope.
- **Expand `derived_session_uuid` to fold a phase id into the UUID without partitioning the role session dir.** Rejected: a different UUID alone tells Claude to start fresh, but the runner's `has_resumable_session(role_dir)` check would still see the dir non-empty (from phase 01–02 signals) and pick `Resume`, which would send the continuation prompt to a session UUID Claude has never seen → empty-context first turn → deterministic agent confusion. UUID partitioning and signal-dir partitioning have to move together.
- **Per-phase session subdir keyed by phase id (every phase gets its own namespace).** Rejected: phases 01 → 02 genuinely benefit from a shared transcript — phase 02 reuses phase 01's candidate identity, AFK-safety reasoning, and design-tree grilling for PRD authoring with no extra prompting. Splitting them would force phase 02 to re-derive candidate context, defeating the purpose of selecting it in phase 01.
- **Per-group session subdir** — the chosen design. Two groups: `main` (phases 01/02/04) and `issues` (phase 03). `RunRequest` gains `session_namespace: str = ""`, threaded into both `derived_session_uuid` and `agent_runner`'s `role_session_dir` computation. Cross-phase tracking files (`_phase_progress`, `_phase_in_flight`, the future `_prd_issue` if persistence is ever introduced) stay at the role-level dir, sibling to the namespace subdirs, untouched by per-namespace `wipe-before-Fresh`. Default empty namespace preserves existing behaviour for every other agent role byte-for-byte.
- **Persist the PRD issue number to disk** at `.pycastle-session/improve/_prd_issue` after phase 02 completes; read on phase 03 entry to drive prompt-arg injection. Rejected: the only recoverable state it adds is the gap between phase 02's progress write and phase 03's first agent invocation — a window of sub-second orchestrator code with no I/O. The orphan-reset alternative (crash in that window → fresh cycle, dead PRD on GitHub) costs nothing in normal operation and, in the rare crash case, produces one orphan PRD with no `ready-for-agent` label that no automation will ever pick up. Manual cleanup is acceptable; persistence machinery for that window is not.
- **Keep phase 02's prompt-level dedup check (the `gh issue list --search "[improve-{SID}] in:title"` query at the top of `02-prd.md`).** Rejected concurrently with this ADR: the dedup was load-bearing only when phase 02 could legally re-run on an already-completed cycle (Resume path through phase 02). With orphan-reset, every phase 02 invocation either runs to completion or is wiped + restarted — deduplication has no remaining caller. Phase 03 and phase 04 dedup checks remain because they file *multiple* issues per phase and Resume mid-phase is real.
- **Extend `IssueOutput` with a phase tag, or introduce a new `ImprovePrdOutput(number)` dataclass.** Rejected in favour of reusing `IssueOutput` directly: the parser already differentiates phase 02 from phase 03 by tag *format* — phase 02 emits JSON-form `<issue>{...}</issue>`, phase 03 emits bare-integer `<issue>N</issue>` — and the `IssueOutput` shape (number + labels) carries everything `improve_phase` needs. A new dataclass adds a union variant for naming clarity only. The parser carries a one-line comment recording the convention.
- **Re-fetch PRD body via the agent inside phase 03** (passing only the issue number as a placeholder, agent runs `gh issue view`). Rejected: introduces a deterministic extra agent turn at the start of every phase 03 just to re-derive a value the host already has, and means phase 03 cannot run if `gh` is mis-authenticated inside the container even though the host already authenticated to file the PRD. Host-side fetch into prompt args mirrors the established `implement-prompt` / `review-prompt` pattern.
- **Cache PRD title and body to disk** alongside the number, so phase 03 resume after a long pause does not re-hit the GitHub API. Rejected: the issue body might have been edited between phase 02 filing and phase 03 running (a human could have appended clarifications); a stale cache would feed lying placeholders into the agent. Re-fetching every phase 03 entry costs one HTTP round-trip and is correct by construction.

## Consequences

- `RunRequest` gains `session_namespace: str = ""`. `agent_runner.run` computes `role_session_dir = mount_path / ".pycastle-session" / role.value / namespace` (joined cleanly when namespace is empty so non-improve roles see no path change). `derived_session_uuid(role, worktree_path, namespace="")` folds the namespace into the UUID derivation chain `uuid.uuid5(uuid.uuid5(NAMESPACE_DNS, "pycastle.<role>.<namespace>"), worktree)` only when namespace is non-empty — empty namespace produces byte-identical UUIDs to today's two-arg call.
- `improve_phase` selects `namespace = "issues"` for `03-issues.md` and `namespace = "main"` for every other phase, threaded through the `RunRequest`. Phase 04 stays in `main` because it only fires on the NO-CANDIDATE branch from phase 01 (never co-runs with phases 02/03), and its access to phase 01's shortlist-rejection reasoning is exactly what the no-candidate report needs.
- `_phase_progress` and `_phase_in_flight` continue to live at `.pycastle-session/improve/` (role-level), sibling to the new `main/` and `issues/` subdirs. The `shutil.rmtree(role_session_dir, ignore_errors=True)` at the end of a successful improve cycle still operates at the role-level path, wiping every namespace and tracking file together. `wipe-before-Fresh` (`shutil.rmtree(role_session_dir)` at the start of a Fresh agent run) operates only on the per-namespace path the runner is about to use, leaving sibling namespaces and the role-level tracking files intact.
- `IssueOutput(labels, number)` is now produced by the IMPROVE-role parser when a JSON-form `<issue>{"number": N, "labels": [...]}</issue>` tag is present alongside `<promise>COMPLETE</promise>` in the agent output stream. Bare-integer `<issue>N</issue>` tags (phase 03's per-sub-issue listings) continue to be ignored — phase 03 returns `CompletionOutput`. The differentiation is by tag format; a one-line comment in `process_stream` records the convention so a future change does not accidentally make phase 03 emit JSON form and silently flip its return type.
- `_ImproveDeps` gains `gh_svc: GithubService`. `GithubService` gains `get_issue(number: int) -> dict[str, str]` returning `{"title", "body"}` in a single GET (as opposed to the existing `get_issue_title` plus a separate body fetch). `improve_phase` calls `gh_svc.get_issue(prd_number)` and `gh_svc.get_issue_comments(prd_number)` once per phase 03 entry — both Fresh and Resume — to assemble the `ISSUE_*` prompt args. On Resume the assembled args are unused (the runner sends the continuation prompt only) but assembling them keeps the call site unconditional.
- Phase 02's prompt no longer carries the `## Dedup check` section. Phase 02 emits `<issue>{"number": N, "labels": []}</issue>` (JSON form) instead of `<issue>N</issue>`. Phase 03's prompt is rewritten to mirror `implement-prompt.md`'s structure: a `# CONTEXT` block with `{{ISSUE_NUMBER}}` / `{{ISSUE_TITLE}}` / `{{ISSUE_BODY}}` / `{{ISSUE_COMMENTS}}` at the top, an explicit `## 1. Explore` step (read `CONTEXT.md` / `CONTEXT-MAP.md`, skim relevant ADRs, read modules named in the PRD) replacing the "Reuse the codebase exploration… don't re-scan" line, and the existing dedup check, CONTEXT.md update detection, AFK-only constraint, vertical-slice rules, self-quiz, sub-issue body template, sub-issue registration, and `<issue>N</issue>`/`<promise>COMPLETE</promise>` output preserved verbatim.
- Orphan-reset condition in `improve_phase`: at loop entry, if `last_id == "02-prd"` AND `in_flight_id != "03-issues"`, the in-memory `prd_number` was lost (process restart) and there is no recovery path. The orchestrator unlinks `_phase_progress`, sets `last_id = None`, and the loop restarts from phase 01. The orphan PRD on GitHub keeps the `[improve-<old-sid>]` magic title prefix and no triage label — invisible to `cfg.issue_label` queries, requires manual `gh issue close` to clean up. A future garbage-collect sweep ("close PRDs whose `<short-sid>` no longer matches any active improve session") is reasonable follow-up work but not part of this change.
- `send_role_prompt_on_resume` logic in `improve_phase` is unchanged. For phase 03's first entry `last_id == "02-prd"` (truthy) and `in_flight_id != "03-issues"`, so the flag computes to `True`. The runner sees the `issues/` namespace dir does not exist → `FRESH` → sends the full role prompt regardless of the flag. The flag becomes effectively dead code on phase 03's first entry but stays correct on phase 03 mid-flight Resume (where `in_flight_id == "03-issues"` flips the flag to `False`, runner sees `issues/` non-empty → `Resume` → continuation prompt only).
- The **prompt-shape contract** mapping in CONTEXT.md gains one cell: phase 03's first entry is **Fresh (new namespace)** — role prompt only, with `--session-id` pointing at the namespaced UUID. **Resume (new phase)** continues to apply for phase 02 after phase 01-picked, and phase 04 after phase 01-no-candidate, both within the `main` namespace.
- ADR 0006's closing consequence — "improve mode (#408) builds multi-prompt-within-one-run on top of this primitive without modifying the resume infrastructure" — is superseded by this ADR. The resume primitive is unchanged in shape; the namespace dimension is additive (default-empty parameter on `derived_session_uuid` and `RunRequest`), and the single-prompt agent roles (Implementer, Reviewer, Merger, Planner, preflight-issue) remain byte-identical because they pass the default empty namespace.
- Blocked by issue #548 (`PromptRenderer` infrastructure with scoped placeholder tiers, ADR 0009). Phase 03's new template references `{{ISSUE_NUMBER}}` / `{{ISSUE_TITLE}}` / `{{ISSUE_BODY}}` / `{{ISSUE_COMMENTS}}`, which today belong to the `PER_ISSUE` scope (used by `IMPLEMENT` and `REVIEW`). The renderer's scope registry must accept the new template variant against a scope that includes both the existing `IMPROVE_SESSION` placeholder set (`IMPROVE_SHORT_SID`) and the issue placeholders. The simplest extension is a new scope (e.g. `IMPROVE_ISSUES`) carrying the union; alternatives include reusing `PER_ISSUE` and adding `IMPROVE_SHORT_SID` to it, or expanding `IMPROVE_SESSION` with the issue placeholders. The choice is internal to the renderer slice and not load-bearing for this ADR — only the placeholder set in the rendered template is.
- A separate follow-up PRD is filed concurrently to introduce a `render_for_issue(template, issue_number, extra_scope_args)` convenience method on the renderer, which would absorb the `gh issue view` → `format_issue_comments` → render boilerplate currently duplicated across `iteration/implement.py` and (after this ADR) `iteration/improve.py`. The convenience method also makes sense for `review-prompt.md`, giving three call sites at consolidation time. The current improve PR inlines the boilerplate to keep the dependency tree narrow.
