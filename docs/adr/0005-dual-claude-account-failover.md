# Dual Claude account failover via OAuth token pool

Pycastle previously supported three Claude authentication paths in parallel: `CLAUDE_CODE_OAUTH_TOKEN` (env var, headless), `ANTHROPIC_API_KEY` (env var, pay-per-token), and `CLAUDE_ACCOUNT_JSON` (the contents of `~/.claude.json`, injected as a file inside each container). When a Pro/Max subscription hit its rate limit the orchestrator's only response was to sleep until the reported `reset_time + 2 min`, even when a second subscription was available on the same machine. We are collapsing all Claude auth onto OAuth tokens (`claude setup-token`-generated), supporting a primary plus optional secondary account, and replacing the unconditional sleep with a pool-driven failover: an in-memory `AccountPool` picks a non-exhausted token at each agent spawn and only sleeps when *every* account is exhausted, until the earliest pool wake-time.

## Considered Options

- **Status quo (single account, sleep on limit).** Rejected: leaves a second paid subscription idle for hours per day. The user-facing pain â€” "instead of sleeping, restart with different claude token" â€” is exactly what the issue asks for.
- **Two `~/.claude.json` files on disk, secondary path configurable.** Considered when we believed `claude setup-token` might not work for Pro subscriptions. Rejected after verification: `setup-token` produces a valid long-lived OAuth token for Pro accounts, so the heavier file-based design is unnecessary. Env-only auth keeps secrets in one place (`.env`), removes the Windows path bug surface (#467), and lets the AccountPool work over uniform `(name, token)` tuples.
- **N-token list (`CLAUDE_CODE_OAUTH_TOKEN_1..N`).** Rejected: the concrete use case is exactly two accounts, numbered keys hide priority semantics inside the env var name, and migration churns existing users. Two named keys (primary unchanged, secondary additive) preserves the existing `.env` exactly.
- **Mid-flight container token swap.** Rejected: Docker doesn't support mutating `container_env` of a running container, and the existing worktree preservation + `RALPH:` commit-prefix skip mechanism already makes restart-with-different-token cheap. Killing and respawning at the iteration boundary is structurally cleaner than partial-container plumbing.
- **Persisted exhaustion state across pycastle restarts.** Rejected: in-memory state plus an optimistic re-probe on restart costs at most one wasted Claude call per process start, which is negligible. Persistence adds a stale-state failure mode (saved `reset_time` no longer matches reality) and a file-I/O concern.
- **Keep `ANTHROPIC_API_KEY` as an orthogonal third path.** Rejected as part of this change: API-key auth has no usage-limit semantics (pay-per-token), so it doesn't fit the pool model, and supporting it forces a carve-out branch in every credential code path. Subscription-only auth simplifies the surface.

## Consequences

- `~/.claude.json` is no longer read by pycastle; users who authenticated only via `claude login` must run `claude setup-token` once and paste the result into `.env`. One-time migration step.
- `ANTHROPIC_API_KEY` is removed from `_ENV_KEYS`, the init template, and all credential plumbing. Pycastle is committed to subscription auth only; users who preferred API-key billing to avoid quota issues lose that option.
- `AccountPool` is constructed in `main.py` alongside `_load_env` and passed to `AgentRunner.__init__` as a separate injectable. It is *not* threaded through `Deps` â€” the iteration layer continues to have no access to credentials.
- The existing `UsageLimitError` â†’ `AbortedUsageLimit` plumbing is reused unchanged; only the orchestrator's match arm at `orchestrator.py:182` is gated on `pool.has_available()` before sleeping. New behavior is a strict superset â€” single-token users see no change.
- Failover is per-agent-run granularity. With parallel agents, each that 429s independently calls `mark_exhausted` on the same token (idempotent) and the iteration unwinds via the standard preserve-and-restart flow. No mid-flight swap, no cross-agent cancellation broadcast.
- The `runtime injection` glossary entry is removed; `_inject_claude_credentials` and the `CLAUDE_ACCOUNT_JSON` env-filter in `_build_session` are deleted.
- `pycastle init` is unchanged in shape â€” it still prompts for a single OAuth token. The secondary is documented as an optional `.env` addition rather than a wizard branch, since most users have one account.
