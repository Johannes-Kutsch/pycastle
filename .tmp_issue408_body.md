## Problem Statement

At the end of a Claude subscription window I often have leftover token budget but no `ready-for-agent` issues queued. I want pycastle to *find its own work* during these idle gaps — surface a meaningful codebase improvement, decompose it into AFK-shippable tracer-bullet issues, and feed them back into the regular iteration loop — so I can leave it running overnight and wake up to merged improvements rather than wasted budget.

The work needs to survive pycastle's existing failure modes: pool sleep across all Claude accounts, container teardown between iterations, and Ctrl-C from the user. None of these should silently drop the picker's analysis or leave half-filed issues on GitHub.

> **Depends on #507** (session resume across container teardown) and **#513** (worktree context manager consolidation). #507 delivers the `--session-id <uuid>` / `--resume <uuid>` primitive and worktree-internal session storage. #513/#514 delivers two consolidated context managers: `transient_worktree(name, *, sha, deps)` for always-teardown scratch (plan-sandbox, pre-flight-sandbox) and `managed_worktree(name, *, branch, sha, delete_branch_on_teardown, deps)` for branched worktrees with the broadened preservation rule (`dirty OR usage_limit_exc OR has_resumable_session`). The improve-sandbox piggy-backs on the merge-sandbox shape: `managed_worktree(name="improve-sandbox", branch="pycastle/improve-sandbox", sha=safe_sha, delete_branch_on_teardown=True, deps=deps)`. The branch is unused (improve never commits) but the structural symmetry gives broadened preservation across pool sleep for free, and on clean exit the unused branch is torn down with the worktree. Improve mode adds multi-prompt-within-one-run on top of these primitives — and owns one cross-teardown concern of its own (the **phase progress file**, see below) that #507 does not cover.

## Solution

A new `--improve=until_sleep|endless` flag on `pycastle run`. When set, the iteration's existing pre-plan branch (0 / 1 / 2+ open issues) gains a new "0 issues" sub-branch: dispatch an **improve agent** that scans the codebase, picks one improvement opportunity, writes a PRD, and files a parent issue plus 1–5 `ready-for-agent` sub-issues with blocker links. After the improve agent returns, the iteration re-checks open-issue count and continues normally (1 → direct implement, 2+ → plan, 0 → exit).

The improve agent is a single container run with three prompts injected sequentially across one preserved Claude conversation: scan/pick → PRD → issues. Conversation continuity within the run is achieved by extending the Work-phase contract from "one prompt per container run" to "an ordered list of prompts per container run." The first invocation uses `claude --session-id <uuid>` (the UUID coming from the same orchestrator-assigned identity #507 establishes); subsequent invocations within the same run use `claude --resume <uuid>`.

Cross-teardown survival has two halves. The Claude conversation is preserved entirely by #507's primitive (session UUID + worktree-internal session storage). The **prompt-progress cursor** — which of the three phases have already been confirmed `<promise>COMPLETE</promise>` — is preserved by improve mode itself via a small **phase progress file** at `<worktree>/.pycastle-session/improve/_phase_progress` containing an integer count of confirmed-complete phases. The file is updated after each parsed `<promise>COMPLETE</promise>`. On Resume, the orchestrator reads it and resumes the prompt iterator at the next un-confirmed phase: the continuation prompt is sent first, then either the next un-delivered phase prompt (if the last phase was confirmed COMPLETE) or nothing (if the last phase was mid-flight — the agent picks up from the conversation history). The orchestrator never *skips ahead* without seeing COMPLETE; re-prompting an in-flight phase on Resume is acceptable because phases are idempotent enough at the prompt level.

Stop conditions:
- `--improve=until_sleep` (default if `--improve` is bare): once a pool sleep has occurred AND the post-sleep wake-up clears the resulting backlog, exit; do not generate further improvements.
- `--improve=endless`: keep generating after every backlog clear; the only stop is Ctrl-C.
- Either mode: empty improve result (agent files 0 issues) falls through to the existing 0-case → exit.

## User Stories

1. As a pycastle user, I want a `--improve=until_sleep|endless` flag on `pycastle run`, so I can opt into idle-fill behaviour without it ever triggering by accident.
2. As a pycastle user, I want `until_sleep` to be the default when `--improve` is given without a value, so that overnight runs naturally bound themselves to roughly one subscription window.
3. As a pycastle user, I want `--improve=endless` to keep generating after every pool sleep, so I can run pycastle across multiple windows when I am away for a long stretch.
4. As a pycastle user, I want existing `ready-for-agent` issues to take priority over improvement generation, so my real backlog clears before the agent invents new work.
5. As a pycastle user, I want any in-flight issues (open `pycastle/issue-N` branches/worktrees from a previous run) to take priority over improvement generation, so improve never piles new issues on top of unfinished work.
6. As a pycastle user, I want `ready-for-human` issues to be ignored by the improve trigger, so HITL-pending work doesn't block overnight idle-fill.
7. As a pycastle user, I want preflight (including the preflight pull and safe-SHA pin) to always run before improve, so the improve agent operates on the latest verified-clean committed state.
8. As a pycastle user, I want each improve run to file exactly one parent issue carrying the PRD as its body, so the rationale lives next to the work it produced.
9. As a pycastle user, I want each improve run to file 1–5 sub-issues with blocker links between them, so the regular iteration loop processes them in dependency order without me re-triaging.
10. As a pycastle user, I want sub-issues to land directly with `ready-for-agent` (no `needs-triage`), so the next iteration of `pycastle run` picks them up without me waking up.
11. As a pycastle user, I want the improve agent's prompt to forbid picking opportunities that need a human decision, so we don't generate AFK issues that will silently get stuck.
12. As a pycastle user, I want `IMPROVE_MAX_ISSUES` (default 5) to cap sub-issues per run, so an over-eager agent in `endless` mode can't flood the backlog overnight.
13. As a pycastle user, I want the improve agent to share one Claude conversation across its three prompts within a single container run, so phase 2 (PRD) and phase 3 (issue split) build on phase 1's analysis rather than re-deriving it.
14. As a pycastle user, I want each prompt phase to end with `<promise>COMPLETE</promise>`, so the orchestrator can detect phase completion using the existing agent output protocol with no new tags.
15. As a pycastle user, I want phase isolation (the agent does not see phases 2/3 while doing phase 1), so the picker is not biased by hindsight about splittability and the prompts can be authored independently.
16. As a pycastle user, I want the improve agent to run in a dedicated **improve-sandbox worktree** (not the plan-sandbox), so the term, role, and lifecycle of the read-only sandboxes stay distinguishable in CONTEXT.md.
17. As a pycastle user, I want the improve-sandbox to skip preflight checks inside it (mirroring the plan-sandbox), so the improve agent doesn't waste time re-running checks the orchestrator just verified.
18. As a pycastle user, I want pool sleep during an improve run to *not* abort the agent's analysis — the Claude conversation is preserved by #507's primitive, and the **phase progress file** owned by improve mode preserves which of the three prompts have already been confirmed COMPLETE so the orchestrator resumes the prompt iterator at the right phase.
18a. As a pycastle developer, I want a `<worktree>/.pycastle-session/improve/_phase_progress` file written after each parsed `<promise>COMPLETE</promise>`, so a teardown mid-phase-2 (or mid-phase-3) does not cause the orchestrator to either skip an un-completed phase or repeat a confirmed-complete phase on Resume.
18b. As a pycastle developer, I want the improve-sandbox to reuse the same `managed_worktree` shape as the merge-sandbox (`branch="pycastle/improve-sandbox", delete_branch_on_teardown=True`), so the broadened preservation rule applies mechanically across pool sleep and improve mode does not introduce a third sandbox context manager or a fourth preservation policy.
19. As a pycastle user, I want `until_sleep` mode to track a `slept_once` flag at the orchestrator level, so the loop can distinguish "we have not yet hit our token ceiling" from "we are now past it" without inspecting individual account state.
20. As a pycastle user, I want regular `pycastle run` (no `--improve`) to behave exactly as today, so the new feature is purely opt-in and the existing 3-phase loop stays pristine.
21. As a pycastle user, I want a per-role **STAGE_OVERRIDES** entry for the improve agent, so I can pin a specific model and effort level for the picker independently of the planner.
22. As a pycastle user, I want the improve agent to use the existing **AccountPool** for token rotation, so it benefits from the same dual-Claude-account failover as every other agent.
23. As a pycastle user, I want any error during improve to surface through the existing **auto bug reporter**, so crashes file an upstream bug just like every other CLI subcommand.
24. As a pycastle developer, I want the improve dispatch logic to live behind a small pure decision function (open-issue count, in-flight count, `--improve` mode, `slept_once` → `Continue|DispatchImprove|Done`), so the routing rules are unit-testable without spinning up containers.
25. As a pycastle developer, I want the multi-prompt sequential mechanism to be expressed as a list-of-prompts extension to the existing Work-phase contract (N=1 today for every other agent, N=3 for improve), so the path is reused if any future agent wants the same shape.

## Implementation Decisions

- **Surface**: new `--improve` flag on `pycastle run` accepting `until_sleep` or `endless`; default value when `--improve` is bare is `until_sleep`. No new top-level subcommand.
- **Loop placement**: improve dispatches as a new branch inside the existing pre-plan 0/1/2+ open-issue check. Trigger condition: `0 ready-for-agent issues AND 0 in-flight issues AND --improve set`. After improve runs, the iteration re-checks the same condition; the existing 0/1/2+ branches handle the post-improve state.
- **Stop semantics**: orchestrator tracks a `slept_once` flag. In `until_sleep`, improve never re-fires after `slept_once`; the next clear-of-backlog completes and the run exits via the existing 0-case. In `endless`, `slept_once` is ignored; only Ctrl-C stops the loop. Empty improve result (agent files 0 issues) ends the run via the same 0-case fall-through (no special exit code).
- **Sandbox**: a new **improve-sandbox worktree** at `.pycastle/.worktrees/improve-sandbox`, on the stable branch `pycastle/improve-sandbox`. Checkout of safe SHA, preflight skipped inside it. Implemented as a `managed_worktree(name="improve-sandbox", branch="pycastle/improve-sandbox", sha=safe_sha, delete_branch_on_teardown=True, deps=deps)` call site — same shape as the merge-sandbox. The branch is unused (improve never commits); `delete_branch_on_teardown=True` ensures it is torn down with the worktree on clean exit. The broadened preservation rule (`dirty OR usage_limit_exc OR has_resumable_session`) keeps the worktree (and its phase progress file + Claude session storage) alive across pool sleep, mechanically, with no new policy. Distinct from plan-sandbox and pre-flight-sandbox by **lifecycle** (broadened-preservation vs. always-teardown) and **purpose** (multi-prompt analysis vs. one-shot scratch), not by a separate context manager.
- **Agent**: one container run, one role (`improve`), three prompts injected sequentially:
  - Phase 1: scan codebase / pick one opportunity (modelled on `/improve-codebase-architecture`)
  - Phase 2: write PRD (modelled on `/to-prd`)
  - Phase 3: split into 1–5 sub-issues with blocker links (modelled on `/to-issues`)
- **Multi-prompt mechanism within one container run**: extend the Work-phase contract from a single prompt to an ordered list of prompts. The first claude invocation uses `--session-id <uuid>` (UUID supplied by #507's per-run identity machinery); subsequent invocations within the same run use `--resume <uuid>`. Each phase ends with `<promise>COMPLETE</promise>`, reusing the existing agent output protocol with no new tags. For agents that take only one prompt (Implementer, Reviewer, Merger), N=1 and the existing single-stream behaviour is preserved.
- **Cross-teardown resume after pool sleep**: split between #507 and improve mode.
  - #507 owns the **Claude conversation**: session UUID, worktree-internal session storage, resume detection (`decide_agent_run_kind`), continuation prompt.
  - Improve mode owns the **prompt-progress cursor**: a `<worktree>/.pycastle-session/improve/_phase_progress` file containing an integer count of confirmed-complete phases. Written after each parsed `<promise>COMPLETE</promise>`, *inside* the worktree context manager (so it is preserved alongside the session). Read on every Resume to position the prompt iterator. Cleaned up by **role session cleanup** at the same time as the rest of the role session dir on successful completion.
  - On Resume, the orchestrator: (a) reads the progress file, (b) sends only the **continuation prompt** for the in-flight phase (whose conversation history will resume), (c) when the in-flight phase emits its delayed COMPLETE, advances the iterator and sends the next un-delivered phase prompt. If the progress file is absent or malformed on Resume, the orchestrator restarts at phase 0 — fail-soft, identical to a fresh run.
- **Output protocol**: the agent files all GitHub issues itself via the in-container `gh` CLI (GH_TOKEN-injected per ADR 0004). The agent emits `<promise>COMPLETE</promise>` at the end of each phase only — no `<issue>` tag, no inter-phase structured output, since neither the orchestrator nor downstream phases need to reference issue numbers.
- **Issue shape**: parent issue carries the PRD body; sub-issues are 1–5 tracer-bullet slices, all directly labelled `ready-for-agent`, with blocker labels/links between them as needed. Parent issue may be `ready-for-human` (so the operator notices the rationale post-hoc) or `ready-for-agent` — TBD in implementation, not load-bearing.
- **Config additions**:
  - `IMPROVE_MAX_ISSUES: int = 5` — hard cap on sub-issues per improve run.
  - `STAGE_OVERRIDES["improve"]` — a sixth entry alongside `plan`/`implement`/`review`/`merge`/`preflight-issue`, with `model` and `effort` fields. CLI defaults via empty strings, same convention as existing entries.
- **Safety**: the picker prompt explicitly forbids picking opportunities that need a human decision; this is the sole enforcement mechanism for "do not alter front-facing functionality" — we trust the agent's self-grilling step rather than building a static guard.
- **Dispatch decision function**: a small pure function `decide_iteration_action(open_afk_count, in_flight_count, improve_mode, slept_once) → IterationAction` returning `RunPlan | RunImplementDirect | DispatchImprove | Done`. Used by the orchestrator's iteration entry. The resume-vs-fresh decision for the improve agent itself is delegated to #507's `decide_agent_run_kind`.

## Modules to Build / Modify

- `iteration/improve.py` (new): the improve phase, including the multi-prompt sequence, sandbox lifecycle (one `managed_worktree(name="improve-sandbox", branch="pycastle/improve-sandbox", delete_branch_on_teardown=True, ...)` call site, mirroring `merge_phase`), the **phase progress file** read/write logic, and the call into #507's primitive for run-kind detection. Mirrors the depth and shape of `iteration/planning.py`.
- `iteration/_dispatch.py` (new, or fold into iteration entry): the pure decision function `decide_iteration_action`. Deep, testable, no side effects.
- `worktree.py` (no changes from this issue): the existing `managed_worktree` is the call site for the improve-sandbox, used identically to `merge_phase`'s merge-sandbox call. No new context manager and no new preservation policy added here.
- `container_runner.py` (modify): extend the Work phase to accept an ordered list of prompts; on N>1, drive an inner loop of `claude --resume <uuid>` invocations between phases. Emit one `AgentOutput` per phase boundary; final `AgentOutput` is the agent run's result. For N=1, behave as today.
- `agent_runner.py` (modify): expose an `AgentRole.IMPROVE` value and wire it into prompt resolution, STAGE_OVERRIDES lookup, and AccountPool integration. The session UUID is supplied by #507's run-kind machinery.
- `services/github_service.py` (modify if needed): confirm parent/sub-issue and blocker-link helpers are available; add only what is missing.
- `defaults/config.py` (modify): add `IMPROVE_MAX_ISSUES` and the `improve` entry in `STAGE_OVERRIDES`.
- `prompts/improve-codebase/` (new directory under defaults and the consuming project's `pycastle/prompts/`):
  - `01-scan.md` — picker prompt (scope TBD)
  - `02-prd.md` — PRD prompt (scope TBD)
  - `03-issues.md` — issue-split prompt (scope TBD)
  - The shared post-interrupt continuation prompt comes from #507; improve does not own it.
- `main.py` (modify): add the `--improve` Click option with `until_sleep|endless` choices and bare-flag default of `until_sleep`; thread it into the orchestrator.
- Orchestrator iteration entry (modify): track `slept_once`, route through `decide_iteration_action`, dispatch `improve_phase` for the new branch.

## Testing Decisions

- **Test through public seams; no private-method tests** (per existing project guidance and standing user feedback).
- **`decide_iteration_action`** — exhaustive unit tests over the decision space: every combination of (open_afk_count ∈ {0, 1, 2+}, in_flight_count ∈ {0, ≥1}, improve_mode ∈ {off, until_sleep, endless}, slept_once ∈ {true, false}) maps to the expected action. Pure function, no fixtures.
- **Multi-prompt Work-phase extension** — integration test with a fake DockerSession asserting that N=3 prompts produce three claude invocations: the first with `--session-id <uuid>`, the rest with `--resume <uuid>`, all sharing one UUID. For N=1, assert behaviour is byte-identical to the pre-change path.
- **Phase progress file** — unit tests at the public seam of `iteration/improve.py`: assert the file is written with the correct integer after each parsed `<promise>COMPLETE</promise>`; assert it lives at `<worktree>/.pycastle-session/improve/_phase_progress` (so cleanup and preservation are inherited from the session dir); assert that on Resume with progress=0/1/2/3 the orchestrator dispatches the correct prompt sequence; assert that an absent or malformed progress file is treated as progress=0 (fail-soft to fresh run); assert that successful improve completion (progress=3) triggers role session cleanup which also removes the progress file.
- **Improve dispatch end-to-end** — an iteration test using existing orchestrator fixtures with mocked `GithubService` and `AgentRunner`: assert that with `--improve=until_sleep`, an empty backlog triggers improve, and that on the simulated post-improve recheck the dispatcher routes correctly into `RunImplementDirect` / `RunPlan` / `Done`.
- **Stop semantics** — `until_sleep` exits after `slept_once=True` and a clear; `endless` continues across multiple simulated sleeps; empty improve result exits via the 0-case in both modes.
- **STAGE_OVERRIDES["improve"]** — config-load tests covering valid/invalid effort and model values, mirroring existing STAGE_OVERRIDES tests.
- **Cross-teardown resume of improve** — covered by #507's tests against Implementer/Reviewer/Merger; this issue does not duplicate that coverage. A single smoke test that the improve role is recognised by #507's `decide_agent_run_kind` is sufficient.
- **Prior art**: existing tests in `tests/test_planning.py`, `tests/test_implement.py`, and `tests/test_default_prompts.py` are the templates. Use the same `Deps`-substituting inline dataclass pattern (per `_WorktreeDeps`) for any phase-level tests.
- **Out-of-bounds**: do NOT write tests that assert specific prompt contents for `01-scan.md` / `02-prd.md` / `03-issues.md` — those are author-time concerns and will be grilled in a separate session. A structural guard (file exists, ends with phase-completion instruction, no `gh` shell expansions) is acceptable.

## Out of Scope

- **Session resume across container teardown** — owned by #507; this PRD assumes the primitive is already in place.
- **Generalising session-resume to Implementer / Reviewer / Merger** — owned by #507.
- **The shared continuation prompt** — owned by #507.
- **`CLAUDE_CONFIG_DIR` plumbing, the per-worktree session directory, the `pycastle init` gitignore scaffolding for `/.pycastle-session/`** — all owned by #507.
- **Scope and content of `01-scan.md`, `02-prd.md`, `03-issues.md`** — to be designed in a separate grilling session focused on each prompt's exact instructions, including the self-grilling step ("question yourself, clearly lay out why this is the best solution") and the front-facing-functionality safety language.
- **Static enforcement of "no front-facing changes"** — not in scope. Enforcement is by prompt instruction and the agent's own self-grilling. No AST guards, no diff scanners.
- **Persisting improve outcomes outside GitHub** — no PRD file commits, no logs beyond the existing `errors.log`. The parent issue is the durable artefact.
- **Auto-merging the generated improvement issues** — no special path. They go through the regular `pycastle run` 3-phase loop like any other `ready-for-agent` issue.

## Further Notes

- **Dependency order**: #507 lands first (session-resume primitive), then #513/#514 (worktree consolidation into `transient_worktree` + `managed_worktree`), then this issue. Each prerequisite is exercised by existing call sites before the next builds on it: #507 by Implementer/Reviewer/Merger, #513/#514 by every existing worktree call site (no behaviour change). Improve mode lands third, reusing the merge-sandbox `managed_worktree` shape and building multi-prompt-within-one-run plus the phase progress cursor on top of those two primitives.
- **Resumable grilling**: the big remaining design surfaces for this issue are: (1) the three prompt scopes (`01-scan.md`, `02-prd.md`, `03-issues.md`); (2) the parent issue's exact label and title format; (3) STAGE_OVERRIDES default model and effort for the improve role.
- **Domain language additions** (to land in `CONTEXT.md` during implementation):
  - **improve mode**, **improve agent**, **improve-sandbox worktree**, **`--improve` flag**, **`until_sleep`**, **`endless`**, **`IMPROVE_MAX_ISSUES`**, **slept_once flag** (orchestrator-level state), **multi-prompt agent run** (the N>1 Work-phase shape), **phase progress file** (`<worktree>/.pycastle-session/improve/_phase_progress`, the prompt-iterator cursor preserved across container teardown).
- **Standing user feedback honoured in this design**:
  - No private-method tests (existing memory): all tests above are scoped to public seams.
  - Issue label / type pairing: AFK sub-issues land `ready-for-agent`, parent issue surfaces the rationale; no batched same-label-across-all-issues anti-pattern.
- **Why a parent issue rather than a committed PRD file**: keeps improve mode read-only against the repo (no commits during a phase that isn't an Implementer), reuses GitHub's parent/sub-issue surface, and avoids the awkwardness of generated PRD files outliving the issues they spawn.
- **Why concatenated session resume rather than mega-prompt**: the user explicitly preferred sequential prompts to avoid overwhelming the agent with all three steps upfront; the implementation cost of `--session-id` + `--resume` is bounded and reuses the primitive #507 establishes.
