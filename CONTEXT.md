# Ubiquitous Language

## Package & Distribution

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **pycastle** | Installable Python package that orchestrates Claude Code agents | orchestrator, tool |
| **consuming project** | A project that installs pycastle and customizes it via a local pycastle directory | host project, parent project |
| **pycastle directory** | `pycastle/` inside a consuming project, holding local overrides for config, Dockerfile, .env, prompts | config dir, override dir |
| **defaults** | Files bundled in the package serving as scaffolding templates and runtime fallbacks | base config, starter files |
| **pycastle init** | CLI command copying defaults into the consuming project's pycastle directory, then running the init wizard | setup, scaffold |
| **init wizard** | Interactive step-by-step flow inside `pycastle init` collecting credentials and optionally creating GitHub labels | setup wizard, onboarding |
| **pycastle labels** | CLI subcommand creating or resetting GitHub labels in a target repo using the canonical label set | label setup, label sync |
| **auto-discovery** | Runtime lookup: pycastle directory in CWD → `pycastle home` → defaults; project marker = `pycastle/` exists in CWD | — |
| **pycastle home** | User-scope global directory holding optional `config.py` and `.env`; resolved via `platformdirs` and overridable via `PYCASTLE_HOME`; only `config.py` and `.env` are globalizable (ADR 0003) | global config dir, user config dir |
| **PYCASTLE_HOME** | Env var overriding `platformdirs` default; resolution: explicit `global_dir` arg → `PYCASTLE_HOME` → `platformdirs` | — |
| **layered config merge** | Precedence stack: `defaults module → pycastle home/config.py → pycastle dir/config.py`, applied field-by-field; `.env` extends to `defaults → global .env → local .env → process env`, per-key (ADR 0003) | config layering, config precedence |
| **global-forbidden fields** | `Config` fields that may not be set globally (raise `ConfigValidationError` at load): `pycastle_dir`, `prompts_dir`, `logs_dir`, `worktrees_dir`, `env_file`, `dockerfile`, `docker_image_name` | global-forbidden path fields, path field guard |
| **scope flag** | `--global` / `--local` mutually-exclusive flag pair on `pycastle init`; selects scaffold target; absent flag triggers interactive prompt; `Dockerfile`, `prompts/`, `.gitignore` always local | global/local flag |
| **layer summary** | One-line startup message listing which config layers were found, routed via `StatusDisplay.print("", ...)` | config layer line |

| **auto_file_bugs** | Boolean (default `False`) gating *all* upstream auto-issue filing into `bug_report_repo`; when `False`, each site falls back to non-filing behaviour; flipped on only by pycastle developers via global config (ADR 0022) | bug toggle, auto report flag |
| **bug_report_repo** | Config (default `"Johannes-Kutsch/pycastle"`) — `owner/repo` target for auto-filed bug reports; validated against `^[^/]+/[^/]+$`; not global-forbidden | upstream repo, bug repo |

## GitHub Integration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **label** | GitHub issue classification marker with name, description, hex color | tag, category |
| **canonical label set** | Ten labels hardcoded in the package: `bug`, `needs-info`, `needs-triage`, `ready-for-agent`, `ready-for-human`, `wontfix`, three slice-mode labels (`refactor-slice`, `behavior-slice`, `docs-slice`), and `needs-slice-type` | default labels, label config |
| **label reset** | Option to delete all existing labels before creating canonical set | label wipe, clean labels |
| **issue label** | Label value (default `ready-for-agent`) marking issue as eligible for agent processing | trigger label, agent label |
| **slice-mode label** | One of `refactor-slice`, `behavior-slice`, `docs-slice`; mandatory on every issue producing code or docs; selects implement prompt. Exactly one per issue — missing/multiple/unknown filtered out of planning candidates and marked `needs-slice-type`. Classified by **slice-mode classifier** (ADR 0016) | mode label, slice label |
| **needs-slice-type** | Label applied by `planning_phase` to any `ready-for-agent` issue lacking exactly one slice-mode label. Synced both directions each iteration. Coexists with `ready-for-agent` (issue not demoted); marks issue ineligible for this iteration's implement phase. Malformed issue stays in `all_open_issues` so dependents see it as a blocker | slice-needed label |
| **refactor-slice** | Slice mode for issues whose steps cannot be verified by a new test of observable behavior (symbol moves, renames, protocol introductions, import rewires, dead-code removal). Multiple steps may bundle. Lands first; dependents list it in `Blocked by`. Dispatches to `implement/refactor.md` (ADR 0016) | refactor prep, refactor issue |
| **behavior-slice** | Slice mode introducing/changing observable behavior verifiable by a new test. Acceptance criteria describe behavior + observable surface, never test-code structure. Dispatches to `implement/behavior.md` with the behavior gate (ADR 0016) | feature slice, behavior issue |
| **docs-slice** | Slice mode whose only artifacts are markdown edits (CONTEXT.md, ADRs, README). No code, no tests. Dispatches to `implement/docs.md` (ADR 0016) | docs issue, documentation slice |
| **slice-mode classifier** | Module at `src/pycastle/slice_classifier.py` owning the slice-mode label concept. Exposes `SliceMode` enum (variants for refactor/behavior/docs, each carrying display name and `PromptTemplate`; label string resolved from `Config` at lookup), the `SliceClassification` discriminated union, pure `classify_slice(issue, cfg) -> SliceClassification`, and `slice_labels(cfg) -> frozenset[str]`. Every slice-label call site routes through this module (ADR 0016) | slice classifier, mode classifier |

## Agents

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **agent** | Claude Code instance running inside an isolated Docker container | bot, worker |
| **Planner** | Plan-phase agent; runs in a clean checkout of the safe SHA; outputs a plan | planning agent |
| **Implementer** | Implement-phase agent; one per issue | coding agent, implementation agent |
| **Reviewer** | Post-Implementer agent validating changes before merge. Renders `PER_ISSUE` scope (`PromptTemplate.REVIEW`); does **not** receive a precomputed diff — runs `git diff main...` inside the container so branch lag doesn't pull unrelated content | review agent |
| **Merger** | Merge-phase agent spawned only when at least one conflicting branch exists; integrates conflicting branches and closes their issues | merge agent, integration agent |
| **preflight-issue agent** | Agent spawned by `PreflightCache.get_safe_sha()` on `PREFLIGHT_CHECKS` failure; explores codebase, evaluates HITL, files one structured GitHub issue, outputs `<issue>NUMBER</issue>` | bug-report agent, error reporter |
| **divergence-resolver** | Agent role (`AgentRole.DIVERGENCE_RESOLVER`) spawned by `get_safe_sha()` as the third escalation rung when `pull_with_merge_fallback` raises on textual conflicts; runs inside `diverge-sandbox` (built from local main's tip), merges `origin/<branch>` and resolves all conflicts. Textual conflict resolution only — explicitly forbidden from running preflight checks; post-merge breakage is caught by the same `get_safe_sha()` call's subsequent preflight pass on the new HEAD and routed through the preflight-fix path. Success → `<promise>COMPLETE</promise>` and fast-forward; failure → `<promise>FAILED</promise>` and original `GitCommandError` propagates. Runs under bare flag profile (ADR 0024) | conflict resolver, merge conflict agent |
| **HITL verdict** | Routing decision in the preflight issue's label: `ready-for-agent` → spawn one Implementer; `ready-for-human` → orchestrator aborts | HITL decision, routing verdict |
| **agent flag profile** | Per-`AgentRole` set of Claude CLI flags selected by `claude_service.build_command`; hardcoded (not user-tunable via `STAGE_OVERRIDES`) because the flags encode each role's tool/permission contract. Universal: `--disable-slash-commands`, `--exclude-dynamic-system-prompt-sections`. **Bare set** (`--bare`): Planner (`+ --tools "Read,Glob"`), divergence-resolver. **Investigator restriction** (`--disallowedTools "Edit Write NotebookEdit"`): preflight-issue, improve scan. **Full-tool**: Implementer, Reviewer, Merger. All non-bare roles also get `--strict-mcp-config --mcp-config '{}'` (ADR 0024) | flag matrix, per-role flags |
| **commit body composition** | Host-side commit body produced by `run_issue()` on clean exit: `f"Implement #{n} - {message_or_title}"` (Implementer) / `f"Review #{n} - {message_or_title}"` (Reviewer); `message` = `<commit_message>` if present, else issue title. No orchestrator-readable stage marker in body — stage-done is filesystem-based (ADR 0007) | commit message format |
| **`<commit_message>` tag** | Optional XML tag from Implementers/Reviewers carrying plain description of changes; orchestrator uses content as commit body when present, synthetic fallback when absent. Absence is **not** a failed run (ADR 0007) | — |

## Iteration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **orchestrator** | Main loop coordinating agent phases across GitHub issues | runner, coordinator |
| **iteration** | One complete plan-bounded loop ending after all planned issues attempted; up to MAX_ITERATIONS per `pycastle run`; parallel mode: one merge phase closes the iteration; sequential mode: each issue gets its own merge before next | cycle, round, pipeline |
| **3-phase loop** | One iteration's structure: plan, implement+review, merge | pipeline, workflow |
| **sequential mode** | Auto-activates when `max_parallel = 1`; each issue processed individually (implement → review → merge) with safe SHA re-pinned from HEAD after each merge; eliminates merge conflicts from parallel branches sharing the same base | serial mode, one-at-a-time mode |
| **plan phase** | Phase where Planner analyzes open issues and produces a plan | planning step |
| **implement phase** | Phase where Implementers fix issues in isolated worktrees | coding step |
| **review phase** | Phase where Reviewer checks an Implementer's changes before merge | code review step |
| **merge phase** | Phase where completed branches are integrated, issues closed, Merger spawned on conflicts | integration step |
| **safe SHA** | SHA most recently observed at HEAD by `PreflightCache.get_safe_sha()` and confirmed by `PREFLIGHT_CHECKS` in `preflight-sandbox`; single-slot verdict surviving across iterations until HEAD advances. Fetched on demand by `improve_phase`, `planning_phase`, `merge_phase`. `implement_phase` consumes the SHA from `PlanReady.sha` instead (ADR 0014) | verified SHA, clean SHA |
| **preflight pull** | `git pull --ff-only` (with merge fallback) inside every `get_safe_sha()` call; waits for clean working tree; on textual conflict spawns divergence-resolver in `diverge-sandbox`; aborts if remote unreachable or resolver fails | — |
| **`PreflightCache`** | Process-scoped object owning the safe-SHA verdict. Constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Single private slot `_verdict: PreflightReady | PreflightAFK | PreflightHITL | None` plus `asyncio.Lock` serialising concurrent callers. `async def get_safe_sha(deps) -> PreflightResult`: under lock, waits for clean tree, runs `pull_with_merge_fallback` (ff-only → clean merge → divergence-resolver), reads HEAD; if slot SHA matches HEAD returns cached verdict; else opens `transient_worktree("preflight-sandbox", sha=HEAD)`, runs `PREFLIGHT_CHECKS`, stores `PreflightReady(sha=HEAD)` on success or runs preflight-issue agent and stores `PreflightAFK | PreflightHITL` on failure. Single-slot suffices because the escalation ladder only moves HEAD forward (ADR 0013) | preflight cache, safe-sha cache |
| **preflight-fix path** | Orchestrator routing when `get_safe_sha()` returns `PreflightAFK`: freshly-filed preflight issue is handed straight to `implement_phase` in the same iteration; `planning_phase` is bypassed (planning skip); normal Reviewer → merge pipeline follows | preflight fast path |
| **programmatic merge path** | Fast-path merge logic running `git merge --no-edit` directly via subprocess without the Merger; used when all branches merge cleanly | fast path, direct merge |
| **clean merge** | `git merge --no-edit` that exits zero with no conflict resolution | conflict-free merge, successful merge |
| **conflicting branch** | Branch whose `git merge --no-edit` exits non-zero; `git merge --abort` runs immediately, branch collected for Merger | failed merge branch |
| **in-flight issue** | Open issue with an existing `pycastle/issue-<n>` branch or worktree from a previous interrupted iteration | mid-flight issue, resumed issue |
| **merge-time preflight skip** | When `merge_phase`'s `get_safe_sha()` (over post-clean-merge HEAD, before Merger) returns `PreflightAFK | PreflightHITL`: log, skip Merger, return `MergeResult` with conflict issues pending. Cache's single slot holds the broken-HEAD verdict, so next iteration's first call returns the same verdict (no duplicate) and orchestrator routes the existing issue through `implement_phase` | merge preflight abort |
| **planning skip** | Pipeline-level bypass of `planning_phase` triggered in two cases: (1) **in-flight present** — any open issue is in-flight, so `run_iteration` hands the in-flight set to `implement_phase`; (2) **preflight-fix AFK** — `get_safe_sha()` returned `PreflightAFK` carrying a freshly-filed issue, routed directly to `implement_phase` | plan bypass |
| **implement skip** | Behavior in `run_issue` when `.pycastle-session/implementer/` exists and `RoleSession(...).is_resumable()` is false (Implementer ran to clean exit): Implementer not spawned, Reviewer runs directly (ADR 0007) | — |
| **review skip** | Behavior in `run_issue` when `.pycastle-session/reviewer/` exists and `RoleSession(...).is_resumable()` is false: both Implementer and Reviewer skipped, issue counted complete (ADR 0007) | — |
| **stage-done signal** | Per-role completion predicate `RoleSession(<wt>, <role>).is_done()`. Role dir created when agent first starts; on clean exit orchestrator clears session JSONL but leaves dir, flipping predicate from "in progress" to "done." Honoured by `managed_worktree.__aexit__` via the session role-dir-present predicate so the sentinel survives teardown between implement and review (ADR 0007) | stage completion check |
| **plan** | Structured JSON from Planner listing issues and branch names; `plan_phase()` sorts by ascending issue number | plan output, plan JSON |
| **issue** | GitHub issue labeled for agent processing, representing one unit of work | ticket, task, card |
| **AFK issue** | Issue the Planner assigns to an Implementer because it can be resolved autonomously; labeled `ready-for-agent` | agent issue, auto issue |
| **HITL issue** | Issue requiring human intervention; labeled `ready-for-human` — Planner must never assign it | manual issue, human issue |
| **blocker** | Issue that must be resolved before another can be worked on. Any open non-`wontfix` issue counts as a hard blocker; `wontfix` treated as closed | dependency, prerequisite |
| **all-open issues list** | Lightweight `[{number, title, labels}]` projection of every open issue injected into plan prompt as `ALL_OPEN_ISSUES_JSON`; gives Planner cross-label visibility for blocker detection | blocker visibility list |
| **ready-for-agent issues list** | Hydrated `[{number, title, body, labels, comments}]` projection of open issues carrying `issue_label`; injected as `READY_FOR_AGENT_ISSUES_JSON`; candidate set Planner picks from | AFK issues list, candidate set |
| **AllBlocked** | Outcome returned by `planning_phase` when Planner produces zero implementable issues; `run_iteration` ends with `Done` — no within-iteration improve dispatch from this path | empty plan, all-blocked |
| **dependency graph** | Blocker relationships analyzed by Planner to determine the safe working set | issue graph, dependency map |
| **worktree** | Isolated git working tree on the host, bind-mounted into an agent container; one of: named-branch worktree (Implementer/Reviewer/Merger), merge-sandbox, diverge-sandbox, improve-sandbox, or detached plan-sandbox | workspace, branch dir |
| **plan-sandbox worktree** | Temporary detached checkout of safe SHA created by `planning_phase`; always removed in `try/finally`; never branch-associated; at `.pycastle/.worktrees/plan-sandbox` | planner worktree |
| **merge-sandbox worktree** | Temporary named-branch worktree (`pycastle/merge-sandbox`) created by `merge_phase` from HEAD after clean merges; Merger runs inside; always removed in `try/finally`; on success `merge_phase` fast-forwards `main` from the branch before cleanup | merger worktree, conflict worktree |
| **diverge-sandbox worktree** | Temporary named-branch worktree (`pycastle/diverge-sandbox`) created by `get_safe_sha()` from local main's tip when `pull_with_merge_fallback` raises on textual conflicts; divergence-resolver runs inside; always removed in `try/finally`; on success fast-forwards `main` before cleanup | divergence worktree |
| **improve-sandbox worktree** | Named-branch worktree (`pycastle/improve-sandbox`) created by `improve_phase` from safe SHA; agent scans, writes PRD, files sub-issues — branch never committed to but exists so the broadened preservation rule keeps the worktree alive across pool sleep; preflight skipped inside; uses `managed_worktree(..., delete_branch_on_teardown=True, ...)` mirroring merge-sandbox | improve worktree, picker worktree |
| **branch** | Git branch name assigned to an issue in the plan; pattern `pycastle/issue-<n>-<slug>` | feature branch, issue branch |
| **orphan worktree** | Worktree dir under `.pycastle/.worktrees/` no longer registered in git, typically from a crashed agent | stale worktree, leftover worktree |
| **orphan sweep** | Startup operation cross-referencing `.pycastle/.worktrees/` against `git worktree list --porcelain`, deleting unregistered dirs and removing `.worktrees` parent if empty | worktree cleanup, stale cleanup |
| **collision detection** | Per-branch `asyncio.Lock` preventing two parallel agents from simultaneously creating worktrees for the same branch | — |

| **`{{CHECKS}}`** | Global placeholder produced from `config.PREFLIGHT_CHECKS`; injects quality check commands | — |

## Agent Output Protocol

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **agent output protocol** | Contract between prompts and orchestrator: XML tags (`<plan>`, `<issue>`, `<commit_message>`, `<promise>`), plus the module owning the NDJSON → typed output pipeline | output format, agent tags, agent signals |
| **`<plan>` tag** | XML tag from Planner with JSON payload `{issues, blocked}`; `issues` lists unblocked AFK issues; `blocked` lists held-back AFK with `{number, blocked_by, reason}` | plan output, plan block |

Dropped: `turn.started`, `item.started`/`item.updated`, `item.completed type=reasoning`/`command_execution`/`file_change`/`mcp_tool_call`. No `Result` yield — codex has no equivalent; `process_stream_from_events` terminates on `turn.completed` | codex stream-json, codex event stream |
| **codex usage-limit regex** | Detection + reset-time extraction for codex's `turn.failed.error.message`. Trigger substring: `You've hit your usage limit`. Reset-time extraction: trailing `(?:or )?try again at <time>` clause in two shapes — same-day `H:MM AM/PM` and cross-day `Month Dth, YYYY H:MM AM/PM`. With `TZ=UTC` the extracted time is UTC by construction. No match (degraded `Try again later.` variant) → `UsageLimit(reset_time=None)`, falls through to next-hour `+2 min` default. Brittle by necessity: codex's `--json` flattens structured `RateLimitSnapshot` to a human-readable string (#778) | codex limit parser, codex reset regex |
| **codex sandbox + approval pins** | `CodexService.build_command` unconditionally emits `--sandbox danger-full-access` and `-c approval_policy=never`. Pycastle's container is the isolation boundary; inner codex sandbox would only fight existing container layout; `--ask-for-approval` does not exist on `codex exec` anyway. Not user-tunable in v1 | codex hardening flags, codex auto-approve |
| **codex effort flag** | Reasoning-effort for codex. No CLI flag exists on `codex exec`; passed via `-c model_reasoning_effort=<value>`. Accepted: `none | minimal | low | medium (default) | high | xhigh`. Overlaps with claude effort levels on `{low, medium, high, xhigh}` only — codex-only `{none, minimal}`, claude-only `{max}` — which is why `valid_efforts()` is per-service | codex --effort, reasoning effort |
| **codex auth verify** | Init wizard's codex-branch step: assert host `~/.codex/auth.json` exists; if not, print `"No codex credentials found at ~/.codex/auth.json. Run 'codex login' and re-run 'pycastle init'."` and exit non-zero. Matches `claude setup-token` precedent. Failure does not roll back claude's `.env` write from a prior step (init is idempotent) | codex login check, codex creds preflight |
| **service-selection prompt** | First prompt in init wizard: `"Which agent services do you want to use? [claude/codex/both]"` (default `claude` for back-compat). Drives per-service env-collection branches and Dockerfile template selection | init service prompt, agent selector |
| **Dockerfile template selection** | Init-time choice between two static templates in `src/pycastle/defaults/`: `Dockerfile.claude` (current behaviour) and `Dockerfile.claude-codex` (adds Node.js + `npm install -g @openai/codex`). Init picks based on service-selection answer and copies into `pycastle/Dockerfile`. `pycastle init --refresh` re-picks by walking config (no re-prompt) and overwrites. Refresh does NOT re-seed codex credentials | per-service Dockerfile, init Dockerfile pick |

## Test Anti-Patterns (Red Flags)

| Term | Definition | Why it's a problem |
| --- | --- | --- |
| **Verifying through external means** | Tests calling subprocess, querying DBs, or checking external state directly instead of testing through the service interface | Requires external tools in test env; failures reflect environment, not code bugs |
| **Mocking internal collaborators** | Tests mocking classes/functions within the same codebase rather than using DI | Brittle tests coupled to implementation; breaks on refactoring |
| **Testing private methods** | Tests calling `_`-prefixed functions or accessing private attributes | Private methods are implementation details; test public behavior |
| **Asserting on call counts/order** | Tests verifying internal function calls happened N times or in specific sequence | Breaks on refactoring; couples to implementation rather than behavior |
| **Test name describes HOW not WHAT** | Test names like `test_calls_git_config` instead of `test_setup_configures_git_identity` | Developers cannot understand intent from the name alone |

## Relationships

- **STAGE_OVERRIDES** has six entries, one per agent role; empty string for `model` or `effort` means CLI default. `preflight-issue` and `improve` entries are not tied to a phase in the 3-phase loop.
- **`load_config()`** is pure (no subprocess); `docker_image_name` is derived from sanitised `Path.cwd().name` if unset post-merge; model strings stored as-is (ADR 0002).
- **Layered config merge** applies at every CLI subcommand entry; every subcommand prints the resolved layer summary at startup (ADR 0003).
- **`PreflightCache`** is constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Its `asyncio.Lock` serialises concurrent callers; parallel `run_issue` queue, second-and-later see a populated slot. Iterations with no save-SHA consumer (idle without `--improve`, in-flight-only) never invoke `get_safe_sha()`. Single-slot suffices because HEAD only moves forward.
- **HITL verdict** is read by `get_safe_sha()` from the GitHub issue label after the preflight-issue agent completes; `ready-for-agent` → `PreflightAFK` (triggers preflight-fix path); `ready-for-human` → `PreflightHITL` (aborts iteration with `AbortedHITL`).
- **Planning skip** has two triggers, both routing to `implement_phase`: in-flight present, or preflight-fix AFK. **Implement skip** and **review skip** are checked inside `run_issue` before any worktree is created; both triggered by stage-done signal (ADR 0007).
- **Merge-time preflight skip** leaves conflict issues open; they become in-flight on the next iteration. Cache's single slot holds the broken-HEAD verdict — next iteration's first `get_safe_sha()` returns the same verdict without re-running or duplicating an issue.
- **Sequential mode**: after each merge, safe SHA re-pinned to new HEAD; failed issue stays `ready-for-agent` and queue continues; Merger remains a fallback.
- **Collision detection** uses a per-branch `asyncio.Lock` held in `implement_phase` for the full duration of each `run_issue` call.
- **`transient_worktree`** is used by `planning_phase` (plan-sandbox). **`managed_worktree`** is used by `merge_phase`, `run_issue` (delete=False), and improve mode (delete=True). Gitdir overlay stays at the `run_issue` call site, inlined around `managed_worktree`.
- Host mounts per container: host repo → RO at `/home/agent/repo`; worktree → RW at `/home/agent/workspace`; `<host-repo>/.git` → RW at `/.pycastle-parent-git`; on Windows, gitdir overlay → RO over `/home/agent/workspace/.git`.
- **Recently-closed filter** lives entirely inside `GithubService`; scoped to `get_open_issues` only. Other list-style reads (`get_open_issue_numbers`, `get_open_sub_issues`) are not filtered (ADR 0011).
- **GitHub authentication** is single-source: `GH_TOKEN` only (ADR 0004). Host code uses `urllib`; agent containers use `gh` (installed in Dockerfile per ADR 0021). `(owner, repo)` derived from `git remote get-url origin` — no `gh repo view`.
- **`Deps`** does not carry `env`; credentials are extracted in `main.py` and passed directly to `AgentRunner` at construction.
- Each phase module declares its own per-phase dependency protocol listing only its actual field accesses; `Deps` satisfies all structurally.
- Rich markup must never be embedded in a `StatusDisplay.print` message; colouring is via the `style` parameter only.
- All orchestrator-level terminal output is routed through `StatusDisplay.print()`; bare `print()` is not used while a `StatusDisplay` is active.
- During Work, the container runner injects the pre-rendered prompt (built upstream by `AgentRunner._build_prompt`, ADR 0012); passes decoded NDJSON + `on_turn` to `process_stream`, which raises `UsageLimitError(reset_time)` on 429. Setup and Pre-flight produce no console output — activity reflected only in body column.
- **Auto bug reporter** is *outside* the iteration layer and does not depend on `Deps`, `Logger`, or `StatusDisplay` — must function during config-load crashes. Cross-repo POST goes through `GithubService.create_issue_in(owner_repo, ...)`. The reporter is the *only* feature that talks to a GitHub repo other than the consumer's `origin`.
- **`AgentRunner`** constructs a `DockerSession` (via `build_volume_spec`) and a `ContainerRunner`, then orchestrates the three lifecycle phases. At session-build time it asks `AccountPool` for the active OAuth token and injects it as `CLAUDE_CODE_OAUTH_TOKEN`. On `UsageLimitError` it calls `pool.mark_exhausted(token, reset_time)` before re-raising.
- **Agent run kind** dispatch is reached only when the agent would otherwise spawn: stage-done signal check in `run_issue` decides Skip first; only not-skipped roles invoke `decide_agent_run_kind`. Skip and Resume share the same role-dir-driven mechanism (ADR 0007).
- A `Fresh` run begins with wipe-before-Fresh then launches `claude --session-id` with the role's normal prompt. `Resume` launches `claude --resume` with continuation prompt alone. Non-typed Resume crashes use the non-typed Resume retry, not relaunch-as-Fresh. Container env carries `CLAUDE_CONFIG_DIR=/home/agent/workspace/.pycastle-session/<role>/` in every case.
- Each `(role, worktree_path)` tuple gets a structurally unique derived session UUID, so parallel agents on different worktrees cannot collide on any Claude-internal session-id-keyed cache.
- **Broadened preservation rule** enforced in `managed_worktree.__aexit__` for every branched call site. When `delete_branch_on_teardown=True`, `teardown_worktree` and `delete_branch` skipped together when predicate holds. **Worktree reuse on enter** in `__aenter__` skips `_create_worktree` when path exists, branch matches, predicate holds. `transient_worktree` does not participate.
- On agent success, orchestrator runs **role session cleanup** *inside* the worktree context manager, after commit (Implementer/Reviewer) or merge (Merger) and before `__aexit__`. Issue worktree: cleanup clears session JSONL but leaves role dir as stage-done sentinel — predicate preserves the worktree. Improve-sandbox and merge-sandbox: cleanup calls `RoleSession(...).discard()` so dir is removed entirely, predicate goes False, sandbox torn down. Cleanup failure logged, never raised.
- `AgentTimeoutError` is preservation-worthy: single-agent phases run `timeout_retries` (default 1); on exhaustion, top-level `try/except AgentTimeoutError` (mirror of centralized usage-limit catch) converts to `AbortedTimeout(failed_role, worktree_path)`. Orchestrator continues to next iteration without sleep. Parallel `implement_phase` aggregates timeouts as `ImplementResult.errors`.
- Session resume and skip are two role-dir-driven mechanisms sharing the directory and predicate, differing only in predicate outcome (ADR 0007).
- **Merger preservation** matches Implementer/Reviewer: merge-sandbox preserved whenever broadened predicate holds.
- Session-storage gitignoring uses `.git/info/exclude` entries appended at orchestrator startup. Shared parent `.git` means exclusion applies in every worktree without per-worktree files.
- **`AccountPool`** is an in-memory ordered registry of `(name, token, exhausted_until)` tuples seeded once in `main.py` from `CLAUDE_CODE_OAUTH_TOKEN` (primary) and optional `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY` (secondary, preferred when present). Exposes `pick()`, `mark_exhausted(token, reset_time)` setting per-token wake-time, `has_available()`, `earliest_wake_time()`. Constructed in `main.py`, passed to `AgentRunner.__init__` as separate injectable. Failover at per-agent-run granularity. State in-memory only — process restart re-probes secondary optimistically (ADR 0005). Per ADR 0015: the class moves into `ClaudeService` as a private detail; env-var contract and dual-token semantics preserved byte-for-byte. Orchestrator-level failover moves up to the stage fallback chain.
