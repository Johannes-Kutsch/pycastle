# `PromptRenderer` with scoped placeholder tiers

A single `PromptRenderer` owns prompt rendering for a run. It is constructed once at `pycastle run` / `pycastle improve` startup, computes the **global placeholder pool** from `Config` and the `coding-standards/` files once and caches it, and at construction walks every variant of a new `PromptTemplate` enum to validate that each template's `{{TOKEN}}`s are members of `(global pool ∪ template.scope.placeholders)` — an unknown or out-of-scope token aborts the run before any agent starts. Each `PromptTemplate` variant carries a filename and a **prompt scope** (one of `PER_ISSUE`, `MERGE`, `PLAN`, `PREFLIGHT`, `IMPROVE_SCAN`, `IMPROVE_SESSION`, `RESUME`); each scope owns a fixed dynamic placeholder set. `agent_runner` holds the renderer and exposes a `RunRequest` carrying `(template: PromptTemplate, scope_args: dict[str, str])`; `render()` requires `scope_args.keys() == template.scope.placeholders` exactly and fails loud on missing, extra, or typo'd keys. The five iteration callsites stop building per-callsite `prompt_args` dicts and stop hard-coding `prompts_dir / "<filename>.md"`.

The trigger was issue #546: editing a prompt template required editing the test suite. `tests/test_default_prompts.py` pinned every shipped template's exact placeholder set via `_renders_without_error` / `_fails_without_<KEY>_arg` / `_expands_<KEY>_placeholder` triplets per template, plus end-to-end render tests for body/comments interpolation. Adding `{{NEW_THING}}` to `implement-prompt.md` forced a coordinated edit across the prompt, the implement callsite's `prompt_args` literal, and three or four lines in the test file. The coupling discouraged prompt iteration — the actual product surface — to satisfy a test suite that mostly re-asserted what the templates already say. The refactor moves the strictness from "tests pin the shipped template's args contract" to "the renderer ctor validates the shipped templates against a code-side scope registry," so prompt edits no longer touch tests at all. The whole of `test_default_prompts.py` is deleted; `test_prompt_pipeline.py` grows fixture-based machinery tests (substitution, missing-arg fail-loud, ctor rejects unknown token, ctor rejects out-of-scope token, exact-match scope_args, security regressions) that exercise the rendering pipeline against synthetic templates in `tmp_path`.

## Considered Options

- **Status quo (per-callsite dicts + per-template tests).** Rejected: this is the failure mode. The five iteration callsites each construct their own `prompt_args` literal; the test suite enforces that each shipped template's literal matches. Adding or removing a placeholder is a three-place edit. Three places agree, but only by ceremony — there is no single source of truth.
- **Universal args dict, no scopes.** One `dict[str, str]` is built at run start containing every placeholder any prompt could want and is passed unchanged to every `prepare_prompt` call. Rejected: makes no distinction between run-level constants (standards, labels) and contextual values (the issue currently being worked on). The merger has no `ISSUE_NUMBER`; passing an empty string would lie about what's in scope, and templates that accidentally reference `{{ISSUE_NUMBER}}` would render quietly. The "fail loud when a template references a placeholder its phase can't supply" property is the design property issue #546 implies.
- **Free-floating `build_*_args` helpers, callsites still merge.** A `build_global_args(cfg)` plus `build_per_issue_args(issue)` etc., assembled with `{**a, **b}` at each callsite. Rejected: keeps the per-callsite assembly that was the source of the friction. The refactor named one thing (the global pool) and centralized half the assembly — the other half is unchanged and the code reads as half-finished. Tests would still need to either pin per-template contracts (back to status quo) or validate against synthetic templates only (what the chosen design does).
- **Scope-tagged templates, but loose scopes (one per phase).** Five scopes (`PER_ISSUE`, `MERGE`, `PLAN`, `PREFLIGHT`, `IMPROVE`) instead of seven. The improve folder collapses into a single scope offering both standards and `IMPROVE_SHORT_SID`. Rejected: 01-scan and 02–04 actually have disjoint dynamic placeholder needs, and merging them lets `04-no-candidate-report.md` reference `{{TESTING_STANDARDS}}` without complaint. The chosen seven-scope split mirrors the boundary already present in `improve.py:113-117`. Likewise for `RESUME`: collapsing `_resume-prompt.md` into another scope's pool either lies about its context (it runs cross-phase) or lets it inherit dynamic placeholders it must not see.
- **Tight scopes, no global tier — every placeholder belongs to exactly one scope.** Rejected: forces the standards/labels/checks to be re-declared in every scope that needs them (PER_ISSUE, IMPROVE_SCAN both need standards; PREFLIGHT alone needs labels but a future MERGE prompt might too). The "constants vs context" axis is real; collapsing it inflates each scope's pool with run-level state that has no scope dependency. The two-tier model names the axis instead of denying it.
- **User-overridable `prompt_templates` registry in `pycastle/config.py`.** Rejected: adding a `(filename, scope)` entry without also adding a placeholder provider and a callsite is misleading — the registry would accept the entry but no code would ever load the template. The user-facing extensibility surface for prompts remains "edit `.md` files in `prompts_dir`," which the refactor preserves. If "swap a filename" turns out to be a real need, add a thin field for it (`implement_prompt_filename: str = "implement-prompt.md"`) — don't generalize preemptively.
- **Lookup by `(scope, role)` tuple or by `AgentRole` alone.** Rejected: `AgentRole` is the agent-output protocol's enum (which protocol the agent reports back with) and improve has multiple templates per role. Coupling template lookup to `AgentRole` either forces splitting `AgentRole` per template or leaves improve a special case. A dedicated `PromptTemplate` enum is the minimal honest model — there are seven templates, the enum has seven variants.
- **Renderer threaded through every `_*Deps` struct.** Rejected: the agent runner already owns the boundary "callsite hands me a request, I produce an agent invocation" — the renderer belongs there. Threading it through deps that don't otherwise care widens the dependency graph for no benefit.
- **Renderer as a module-level singleton.** Rejected: makes test isolation harder (parallel tests share state) and obscures the lifecycle (when is it constructed? when does validation fire?). Construction-time validation is load-bearing — making the lifecycle implicit hides it.
- **Lenient runtime: scope_args.keys() ⊆ scope.placeholders (subset, not equal).** A typo'd `ISSUE_NUMER` would still fail loud (because the template references `ISSUE_NUMBER` and it's missing), but a caller passing only a subset of the scope's placeholders would succeed if the specific template happened not to reference the omitted ones. Rejected: the contract was chosen as exact match because the scope set is small and uniform — `PER_ISSUE` callers always have all five values available — and exact match makes the call site read the same regardless of which `PER_ISSUE` template is being rendered. The minor cost (computing `BRANCH` even for a hypothetical per-issue template that doesn't use it) is negligible.
- **Drop runtime checks; rely on startup validation alone.** Rejected: startup validates the *templates* against the scope sets; runtime validates the *callers* against the scope sets. Both are needed because the bugs they catch are different — startup catches "template uses a token the renderer can't fill"; runtime catches "caller forgot to compute a value the template expects." Skipping runtime would let `_render` silently raise about a missing legitimate key when the real bug is a typo'd scope_args key.
- **Keep `test_default_prompts.py` in a thinned-out form (one render per template).** A single regression guard per shipped template that asserts no leftover `{{...}}` after rendering against its scope's expected args. Rejected: still couples prompt edits to test edits — adding a `{{NEW_THING}}` inside a scope still requires updating the test fixture's args dict. The chosen design moves all template-vs-scope-vs-args validation into the renderer ctor, so the tests cover the mechanism (against synthetic templates) and not the shipped templates' contents.
- **Two-tier placeholders + seven scopes + `PromptTemplate` enum + `PromptRenderer` constructed at run start + agent runner ownership + exact-match runtime + delete `test_default_prompts.py`.** The chosen design.

## Consequences

- A new `PromptTemplate` enum lists every shipped template (`IMPLEMENT`, `REVIEW`, `MERGE`, `PLAN`, `PREFLIGHT_ISSUE`, `IMPROVE_SCAN`, `IMPROVE_PRD`, `IMPROVE_ISSUES`, `IMPROVE_NO_CANDIDATE`, `RESUME`); each variant carries `filename: str` and `scope: Scope`. Adding a template is a one-line enum addition + a placeholder provider for the scope (if new) + a callsite that uses it.
- A new `Scope` enum lists the seven scopes; each variant carries a `placeholders: frozenset[str]` of its dynamic placeholder pool.
- `PromptRenderer.__init__(cfg)` reads the five `coding-standards/*.md` files (replacing the per-callsite `load_standards` calls in `iteration/implement.py:77` and `iteration/improve.py:113`), composes the ten globals from `cfg`, then iterates `list(PromptTemplate)`, parses each file's `{{TOKEN}}`s, and raises if any token is outside `(global ∪ template.scope.placeholders)`. Construction is the validator. Failure aborts `pycastle run` / `pycastle improve` with a message naming the offending template and token.
- `PromptRenderer.render(template, scope_args)` asserts `set(scope_args) == template.scope.placeholders` and raises a typed error on mismatch — message names the offending keys (missing, extra, or both). On success, preprocesses `` !`shell` `` expressions, merges scope_args with cached globals, substitutes `{{placeholders}}`, returns the rendered string. The substituted-content-is-inert property from `prompt_pipeline._preprocess` is preserved.
- `RunRequest`'s `prompt_file: Path` and `prompt_args: dict[str, str] | None` fields are replaced by `template: PromptTemplate` and `scope_args: dict[str, str]`. `agent_runner.run()` calls `self._renderer.render(request.template, request.scope_args, exec_fn)` instead of `prepare_prompt(request.prompt_file, request.prompt_args, exec_fn)`. `container_runner.work()` no longer takes `prompt_file` / `prompt_args`; it takes the rendered string from `agent_runner`. The `_RESUME_PROMPT_FILE` direct read at `container_runner.py:17` becomes a `render(PromptTemplate.RESUME, {})` call inside the renderer's normal path — resume prompts join the validator's walk and gain protection against future typo'd placeholders.
- The five iteration callsites (`iteration/implement.py:78-86`, `iteration/preflight.py:79-86`, `iteration/merge.py:112-119`, `iteration/planning.py:40`, `iteration/improve.py:113-117`) drop their `prompt_args` literals and their `prompts_dir / "<filename>.md"` constructions. Each becomes a `RunRequest(template=PromptTemplate.X, scope_args={...})`.
- `prompt_pipeline.py` is reshaped: `prepare_prompt`, `load_standards`, the `_STANDARDS_FILES` dict, and `PromptRenderError` move into `PromptRenderer` as private methods or are inlined; the module's public surface becomes `PromptRenderer`, `PromptTemplate`, `Scope`, and the typed error classes.
- `tests/test_default_prompts.py` is deleted (370 lines, every test). `tests/test_prompt_utils.py` is renamed `tests/test_prompt_renderer.py` and grows fixture-based tests against synthetic templates in `tmp_path`: substitution succeeds, missing-arg raises, extra-arg raises, typo'd-arg raises, ctor rejects unknown token, ctor rejects out-of-scope token, ctor accepts globals everywhere, scope_args contract is exact-match, the two existing security regressions (`test_arg_value_containing_shell_token_is_not_executed`, `test_template_shell_expr_runs_arg_shell_token_stays_inert`) keep working unchanged because they already use fixture templates.
- Prompt edits inside an existing scope (adding/removing a `{{TOKEN}}` from a shipped template, where the token is in the scope's pool) require zero test changes. Validation fires at the next `pycastle run` / `pycastle improve` startup. This is the property the refactor exists to create.
- Adding a new placeholder requires (a) deciding global vs scope, (b) adding it to the global pool or a scope's `placeholders` frozenset, (c) supplying it from `Config` (global) or from the callsite's `scope_args` (scope), and (d) using it in a template. The refactor makes (a) explicit and (b) a one-line edit; (c) and (d) are unchanged from today's structure.
- Moving a placeholder between scopes, or moving a template between scopes, is a deliberate edit to the registry — the only remaining "edit two places" coupling is "template scope ↔ callsite that supplies its scope_args," which is unavoidable because they describe the same thing.
- `pycastle build` and `pycastle init` do not construct a renderer. A broken template will not block them. Validation fires only when prompts are about to actually run.
- `CONTEXT.md` adds entries for **placeholder provider**, **global placeholder pool**, **scope placeholder pool**, **prompt scope**, **`PromptTemplate`**, **`PromptRenderer`**; updates **placeholder** to name the two tiers; replaces the **prompt pipeline** entry with **`PromptRenderer`**; and rewrites the Reviewer entry to reference `PromptTemplate.REVIEW` instead of enumerating the placeholder set inline.
