# Centralized `UsageLimitError` catch at the iteration boundary

`run_iteration` wraps its body in a single `try/except UsageLimitError â†’ return AbortedUsageLimit(reset_time=err.reset_time)` and the per-phase try/excepts that previously sat inside the Preflight, Plan, Improve, and Merge dispatcher arms are deleted. `phase_row` learns a third `shutdown_style="interrupted"` and its `finally` clause distinguishes `UsageLimitError` (paint `"interrupted"` and re-raise) from other exceptions (paint `"failed"`/`"error"` and re-raise). `implement_phase` keeps its bespoke `usage_limit_hit: bool` / `usage_limit_reset_time: datetime | None` aggregation on `ImplementResult` because it runs N parallel agents and must commit completed siblings before unwinding â€” the structural exception is preserved, the four single-agent phases are unified.

The trigger was issue #538: the Improve agent hit the usage limit, `UsageLimitError` propagated out of `improve_phase`, and the orchestrator crashed instead of taking the standard sleep / account-failover path. `git log` shows two prior fixes for the identical class â€” `2d55eaf` ("catch UsageLimitError in plan phase to enable account failover") and `8e94c3b` ("fix unhandled UsageLimitError in post-merge preflight run_issue path") â€” confirming this as a recurring foot-gun rather than a one-off. The shape of the bug is always the same: a new agent-running phase is added to the dispatcher, and the author forgets to mirror the per-phase `try/except UsageLimitError â†’ AbortedUsageLimit` boilerplate. The cost is silent until production hits the limit. Latent bug at `iteration/__init__.py:182` (`await merge_phase(completed, deps)`) is fixed by the same change without separate plumbing.

## Considered Options

- **Status quo (per-phase try/except).** Rejected: this is the failure mode. The handlers are byte-identical (`row.close("finished"); return AbortedUsageLimit(reset_time=err.reset_time)`), there is no per-phase variation in behaviour, and forgetting to add one is a silent crash. Three recurrences in tracked history, each found by users hitting the limit in production.
- **Structural sum type per phase (`PhaseResult | UsageLimitHit` returned from every `*_phase`).** Rejected: makes "forgot to handle" a type error rather than a runtime error, but preserves the per-phase boilerplate and forces every phase to thread `reset_time` through its return contract. The duplication being avoided is exactly what this option keeps. Also asymmetric: `implement_phase` already has a richer result struct with completed/errors aggregation, and shoehorning a uniform variant into all five phases collapses information that genuinely differs.
- **Convert at `agent_runner.run` (every caller pattern-matches a `UsageLimitHit` sentinel result).** Rejected: invasive and propagates outward. Every caller of `agent_runner.run` â€” including non-iteration callers â€” would acquire a new variant to match. The dispatcher would still need to distinguish "this phase reported usage limit" from "this phase succeeded," reintroducing per-phase handling at a different layer. `implement_phase` already does roughly this internally, and the pattern doesn't generalise â€” propagating outward, not inward, is the right direction.
- **Lint/AST rule that fails CI when an `agent_runner.run` call site lacks usage-limit handling.** Rejected: tests structure rather than behaviour; brittle to refactors that move call sites or introduce indirection; doesn't actually prevent the bug, only shifts it from production into CI. The structural fix below is cheaper.
- **Top-level `try/except` in the orchestrator's iteration loop, removing `AbortedUsageLimit` from `IterationOutcome`.** Rejected: `IterationOutcome` is the contract between iteration and orchestrator, with four named ways an iteration can end (`Continue`, `Done`, `AbortedHITL`, `AbortedUsageLimit`). Pushing one of those out as an exception breaks symmetry â€” three outcomes are values and one is a control-flow side-channel. Tests of `run_iteration` would have to know which is which. Catching at the iteration boundary keeps the contract intact.
- **`phase_row` swallows `UsageLimitError` entirely.** Rejected: couples a generic UI primitive to one specific domain error and hides an exception the iteration boundary needs to see. The chosen design only teaches `phase_row` a paint style; the exception still re-raises.
- **`phase_row` paints `"failed"` on every exception including `UsageLimitError` (no new style).** Cheaper but mildly contradictory in the live display: a row painted `"failed"` directly above the orchestrator's `"sleeping until 16:00"` message reads as a phase malfunction when the phase actually didn't malfunction â€” an external constraint interrupted it. The third style is one elif; the UX honesty is worth it.
- **Single try/except in `run_iteration` + `phase_row` `"interrupted"` style + keep `implement_phase` aggregation + parametrized regression test per phase.** The chosen design.

## Consequences

- The per-phase `try/except UsageLimitError` blocks at `iteration/__init__.py:71-75` (Preflight) and `iteration/__init__.py:116-120` (Plan) are deleted. The dispatcher arms call `await preflight_phase(deps)` and `await planning_phase(deps, sha, open_issues)` bare. The `DispatchImprove` arm at `iteration/__init__.py:102-104` and the `await merge_phase(completed, deps)` call at `iteration/__init__.py:182` gain no per-arm wrapping â€” they share the same top-level catch.
- `run_iteration`'s body is wrapped: `try: ... except UsageLimitError as err: return AbortedUsageLimit(reset_time=err.reset_time)`. The catch sits at the function level so every dispatcher arm and every helper called from it is covered.
- `phase_row.shutdown_style` accepts `"interrupted"` in addition to `"success"`, `"warning"`, `"error"`. The async-context-manager `finally` at `_rows.py:33-35` becomes: `if not row._closed: if exc_type is UsageLimitError: status_display.remove(caller, "usage limit reached", "interrupted") else: status_display.remove(caller, "failed", "error")`. The exception is re-raised either way.
- `StatusDisplay.remove`'s `shutdown_style` parameter and any `PlainStatusDisplay` rendering acquires the new `"interrupted"` style (colour TBD by the renderer; semantically distinct from success/warning/error). Phase rows for Preflight, Plan, Improve, Merge paint `"interrupted"` on usage-limit hits. The `agent_row` context manager is unchanged â€” agent rows already paint `"failed"` on any exception and the orchestrator-level "interrupted" semantics live one level up.
- `implement_phase`'s `ImplementResult.usage_limit_hit: bool` and `usage_limit_reset_time: datetime | None` fields stay. The dispatcher arm at `iteration/__init__.py:144-146` continues to pattern-match on the result struct rather than catching an exception, because parallel-runner aggregation has work to *commit* (completed siblings) before the iteration unwinds â€” rethrowing after merge would be either lossy or ceremony-heavy.
- The merge-phase latent bug is fixed silently by the same change. No separate plumbing or issue-shaped work is required.
- A parametrized regression test in `tests/test_iteration.py` injects `UsageLimitError` at each single-agent phase entry point (Preflight, Plan, Improve, Merge) and asserts `run_iteration` returns `AbortedUsageLimit(reset_time=...)` with the reset time threaded through. Adding a sixth single-agent phase becomes a one-line parameter addition â€” visible in code review. The existing implement-phase usage-limit test stays as-is (it tests a different surface â€” the structural result struct, not the exception path).
- ADR 0005's reference to "the existing `UsageLimitError` â†’ `AbortedUsageLimit` plumbing is reused unchanged" remains accurate at the orchestrator level (`AbortedUsageLimit` is still the value the orchestrator's match arm consumes). The plumbing is now centralized inside `run_iteration` rather than scattered across dispatcher arms, but the contract crossing into the orchestrator is unchanged.
- Future maintainers adding a phase that calls `agent_runner.run` no longer need to remember `try/except UsageLimitError` â€” the top-level catch covers them. This is the property the refactor exists to create.
