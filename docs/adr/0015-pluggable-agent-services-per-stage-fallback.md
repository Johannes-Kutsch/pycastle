# Pluggable agent services with per-stage fallback

Pycastle currently hard-codes the Claude CLI in `ContainerRunner._build_claude_command` and inlines claude-specific concerns throughout (`CLAUDE_CODE_OAUTH_TOKEN` injection in `AgentRunner._build_session`, claude stream-json parsing in `process_stream`, claude session-id resume in `RoleSession.session_uuid()`, claude account pooling in `AccountPool`). Anthropic is tightening Pro/Max usage limits, so pycastle must be able to fall back to a different agent service (Codex, eventually local-GPU) when claude is exhausted, and to pick a different service per stage when that produces better results. We introduce an `AgentService` abstraction whose seam sits at the *streaming-execution* layer (the service owns command construction, env injection, wire-format parsing, and its own resume state; `ContainerRunner` keeps the cross-cutting concerns: prompt-file write, log persistence, idle-timeout, `on_turn`/`on_tokens` callbacks). Services are declared by name in config; each stage names a preferred service plus an optional fallback `(service, model, effort)` triple. Service-internal account pooling (today's `AccountPool`) becomes a private detail of `ClaudeService`. Cross-service handoff mid-stage works via WIP commits: on `UsageLimitError` / `AgentTimeoutError` the orchestrator commits any dirty changes as `WIP: <role> #N - interrupted` with the agent's last assistant turn as commit body, then unwinds; the next dispatch's prompt is augmented with a "this branch carries WIP from an interrupted run, inspect and continue" clause only when WIP commits are present *and* the picked service has no resume state of its own for this worktree.

## Considered Options

- **CLI-builder seam (service returns command string + env vars).** Rejected: forces every future service to mimic claude's `stream-json` wire format and inherits claude-shaped session-uuid/`--resume` semantics. Codex CLI and a local-model service do not have those properties naturally.
- **Full-agent-run seam (service owns container image, auth pool, returns `AgentOutput` directly).** Rejected: duplicates the docker/session/log/timeout machinery per service and makes ADR-0006 session-resume-across-container-teardown hard to keep working uniformly. Wider than needed for v1; can widen later when local-GPU forces the issue.
- **Streaming-execution seam (the chosen design).** Service yields normalized turn events (`AssistantTurn(text)`, `Tokens(count)`, `Result(text)`, `UsageLimit(reset_time)`) into a shared coordinator that owns log persistence, idle-timeout, prompt-file lifecycle, `on_turn`/`on_tokens` callbacks. Service owns command construction, env injection, wire-format parsing, and its own resume contract.
- **Rich `ParsedTurn` vocabulary (`ToolUse`, `ToolResult`, `Reasoning`, etc.).** Rejected: invents contract surface for nonexistent consumers; the orchestrator only consumes `on_turn(text)`, `on_tokens(n)`, final result text, and the 429 signal. Expand vocabulary when the first feature actually needs per-tool insight.
- **Raw-event opaque blob + per-service extractor callback.** Rejected: equivalent to the 4-event vocabulary but with the type system turned off.
- **Service resume modelled as a shared on-disk shape with parameterized env-var name.** Rejected: silently downgrades resume-less services to always-FRESH, which causes real bugs (improve-phase re-dispatch loops, mid-run retries that should resume). Service must own the resume contract.
- **Cross-service mid-stage handoff via transcript replay (inject last K assistant turns into B's first prompt).** Rejected: invents a cross-service transcript format ahead of needing one and pays token cost every handoff. WIP commit body carries the most-recent assistant turn already; the file diff carries the intent.
- **Cross-service handoff by sticky-service (once stage S starts on service A, never switch mid-stage).** Rejected: defeats the point of multi-service failover — when claude exhausts mid-implement and codex is idle, sleeping is exactly what we want to avoid.
- **Discard A's uncommitted changes on cross-service FRESH.** Originally proposed and rejected: would lose near-complete implement work, which is precisely the scenario session-resume exists to protect against today.
- **Always-on auto-commits during the run (agent-driven).** Rejected: every agent every run pays prompt-complexity cost; granularity decided by the model rather than the orchestrator. WIP commit on unwind is host-driven and zero-cost in the happy path.
- **Global ordered service-priority list, per-stage `service=` shifts the starting position.** Rejected mid-grilling in favor of per-stage `fallback=` because fallback often wants a different `(service, model, effort)` triple than primary — e.g. codex's best model for "implement" is not the same as for "review". Nested `StageOverride(fallback=StageOverride(...))` localises the failover decision in the stage config and allows N-level chains for free.
- **`start_sha` file as a stage-in-progress marker carrying the squash-anchor SHA.** Proposed mid-grilling and rejected: the squash anchor is derivable from git history alone (`git log <branch_base>..HEAD --format=%s` minus commits matching `^WIP: <role> #N -`), and the in-progress signal can ride on a `pending` placeholder file or on the presence of any service subdirectory. Drops to zero new on-disk artifacts vs today.
- **Per-stage `services=[...]` list as the failover chain (the chosen design's flatter cousin).** Rejected: provided no expressive gain over `fallback=` nested override while making N-level chains look like first-class configuration, encouraging baroque fallback trees nobody will actually maintain.

## Consequences

- **`AgentService` protocol** lives at the streaming-execution seam. Methods (final names TBD): `name`, `is_available(now)`, `next_wake_time()`, `mark_exhausted(reset_time)`, `state_dir_relpath(role, namespace) -> str | None`, `is_resumable(state_dir: Path) -> bool`, `build_env(state_dir_container_path) -> dict[str, str]`, and an async `run(...)` yielding the 4-event `ParsedTurn` stream. Services live under `src/pycastle/services/` alongside `ClaudeService`; new services (`CodexService`, eventually a local-model service) implement the protocol.
- **Worktree marker layout** changes minimally:
  - `.pycastle-session/<role>/[<namespace>/]` (existing) — its presence still flags "stage started." Empty dir = stage done; non-empty = stage in progress (or interrupted).
  - `.pycastle-session/<role>/[<namespace>/]<service>/` (new) — per-service resume state. Replaces the prior pattern of writing claude session files directly into the role dir.
  - No `start_sha` file. Stage-completion squash walks back from HEAD skipping `^WIP: <role> #N -` subjects.
- **`RoleSession` splits along two axes.** Stage-completion (per worktree+role+namespace, service-agnostic) stays as today: `is_done()` / `mark_done()` / `start_fresh()`. Service resume state (per worktree+role+namespace+**service**) moves into the service via `state_dir_relpath` + `is_resumable`. `is_stage_done_for(worktree, role)` remains a pure-filesystem check — no `git_svc` argument, no module-boundary churn.
- **Conditional WIP-aware prompt clause.** The role prompt grows a clause only when both conditions hold: (a) WIP commits exist on the branch (`git log` grep for `^WIP: <role> #N -`), and (b) the picked service has no resume state in this worktree (`is_resumable() == False`). Same-service resume gets neither the clause nor the WIP-commit narrative — claude already has its conversation.
- **WIP-commit lifecycle.** On `UsageLimitError` / `AgentTimeoutError` propagation, before container teardown: `git add -A && git commit -m "WIP: <role> #N - interrupted"` with the agent's last assistant turn (extracted from `logs/<agent>.log`) as the commit body. `AgentFailedError` and unexpected exceptions do **not** WIP-commit (failed-agent dirty trees should not be enshrined as handoff state).
- **Squash-at-stage-end.** When the agent emits `<commit_message>` and pycastle is ready to create the canonical commit, walk back from HEAD skipping WIP commits to find the squash anchor, then `git reset --soft <anchor> && git commit -m "Implement #N - <msg>"`. Same-service flows that never hit usage-limit produce zero WIP commits and squash a no-op.
- **FRESH-start hygiene.** Every FRESH agent dispatch runs `git restore --staged --worktree . && git clean -fd` against the worktree before launching the service. Idempotent; protects against leftover dirty state from a crashed prior attempt that did not WIP-commit.
- **Config shape.** `StageOverride` gains `service: str` and `fallback: StageOverride | None`. Top-level `services: dict[str, AgentService]` and `default_service: str` are added. Empty `service=""` resolves to `default_service`. Today's `(model, effort)` fields are unchanged.
- **Dispatch loop.** Per agent run: ask `services[stage.service].is_available(now)` → if yes, use `(stage.service, stage.model, stage.effort)`. Else if `stage.fallback` is set, ask `services[stage.fallback.service].is_available(now)`. Else sleep until `min(next_wake_time across involved services)` and retry. Snap-back is automatic — every dispatch re-evaluates primary first.
- **Service-internal failover stays internal.** Claude's two-OAuth-token pool moves from `AccountPool` (currently constructed in `main.py` and threaded into `AgentRunner`) to a private detail of `ClaudeService`. `ClaudeService.is_available()` returns True iff at least one of its internal accounts is non-exhausted. Account-level snap (token A → token B) is invisible to the orchestrator.
- **`AbortedUsageLimit` arm in `orchestrator.run`** stops asking the (now-removed-from-this-layer) `account_pool` whether to switch or sleep. It asks the *registry* of services configured for the current stage (via `cfg.<stage>_override`), and only sleeps when both primary and fallback report unavailable.
- **`process_stream` and `_HANDLERS` survive unchanged.** The 4-event vocabulary streams `AssistantTurn(text)` into the existing role handlers, which already operate on agent prose — no claude wire format leak. The `_check_usage_limit` regex and `_extract_turn` JSON path become claude-private code inside `ClaudeService`.
- **ADR-0006 amendment.** Session resume now scoped to the picked service. The "non-typed Resume retry" mechanism stays per-service (claude's amendment behavior is preserved); other services define their own retry policy or omit it. The `--resume <SESSION_UUID>` flag and `CLAUDE_CONFIG_DIR` env var become claude-private.
- **ADR-0007 amendment.** Stage-done signal (empty role dir) is preserved verbatim. The role dir's *contents* shift from "claude session files" to "one or more `<service>/` subdirectories" — but the stage-done predicate (`is_dir() AND not any(files)`) is unchanged in semantics.
- **ADR-0005 supersession (partial).** The `AccountPool` class moves from an orchestrator-level construct into `ClaudeService`'s private internals. The dual-token failover behavior is preserved byte-for-byte at the user-visible level; `CLAUDE_CODE_OAUTH_TOKEN` / `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY` env vars unchanged.
- **`pycastle init` and env collection** must be revisited per-service in a follow-up — out of scope for this ADR. v1 ships with `ClaudeService` only (no behavior change) plus the seam in place; the second service (Codex) lands as a separate change.
