# Session resume across container teardown

> **Amended 2026-05-10 (#640).** The fail-soft fresh fallback originally chosen here has been removed. Non-typed exceptions on a Resume run are now handled by a single in-call retry against the same session, then converted to `FailedOutput(failure_class="non_typed_crash")` which routes through the existing **Failure-Analysis agent** path. Sections describing the removed behaviour have been rewritten in place; the rejection reasoning for the alternatives below is preserved as-is (the alternatives remain rejected for the original reasons). See the *Considered Options* entry "Re-launch as Fresh on any non-typed Resume crash (the original fail-soft fresh fallback)" and the *Consequences* entry "non-typed Resume retry" for the current design.

When an agent (Implementer, Reviewer, Merger) is interrupted mid-task — almost always by an account-pool sleep when every Claude account is exhausted, but also by idle timeout, missing `<commit_message>` tag, container crash, or Ctrl-C — the orchestrator preserves the worktree and tears down the container. On the next iteration a fresh agent is spawned with no memory of the analysis the previous agent already did: it re-reads git state, re-explores the codebase, and re-derives the plan it was partway through. The work product survives via git (committed work) and via implement-skip / review-skip (commit-prefix detection bypasses the agent entirely). But the *in-progress reasoning* — the half-formed plan, the files already opened and understood, the rejected approaches about to be discarded — is dropped on every interruption. In a window where the pool sleeps once or twice, this can double the token cost of a single issue. We are introducing a session-resume primitive: every `claude` invocation runs with `--session-id <SESSION_UUID>` (Fresh) or `--resume <SESSION_UUID>` (Resume), where `SESSION_UUID` is deterministically derived per `(role, worktree_path)` and `CLAUDE_CONFIG_DIR` is set to a per-role subdir inside the agent's worktree. A non-empty role session dir at run-dispatch time is the resume signal. On Resume, the prompt sent to Claude is the shared continuation prompt alone (the role prompt is already in the resumed conversation history). On any non-typed failure of `--resume`, the call retries once against the same session — preserving history — and on a second non-typed exception returns `FailedOutput(failure_class="non_typed_crash")`, which routes through the existing Failure-Analysis path so a genuinely-corrupt session is surfaced as a triageable `bug` issue rather than silently wiped (amended #640).

## Considered Options

- **Status quo (always restart from scratch, trust git state).** Rejected: implement-skip / review-skip already covers committed work, but in-flight reasoning is exactly what those mechanisms cannot preserve. The motivating case — pool sleep mid-Implementer — discards the codebase exploration the agent already paid for. The whole point of the primitive is to catch the cost that the existing skip mechanisms structurally cannot.
- **Replace implement-skip / review-skip with session-resume.** Rejected: skip is durable across worktree removal because it reads the branch ref; resume is ephemeral with the worktree by design. Orphan sweep, manual cleanup, or any process that touches `pycastle/.worktrees/` would silently demote a completed phase to "redo" if the only completion signal was worktree-internal. Skip and resume are orthogonal: skip catches *committed* work (durable), resume catches *in-flight* work (ephemeral). Both stay.
- **Host-side volume mount for session storage** (`<host>/.pycastle/sessions/<branch>/<role>/` mounted into the container at `CLAUDE_CONFIG_DIR`). Rejected: introduces a second cleanup path (sessions and worktrees can become out of sync), a new mount surface to reason about, and breaks the symmetry where every artifact lives inside a worktree. The existing worktree-preserve / orphan-sweep machinery transparently handles preservation and cleanup of inside-worktree storage with no new infrastructure; a host-side mount would re-implement those same lifecycle hooks and inevitably drift.
- **Random UUID per agent run, persisted in a `<worktree>/.pycastle-session/<role>/uuid` file.** Rejected: `CLAUDE_CONFIG_DIR` already isolates session storage per `(worktree, role)` tuple, so within any one config dir at most one session ever exists. A randomly-generated UUID adds a file to read/write on every run and one more piece of state to keep consistent with the dir, while solving no real problem.
- **Three fixed per-role UUID constants** (`IMPLEMENTER_SESSION_ID`, `REVIEWER_SESSION_ID`, `MERGER_SESSION_ID`). Rejected in favor of derivation: with parallel agents on different worktrees all sharing the same constant, any unknown Claude-internal cache keyed on `session-id` (rather than fully isolated by `CLAUDE_CONFIG_DIR`) becomes a silent corruption surface that fail-soft cannot catch. A smoke test could verify isolation today but would have to be re-run on every Claude upgrade. Derivation eliminates the question.
- **Single fixed UUID for all roles.** Rejected for the same reason as three fixed constants, plus the role-identification benefit in log lines.
- **Derived UUID per `(role, worktree_path)`** — the chosen design. `uuid.uuid5(uuid.uuid5(NAMESPACE_DNS, "pycastle.<role>"), str(worktree_path.resolve()))`. Each agent run on each worktree gets a structurally unique UUID; cross-stream leakage is impossible by construction; no UUID file is needed; the role is still identifiable in log lines via the role-namespace seed.
- **Trigger resume only on `UsageLimitError`.** Rejected: the existing skip mechanism is signal-driven (commit prefix), not cause-driven, and inheriting that consistency matters. Filtering by interruption cause would require threading the cause through the cancellation path, a new coupling that buys nothing — fail-soft already bounds the worst case (a resumed bad-conversation degrades to a Fresh start). Resume fires on any preservation-worthy interruption.
- **Localized `<worktree>/.pycastle-session/.gitignore` containing `*`.** Rejected: empirically a `*` rule inside the directory does not hide the directory itself from `git status` — git still reports the dir as untracked because no enclosing `.gitignore` excludes it. An agent following a "keep tree clean" instruction could therefore `git add` the session dir and accidentally commit session data.
- **Auto-append `/.pycastle-session/` to the consuming project's root `.gitignore`.** Rejected: writes a tracked file in the consuming repo, requires a migration story for existing projects, and couples `pycastle init` to session storage.
- **Idempotent append to `<repo>/.git/info/exclude`** — the chosen design. `.git/info/exclude` is a local, never-committed exclusion file that is shared across all worktrees of the repo (worktrees share the parent `.git`). The orchestrator startup path (alongside `prune_orphan_worktrees`) appends `.pycastle-session/` and `.claude/` to this file if not already present. No tracked file is touched; no `pycastle init` step is added; the consuming project sees no migration. The defensive in-worktree `.gitignore` files are dropped — the exclude file replaces them.
- **Agent-side completion flag file** (the agent writes a `done` marker that the orchestrator picks up next iteration to drive cleanup). Rejected: the orchestrator already has a sharper completion signal — it receives a typed `AgentOutput` and knows the moment the commit/merge succeeded. A flag file duplicates that signal with no atomicity guarantee, and the second signal will inevitably drift from the first. Cleanup tied to the orchestrator's existing success boundary keeps one canonical "agent succeeded" check.
- **Reset the worktree to safe SHA on fail-soft Fresh recovery.** Rejected: uncommitted changes are exactly the artifact `_agent_worktree` was preserving in the first place; `git reset --hard` to recover from a corrupt session destroys the work-in-progress. The sane fallback is "Fresh launch on the dirty worktree, with the continuation prompt prepended to the role prompt to warn the agent about pre-existing uncommitted state."
- **Keep `decide_agent_run_kind` returning `Skip | Resume | Fresh`.** Rejected in favor of `Resume | Fresh`: Merger has no commit-prefix Skip path, so the third variant is unreachable for one of the three roles. Implementer/Reviewer's existing inline Skip branching in `run_issue` already routes by role-specific consequence (skip implementer block vs. skip both blocks), so a unified Skip return would still need to be re-routed at the call site. Decoupling the dispatch from Skip simplifies both the function and the caller.
- **Detect fail-soft via a typed `ResumeStartupError` raised when the stream yields zero turns.** Rejected: distinguishing "session unreadable" from "session loaded then crashed" via stream introspection is fragile (Claude can emit a partial turn before crashing). Allowlist propagation is the chosen detection mechanism — but the consequence has changed (see next entry).
- **Re-launch as Fresh on any non-typed Resume crash (the original fail-soft fresh fallback).** Originally chosen, **rejected on amendment (#640)**: the original argument was that "worst case we re-do work, which is what Fresh does anyway." In practice the worst case is the *common* case — a genuinely-unloadable session is rare; the common-case non-typed crash (transient docker hiccup, malformed stream frame, unexpected `RuntimeError`) was triggering a silent history wipe and turning a recoverable interruption into the exact "lose all in-progress reasoning" failure the primitive exists to prevent. The current design instead retries once against the same session and, on a second failure, converts to `FailedOutput(failure_class="non_typed_crash")`. The Failure-Analysis agent then files a `bug` + `needs-triage` issue with a Recovery section instructing the human to wipe the session dir manually if they suspect transcript corruption. A genuinely-corrupt session is now loud (a triageable issue) rather than silent (a Fresh re-run with zero history); the "loop wedge" risk is bounded because pycastle returns `AbortedAgentFailure` and exits cleanly rather than retrying forever.
- **Narrow the Merger preservation set to `UsageLimitError` only.** Rejected: under the broader rule, any preservation-worthy interruption keeps the merge-sandbox so the Merger can resume mid-conflict. Real merge conflicts touch ~1–2 branches, so resuming partial conflict-resolution thinking is exactly the savings we want; restarting the Merger from a fresh prompt that lists "unmerged branches" still re-derives the same conflict reasoning the prior run had partway through.
- **Existing dirty-tree-or-usage-limit preservation rule.** Rejected as the sole rule: a session-resumable role dir is a third independent reason to preserve a worktree, even on a clean tree with no `UsageLimitError`. Without this third trigger, an `AgentTimeoutError` on a clean tree would tear down the worktree and discard a recoverable session. The new rule is `dirty OR usage_limit OR session_resumable`.

## Consequences

- Every agent container run launches with `CLAUDE_CONFIG_DIR=/home/agent/workspace/.pycastle-session/<role>/` and one of `--session-id <SESSION_UUID>` (Fresh) or `--resume <SESSION_UUID>` (Resume), where `SESSION_UUID` is `uuid.uuid5(uuid.uuid5(NAMESPACE_DNS, "pycastle.<role>"), str(worktree_path.resolve()))`. The role and worktree are visible in the UUID derivation chain; no constants live in the codebase.
- `decide_agent_run_kind(role, session_dir_present) → Resume | Fresh` is a pure function unit-tested over its full decision space. The `Skip` path stays inline in `run_issue` as today (commit-prefix branching); the dispatch function is only invoked when the agent would otherwise spawn.
- The prompt-shape contract is:
  - **Fresh** — `/tmp/.pycastle_prompt` contains only the role prompt (rendered with role-specific args).
  - **Resume** — `/tmp/.pycastle_prompt` contains only the continuation prompt; the role prompt is already in the resumed conversation history and re-sending it would force the agent to re-do the very analysis we are preserving.
  - The previously-defined **Fresh-after-fail-soft** shape (continuation prompt + role prompt) was removed alongside the fail-soft fresh fallback (#640); its only consumer no longer exists.
- The worktree-preservation rule broadens to `dirty OR usage_limit OR session_resumable`. The `session_resumable` predicate is `role_dir.is_dir() AND any(role_dir.rglob("*"))` — Claude has actually written session content, not just an empty wipe-before-Fresh dir. The same predicate is used both inside `_agent_worktree` / `branch_worktree` (preservation gate on `__aexit__`) and at `decide_agent_run_kind` (Resume vs Fresh).
- `_agent_worktree.__aenter__` and `branch_worktree.__aenter__` gain a reuse path: if the worktree path already exists, the checked-out branch matches the expected branch, and any role under `<wt>/.pycastle-session/` is resumable, the create_worktree call is skipped and the existing worktree is reused as-is (preserving in-flight changes and the session dir). Otherwise the unconditional create — which today force-removes any pre-existing path — runs. This is what makes the preservation rule actually durable across orchestrator iterations; without it, the next iteration's `create_worktree` would clobber the preserved state.
- `branch_worktree` (merge-sandbox) gains the same preservation symmetry as `_agent_worktree`: on `__aexit__`, both `teardown_worktree` and `delete_branch` are skipped together when the predicate holds (deleting the branch while preserving the worktree would orphan its HEAD). When the Merger eventually succeeds in a future iteration, role-session-cleanup empties the merger session dir, the predicate flips to false, and the original unconditional teardown + branch delete fire normally.
- Cleanup-on-success is load-bearing for teardown, not just hygiene: after every successful commit (Implementer/Reviewer) or merge application (Merger) the orchestrator runs `shutil.rmtree(role_session_dir, ignore_errors=True)` *inside* the worktree context manager, before `__aexit__`. Without this, the broadened preservation rule would keep the worktree alive forever after success. Partial cleanup failures are logged but never raise; the next `Fresh` launch's wipe-before-Fresh recovers any leftover state.
- Non-typed Resume retry (amended #640): any exception out of `runner.work()` on a Resume run that is not in `{UsageLimitError, AgentTimeoutError, AgentOutputProtocolError, PreflightFailure}` triggers a single in-call retry. The retry calls `runner.work()` once more with the same `session_uuid` and `run_kind=RESUME` — the role session dir is **not** wiped and conversation history is preserved. A second non-typed exception converts to `FailedOutput(failure_class="non_typed_crash")` rather than re-raising; the existing `FailedOutput` plumbing routes it to the Failure-Analysis agent, which files a `bug` + `needs-triage` issue including a static Recovery section instructing the human to wipe `<SESSION_DIR>` manually if they suspect transcript corruption. The orchestrator returns `AbortedAgentFailure` and exits cleanly. Detection remains allowlist-based, not stream-introspection-based. This replaces the original fail-soft fresh fallback; the rationale is recorded in the corresponding "Considered Options" entry above.
- Session-storage gitignoring is repo-local via `<repo>/.git/info/exclude`: the orchestrator startup (alongside `prune_orphan_worktrees`) appends `.pycastle-session/` and `.claude/` to this file if they are not already present. The append is idempotent. Worktrees share the parent `.git`, so the exclusion applies in every worktree without per-worktree files. No tracked file in the consuming project is touched; `pycastle init` is uncoupled from session storage.
- Skip remains the first decision in the dispatch and is unchanged: a `RALPH:` commit on the branch bypasses the agent entirely, regardless of any preserved session. Resume only fires when the agent would otherwise be spawned. The two mechanisms are observable as distinct paths in the orchestrator output.
- The continuation prompt at `defaults/prompts/_resume-prompt.md` is shared across all three roles. After amendment (#640) it is phrased to assume conversation history is present (true on every path the prompt fires on now that Fresh-after-fail-soft is gone): instructs the agent to inspect uncommitted state via `git status`, scan recent messages in the resumed conversation to identify the last completed step, and continue from there without restarting the plan or re-deriving analysis already in the conversation. Per-role variants are deferred until evidence shows a role needs different guidance.
- Smoke tests: with derived UUIDs, the parallel-fixed-UUID isolation property (former smoke test C-iii) is structurally guaranteed and no longer needs an integration test. The `claude --resume` `--print --output-format stream-json` mode test is verified once before implementation; future regressions surface naturally during `pycastle run` via the fail-soft path. The `.claude/` leak (#3833) test is similarly one-shot — the `.git/info/exclude` entry neutralizes the leak regardless of outcome.
- Improve mode (issue #408) builds multi-prompt-within-one-run on top of this primitive without modifying the resume infrastructure. Establishing the primitive here, exercised by the three existing agent roles, gives improve mode a tested foundation rather than introducing the primitive and the improve feature in the same PR.
