# Recently-closed filter on `GithubService.get_open_issues`

The orchestrator's Merge stage closes issues via `PATCH /repos/{repo}/issues/{n}` (returns 200 on success) and immediately returns control; the next iteration's Preflight then queries `GET /repos/{repo}/issues?state=open&labels=ready-for-agent` to seed planning. GitHub's issue *list* endpoint is eventually consistent with respect to single-issue PATCHes — for a brief window after a successful close, the list response can still include the just-closed issue. This caused iteration N+1's Plan agent to re-pick issues #558 and #559 seconds after iteration N had merged-and-closed them, treating them as still-open and unblocked. We absorb the inconsistency inside `GithubService` itself: every successful `close_issue(n)` adds `n` to a private `_recently_closed: set[int]` on the service instance, and `get_open_issues(label)` filters its API response through that set. The set is **self-healing** — on every `get_open_issues` call, before filtering, any number in the set that is *not* in the API response is dropped (proof that GitHub's list view caught up; safe to forget). Callers above the service see no change in shape, no new parameters, no orchestrator-level state.

## Considered Options

- **Status quo (no client-side guard).** Rejected: this is the bug. The race is small but reproducible, has already corrupted one iteration's plan, and is non-deterministic in a way that makes the orchestrator's iteration log lie about what work it picked.
- **Settle delay between Merge and Preflight (e.g. `await asyncio.sleep(3)`).** Rejected: probabilistic — pick too short and the race re-opens under load, pick too long and you waste iteration budget every cycle. No principled value to choose. Also leaks an "API quirk" workaround into orchestrator timing where future maintainers have nothing to grep for.
- **Verify-then-proceed in Merge** (after `close_issue`, poll the canonical single-issue endpoint until `state=closed`). Rejected as the *primary* fix: the canonical endpoint is consistent, so this proves the close took effect — but the *list* endpoint is served from a separate cache and can still lag. Doesn't solve the actual symptom. (Could be added as belt-and-braces, but adds round-trips for no observable benefit given the chosen design.)
- **Re-verify each candidate in Preflight** by hitting the consistent single-issue endpoint for every result of `get_open_issues`. Rejected: adds N extra requests per iteration to defend against an edge case affecting at most one or two issues per iteration, and it scatters the workaround across the iteration layer instead of containing it in the adapter.
- **Track recently-closed numbers on the iteration loop frame**, threading them as an explicit parameter from Merge into the next Preflight. Rejected: the race is an artefact of the GitHub adapter, not of the orchestrator's domain logic. Putting the filter in the loop forces every future caller of `get_open_issues` to remember the workaround independently and leaks "this list endpoint lies sometimes" into iteration code.
- **Track recently-closed numbers on `Deps`** (e.g. `deps.recently_closed`, mutated by Merge, read by Preflight). Rejected for the same reason — `Deps` is a per-iteration dependency container, not a place for adapter quirks; it would also need explicit reset semantics at iteration boundaries that the self-healing approach makes unnecessary.
- **TTL-based eviction from the recently-closed set** (entries expire after N seconds). Rejected: pick too short and the race re-opens, pick too long and a closed-then-reopened issue stays filtered while it's legitimately back in `ready-for-agent`. No principled TTL value. Self-healing uses GitHub's own response as the consistency signal — zero magic numbers.
- **Bounded LRU cap on the recently-closed set.** Rejected: prevents unbounded growth but doesn't fix the "filtering forever after consistency caught up" problem. Self-healing handles both bounding and forgetting in one mechanism.
- **Iteration-scoped reset** (orchestrator calls `github_svc.clear_recently_closed()` between iterations). Rejected: couples the adapter to the iteration loop's lifecycle, defeating the encapsulation the in-adapter design is meant to provide; also wrong on long-running iterations where Merge happens late.
- **Filter every list-style read on `GithubService`** (e.g. also wrap `get_open_issue_numbers`). Rejected for now in favour of filtering only `get_open_issues`: that's the one path the iteration loop uses to drive planning, and self-healing requires observing each call's response — a method that isn't called between Merge and the next Plan can't self-heal. Other list methods exist but have no current race-affected caller. Revisit if a new caller appears.
- **Filter `get_issue` (single-issue endpoint) too.** Rejected: the single-issue endpoint is consistent — it correctly reports a just-closed issue as closed — and callers may legitimately read fields of a just-closed issue (title, body, labels) for diagnostics. Filtering would either return a synthetic stub or hide real data.
- **Add unconditionally to the set** (on call entry, before the PATCH returns). Rejected: if the PATCH fails (network error, 5xx, auth), nothing was actually closed, and adding `n` would silently hide a still-open issue from the next planner pass — exactly the kind of bug the workaround is meant to prevent. Self-healing would eventually drop it (because the failed-to-close issue would still appear in subsequent responses), but the invariant "membership in the set means we believe this is closed" is cleaner to reason about and easier to test.

## Consequences

- `GithubService` gains a private `_recently_closed: set[int]` attribute, initialised empty in `__init__`. Not exposed on the public API.
- `close_issue(number)` adds `number` to `_recently_closed` after the PATCH returns successfully (i.e. `_request` did not raise). On `GithubAPIError` / `GithubNetworkError` / `GithubAuthError`, the set is untouched. Existing success semantics of `close_issue` (which already swallows whatever the underlying PATCH considers success — the method is `None`-returning) are unchanged; the set add happens on the same code path.
- `get_open_issues(label)` is the only read affected. Logic order: (1) call the list endpoint as today; (2) build the set of issue numbers in the response; (3) for each `n` in `_recently_closed`, if `n` not in response numbers, discard `n` from `_recently_closed` (consistency caught up); (4) filter the response by removing any item whose number is still in `_recently_closed`. Order matters — the discard step uses the *raw* response, so a just-closed issue still in the response stays filtered while one that has fallen out gets forgotten.
- Self-healing means the set is naturally bounded: after one or two iterations any closed issue either falls out of the list endpoint and is discarded from the set, or it reappears (was reopened) and is discarded then too. Unbounded growth requires an unbounded stream of closes with no intervening `get_open_issues` calls under any label, which the orchestrator never produces.
- Side effect on the planner agent's "blocker is satisfied because not in open list" reasoning: the filter removes just-closed issues from the open list, so a still-open downstream issue with `Blocked by: <just-closed>` is correctly seen as unblocked on the very next Plan invocation. The planner's existing prompt does not need to change.
- `get_open_issue_numbers`, `get_open_sub_issues`, `get_parent`, `get_labels`, `get_issue`, `get_issue_comments`, `_paginate` — all untouched. The list/canonical asymmetry is documented in the language file; future contributors who add new list-style reads must decide whether their caller cares about the close→list lag.
- The iteration loop, `merge_phase`, `preflight_phase`, `planning_phase`, and `_close_issues_parallel` get no new parameters and no new state. Existing tests that mock `github_svc` continue to work; new tests target `GithubService` directly using a stub `_request` to drive the close → list → list-again sequence.
- Concurrency: `_close_issues_parallel` invokes `close_issue` from many awaitables on a single asyncio loop. `set.add` is GIL-atomic in CPython and the service is single-threaded; no lock is added. If `GithubService` is ever shared across threads, this changes — flagged but not fixed.
- Test coverage required: (a) close → immediate `get_open_issues` returning the just-closed issue → it is filtered out of the result; (b) close → `get_open_issues` returning the just-closed issue → second `get_open_issues` returning the issue absent → the set is now empty (assert via behaviour: a third call where the issue reappears is *not* filtered, simulating reopen); (c) close that raises `GithubAPIError` → `_recently_closed` unchanged → next `get_open_issues` does not filter the issue. Tests live in `tests/services/test_github_service.py` (or wherever `GithubService` is currently tested) and use the existing service-level test pattern, never reaching for a real network. No tests reach into `_recently_closed` directly — invariants are observed through public method behaviour only.
- Tracker for the original incident: iteration 5 of an improve-mode run re-planned issues #558 and #559 after iteration 4 had merged-and-closed them. The closes did happen (`closedAt: 2026-05-08T20:48:37Z`); the next Preflight's list query returned them as still-open. This ADR closes that race.
