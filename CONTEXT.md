# Ubiquitous Language

## Package & Distribution

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **pycastle** | Installable Python package that orchestrates agentic coding work across supported agent services | orchestrator, tool |
| **consuming project** | A project that installs pycastle and customizes it via a local pycastle directory | host project, parent project |
| **pycastle directory** | Fixed `pycastle/` directory inside a consuming project, holding local config, local overrides, and pycastle-managed scaffold such as `config.py.example` and `setup/`; runtime session state does not live here | config dir, override dir |
| **defaults** | Files bundled in the package serving as scaffolding templates and runtime fallbacks | base config, starter files |
| **pycastle init** | Interactive bootstrap command. It refreshes the managed scaffold allowlist, refreshes `config.py.example`, then runs the init wizard for service selection, scope selection, `.env` key merge, and optional GitHub label setup; it does not create local prompt files, an empty prompts directory, or Python cache artifacts | setup, scaffold |
| **pycastle init --refresh** | Non-interactive scaffold sync path. Refreshes the managed scaffold allowlist and `config.py.example` without running the init wizard, writing runtime session state, creating local prompt files, or copying Python cache artifacts | refresh, scaffold sync |
| **init wizard** | Interactive step-by-step flow inside `pycastle init`: service selection → scope selection → `.env` key merge (adds missing keys, prompts per credential relevant to selected service, offers overwrite for existing values) → GitHub label creation. Idempotent: re-running with a different service incrementally updates the setup rather than skipping; `--refresh` bypasses this flow entirely | setup wizard, onboarding |
| **config.py.example** | Always-overwritten template file placed alongside `config.py` showing all behavioral `Config` fields (minus `auto_file_bugs` and `bug_report_repo`), all model names per service (Claude: `haiku`, `sonnet`, `opus`; Codex: `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, `gpt-5.3-codex`, `gpt-5.3-codex-spark`, `gpt-5.2`; bare OpenCode Go IDs), supported effort values per service, live bundled `StageOverride` defaults, and a commented `kimi-k2.6` Planner override as the harder-planning OpenCode alternative. Written to local `pycastle/` always; also to global dir if it already exists; refreshed by both `pycastle init` and `pycastle init --refresh` | example config, config template |
| **managed scaffold allowlist** | Explicit set of copied pycastle-owned files refreshed by `pycastle init` and `pycastle init --refresh`: `.gitignore`, `setup/cron.sh`, `setup/cron-install.sh`, and `setup/cron-uninstall.sh`. `config.py.example` is refresh-owned but rendered separately, not copied from this allowlist. Files outside the allowlist, including `__pycache__/` and `*.pyc`, are never scaffold | scaffold allowlist, copied scaffold |
| **pycastle .gitignore** | Always-overwritten `.gitignore` inside the pycastle directory. It ignores local secrets and generated/runtime files plus pycastle-managed local scaffold (`config.py`, `config.py.example`, and `setup/`) so a consuming project does not see pycastle bootstrap files as project changes; user-specific ignore rules belong in the repo root `.gitignore` or `.git/info/exclude` | scaffold gitignore |
| **setup scaffold** | The allowlisted pycastle-managed helper scripts under `pycastle/setup/` (for example the cron wrappers). Refreshed by both `pycastle init` and `pycastle init --refresh`; files outside the managed scaffold allowlist are never scaffold | setup scripts, cron scaffold |
| **pycastle labels** | CLI subcommand creating or resetting GitHub labels in a target repo using the canonical label set | label setup, label sync |
| **auto-discovery** | Runtime lookup: pycastle directory in CWD → `pycastle home` → defaults; project marker = `pycastle/` exists in CWD | — |
| **pycastle home** | User-scope global directory holding optional `config.py` and `.env`; resolved via `platformdirs` and overridable via `PYCASTLE_HOME`; only `config.py` and `.env` are globalizable (ADR 0003) | global config dir, user config dir |
| **PYCASTLE_HOME** | Env var overriding `platformdirs` default; resolution: explicit `global_dir` arg → `PYCASTLE_HOME` → `platformdirs` | — |
| **layered config merge** | Precedence stack: `defaults module → pycastle home/config.py → pycastle dir/config.py`, applied field-by-field; `.env` extends to `defaults → global .env → local .env → process env`, per-key (ADR 0003) | config layering, config precedence |
| **fixed project-local layout** | Project-local files always live under fixed CWD-relative paths for local config, `.env`, prompt overrides, worktrees, setup scaffold, and optional Dockerfile override; `logs_dir` remains configurable | configurable path fields, custom project dirs |
| **global-forbidden fields** | `Config` fields that may not be set globally (raise `ConfigValidationError` at load): `docker_image_name` | global-forbidden path fields, path field guard |
| **scope flag** | `--global` / `--local` mutually-exclusive flag pair on `pycastle init`; selects where `config.py` and `.env` live; absent flag triggers interactive prompt | global/local flag |
| **layer summary** | One-line startup message listing which config layers were found, routed via `StatusDisplay.print("", ...)` | config layer line |

## Configuration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **config.py** | User-owned Python file defining behavioral configuration; overrides defaults module field by field. Local `pycastle/config.py` may be created by init and ignored by `pycastle/.gitignore`, but refresh preserves it because ignored does not mean pycastle-managed | settings.py, settings |
| **defaults module** | `src/pycastle/defaults/config.py` — pure default values, no logic | defaults config, fallback config |
| **config loader** | `loader.py` inside `config/`; reads defaults, executes global then local `config.py` via importlib, applies behavioral overrides, ignores removed project-local path keys, rejects forbidden fields, validates efforts, returns immutable `Config`; pure (no subprocess, no service deps) | — |
| **removed project-local path keys** | Stale `config.py` keys for locations now owned by the fixed project-local layout: `pycastle_dir`, `prompts_dir`, `worktrees_dir`, `env_file`, and legacy `dockerfile`; ignored when present | path config, configurable directories |
| **.env** | File holding secrets/credentials only; never committed; lives in local pycastle dir and/or `pycastle home`; per-key layered for host-side credential resolution. Being present in resolved host env does not automatically make a key part of the container environment | environment file, config |
| **GH_TOKEN** | Sole GitHub credential: PAT in `.env`, used by host code and propagated into the base container environment for agent-side `gh` usage; validated at orchestrator startup via `check_auth()` (ADR 0004) | github token, gh pat |
| **CLAUDE_CODE_OAUTH_TOKEN** | Long-lived OAuth token from `claude setup-token`; primary-account token in dual-account model (ADR 0005). Required at startup only when `"claude" in referenced_services(cfg)` — codex-only runs do not need it | claude token, oauth token, primary token |
| **CLAUDE_CODE_OAUTH_TOKEN_SECONDARY** | Optional secondary-account OAuth token; preferred over primary when present and not exhausted (ADR 0005) | secondary token |
| **subscription-access denial** | Claude Code `403` denial saying the organization has disabled Claude subscription access for Claude Code; treated as permanent exhaustion of the active Claude account for the current process, not as a pycastle defect | Claude 403 denial, subscription disabled |
| **primary account** | Claude account whose token is `CLAUDE_CODE_OAUTH_TOKEN`; always present, fallback when secondary exhausted | fallback account |
| **secondary account** | Optional Claude account whose token is `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY`; preferred when available | preferred account |
| **PREFLIGHT_CHECKS** | Config `list[tuple[str, str]]` of `(name, command)` pairs run inside `preflight-sandbox` by `ContainerRunner.preflight` on unseen HEAD; detection-only | preflight commands, check list |
| **HOST_CHECKS** | Config `list[tuple[str, str]]` of `(name, command)` pairs run by `pycastle check` in a temporary local worktree on the current OS; default is the pytest host suite only because other default checks give the same signal on Windows and Linux | Windows checks, local checks, host preflight checks |
| **IMPLEMENT_CHECKS** | Config `list[str]` rendered into FEEDBACK LOOPS of the implement prompt; may differ from PREFLIGHT_CHECKS | feedback commands, implement commands |
| **logs_dir** | Configured log location. In local config it is the exact log directory; in global config it is a parent directory and the effective log directory is `<logs_dir>/<sanitised project name>/`, using the same sanitisation as `docker_image_name` | log path, log folder |
| **field-by-field override** | Loader strategy: for each non-underscore name in consuming `config.py`, `setattr` replaces corresponding name in loader module | full replacement, merge override |
| **STAGE_OVERRIDES** | Config dict with one entry per agent role (`plan`, `implement`, `review`, `merge`, `preflight-issue`, `improve`), each holding model shorthand and effort | stage config, model config |
| **diagnose_on_failure** | Boolean (default `True`) controlling: (1) whether improve phase 1's `NO-CANDIDATE` triggers the no-candidate report; (2) whether `FailedOutput` spawns the Failure-Report agent before returning `AbortedAgentFailure` | failure diagnosis flag, no-candidate toggle |
| **stage override** | Per-agent-role `(service, model, effort)` triple inside STAGE_OVERRIDES; all three fields mandatory on every node (primary and fallback). `service` and `effort` must be non-empty; `model` may be empty (CLI default) or a known shorthand for the named service. `service` determines the `AgentService` command/env/parser used inside the universal agent image. Optional `fallback: StageOverride \| None` for cross-service failover | phase config, agent config |
| **default stage priority chains** | Bundled stage fallback chains used by both runtime defaults and scaffolded config examples: plan `opencode/deepseek-v4-flash/medium -> codex/gpt-5.4-mini/low -> claude/haiku/low`; implement `codex/gpt-5.4/medium -> claude/sonnet/medium`; review `claude/sonnet/medium -> codex/gpt-5.4/medium`; merge `codex/gpt-5.5/medium -> claude/opus/high`; preflight-issue `codex/gpt-5.5/medium -> claude/opus/high`; improve `codex/gpt-5.5/high -> claude/opus/high` | default stage overrides, default agent matrix |
| **OpenCode plan default** | Product policy that `OpenCodeService` participates in bundled defaults only as the first Planner candidate, using `deepseek-v4-flash` at `medium` effort before the existing Codex and Claude plan fallbacks; `kimi-k2.6` is the documented higher-judgment alternative for harder planning rather than the bundled default | OpenCode opt-in policy, bundled OpenCode chain |
| **model shorthand** | Service-specific model identifiers validated at config load against a per-service allowlist via `AgentService.valid_models()`. Claude: `haiku`, `sonnet`, `opus` (ADR 0002). Codex: `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, `gpt-5.3-codex`, `gpt-5.3-codex-spark`, `gpt-5.2`. OpenCode v1 config accepts known bare OpenCode Go model ids only, not arbitrary OpenCode provider strings; `OpenCodeService` translates them to `opencode-go/<id>` at the CLI boundary. Empty string is valid (CLI picks its own default). Unknown shorthands rejected at startup | model alias, model name |
| **effort level** | Service-specific set validated at config load via `AgentService.valid_efforts()`. Claude: `low`, `medium`, `high`, `xhigh`, `max`; Codex: `low`, `medium`, `high`, `xhigh`; OpenCode: `medium`. Codex `none` and `minimal` are intentionally unsupported because low-effort Codex modes suppress search/tool access needed by pycastle agents. Overlap between Claude and Codex: `{low, medium, high, xhigh}` | effort, effort flag |
| **CLI default** | Behavior when `model=""` — service CLI picks its own default model | default model, unset |
| **ConfigValidationError** | Raised on invalid config values: empty `service` or `effort`, unknown service name, model shorthand not in service's allowlist, unknown effort for service | validation error, config error |
| **auto_push** | Boolean (default `True`) controlling whether `merge_phase` pushes to remote after merges | push_after_merge, AUTO_PUSH |
| **docker_image_name** | Docker image tag for the universal agent image; global-forbidden; when unset, derived from sanitised `Path.cwd().name` (`re.sub(r"[^a-z0-9]+", "-", name.lower()).strip("-")`). `pycastle build` builds one image containing all supported service CLIs. Runtime service selection is independent of the resolved `StageOverride.service` | image name, container image |
| **auto_file_bugs** | Boolean (default `False`) gating *all* upstream auto-issue filing into `bug_report_repo`; when `False`, each site falls back to non-filing behaviour; flipped on only by pycastle developers via global config (ADR 0022) | bug toggle, auto report flag |
| **bug_report_repo** | Config (default `"Johannes-Kutsch/pycastle"`) — `owner/repo` target for auto-filed bug reports; validated against `^[^/]+/[^/]+$`; not global-forbidden | upstream repo, bug repo |

## GitHub Integration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **label** | GitHub issue classification marker with name, description, hex color | tag, category |
| **canonical label set** | Ten labels hardcoded in the package: `bug`, `needs-info`, `needs-triage`, `ready-for-agent`, `ready-for-human`, `wontfix`, three slice-mode labels (`refactor-slice`, `behavior-slice`, `docs-slice`), and `needs-slice-type` | default labels, label config |
| **label reset** | Option to delete all existing labels before creating canonical set | label wipe, clean labels |
| **issue label** | Label value (default `ready-for-agent`) marking issue as eligible for agent processing | trigger label, agent label |
| **issue readiness** | In-process eligibility decision for an AFK issue labeled with the configured issue label. The readiness module combines slice-mode classification, the stripped body-length floor, and configured marker labels into a structured ready or blocked outcome. Ready outcomes carry the selected slice mode and implement prompt template; blocked outcomes carry malformed slice-mode and short-body facts so planning can sync `needs-slice-type` and `needs-info` independently. The module performs no GitHub I/O. | AFK readiness, eligibility |
| **slice-mode label** | One of `refactor-slice`, `behavior-slice`, `docs-slice`; classified as part of **issue readiness**. Exactly one configured slice-mode label is required for AFK code-or-docs work, and missing or multiple configured slice-mode labels block the issue from the current implement pass. | mode label, slice label |
| **planning readiness** | In-process decision module that evaluates hydrated ready-for-agent issues before Planner dispatch. Given the issues and `Config`, it returns implementable candidates, malformed body issues, malformed slice-mode issues, label-sync actions, and planning blocker summary inputs. It performs no I/O; `planning_phase` owns the preflight gate, Planner dispatch, status rows, and applying returned label actions through `GithubService` | issue readiness, readiness module |
| **needs-slice-type** | Marker label for malformed slice-mode labels on a `ready-for-agent` issue. `planning_phase` syncs this marker from an **issue readiness** blocked outcome for malformed slice-mode labels. Coexists with `ready-for-agent` (issue not demoted); marks issue ineligible for this iteration's implement phase. Malformed issue stays in `all_open_issues` so dependents see it as a blocker | slice-needed label |
| **needs-info** | Marker label for AFK issues whose stripped body length is below the hardcoded floor (100 chars). The stripped body-length floor is owned by the **issue-readiness module**, and `planning_phase` syncs this marker from an **issue readiness** short-body blocked outcome. Independent of `needs-slice-type` — both may coexist. Coexists with `ready-for-agent`; marks issue ineligible for this iteration's implement phase. Malformed issue stays in `all_open_issues` so dependents see it as a blocker. Same floor is enforced defensively when preflight and host-check diagnostic agents file AFK issues (raises) | body-too-short label |
| **refactor-slice** | Slice mode for issues whose steps cannot be verified by a new test of observable behavior (symbol moves, renames, protocol introductions, import rewires, dead-code removal). Multiple steps may bundle. Lands first; dependents list it in `Blocked by`. Dispatches to `work/refactor.md` (ADR 0016, path updated by ADR 0040) | refactor prep, refactor issue |
| **behavior-slice** | Slice mode introducing/changing observable behavior verifiable by a new test. Acceptance criteria describe behavior + observable surface, never test-code structure. Dispatches to `work/behavior.md` (ADR 0016, path updated by ADR 0040) | feature slice, behavior issue |
| **docs-slice** | Slice mode whose only artifacts are markdown edits (CONTEXT.md, ADRs, README). No code, no tests. Dispatches to `work/docs.md` (ADR 0016, path updated by ADR 0040) | docs issue, documentation slice |
| **issue-readiness module** | In-process module at `src/pycastle/agents/classifier.py` that owns slice-mode classification, stripped-body floor validity, marker-label decisions, and structured ready or blocked results for AFK issues without performing GitHub I/O. Planning, diagnostic issue validation, and implement dispatch consume this module instead of recomposing those invariants locally. | readiness helper, readiness classifier |
| **planning blocker summary** | One-line terminal summary emitted by `planning_phase` from **planning readiness** data when ready-for-agent issues are blocked. Reports blocker counts by category, specifically missing slice-mode labels and bodies below the minimum length floor. Diagnostic only; label sync remains the source of actionable state | planning output, blocker summary |

## Agents

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **agent** | Claude Code instance running inside an isolated Docker container | bot, worker |
| **ripgrep (`rg`)** | Fast code-search utility available in every default agent image for workspace navigation; part of baseline agent infrastructure | prompt search, fallback search |
| **Planner** | Plan-phase agent; runs in a clean checkout of the safe SHA; outputs a plan | planning agent |
| **Implementer** | Implement-phase agent; one per issue | coding agent, implementation agent |
| **Reviewer** | Post-Implementer agent validating changes before merge. Renders `PER_ISSUE` scope (`PromptTemplate.REVIEW`); does **not** receive a precomputed diff — runs `git diff main... --stat` inside the container (step 3). Returns `CommitMessageOutput` (same as Implementer) | review agent |
| **Merger** | Merge-phase agent spawned for one active conflicting branch to resolve files inside fresh host-owned merge state based on the current safe SHA; pycastle owns commit creation, branch ancestry, and issue closing | merge agent, integration agent |
| **preflight-issue agent** | Agent spawned by `PreflightCache.get_safe_sha()` on `PREFLIGHT_CHECKS` failure; explores codebase, evaluates HITL, files one structured GitHub issue, outputs `<issue>NUMBER</issue>` | Pre-Flight Reporter, bug-report agent, error reporter |
| **host check** | Operator-visible quality check loop surfaced as a `Host Check` phase row in terminal status covering branch refresh, clean-tree validation, transient worktree setup, and configured check execution; validates committed `HEAD` in a clean temporary local worktree on the current OS, using the host toolchain rather than the agent container and stopping after failure diagnosis/issue filing; status is one aggregate host-check row that names the current command and explicitly reports failed checks before any issue agent starts; command stdout/stderr is diagnostic payload captured for failures, not live terminal output; the final pass/failure summary remains separate from the status row | Windows check, local preflight, host preflight |
| **host check run module** | Internal module owning the ADR-0036 host check sequence for `pycastle check`: branch refresh, clean-tree validation, transient local worktree lifetime, configured command execution, failure capture, Host Check status ordering, host-check issue agent dispatch, issue validation, and explicit pass/failure outcome creation. The command entrypoint is a thin adapter that loads config, supplies concrete adapters, invokes this module, and prints the final summary. | check command internals, host check runner |
| **host-check issue agent** | Agent spawned in the normal agent environment once per failed host check command by `pycastle check`; receives host OS/platform, checked SHA, failed check name/command/output, deduplicates against open host-check issues, and files or updates either an AFK-fixable issue or a HITL issue. If this agent cannot start because its own Setup phase fails, the setup failure is terminal and preserves the failed host-check context for operator/upstream diagnosis | preflight-checks-failed agent, check triage agent |
| **check-failure issue filing** | Shared issue-filing policy used by both preflight-issue and host-check issue agents: diagnose a failed configured check, apply the same AFK/HITL decision rules, and file one structured GitHub issue; does not require the prompt files to share wording | check triage, failure filing |
| **divergence-resolver** | Agent role (`AgentRole.DIVERGENCE_RESOLVER`) spawned by `get_safe_sha()` as the third escalation rung when `pull_with_merge_fallback` raises on textual conflicts; runs inside `diverge-sandbox` (built from local main's tip), merges `origin/<branch>` and resolves all conflicts. Textual conflict resolution only — explicitly forbidden from running preflight checks; post-merge breakage is caught by the same `get_safe_sha()` call's subsequent preflight pass on the new HEAD and routed through the preflight-fix path. Success → `<promise>COMPLETE</promise>` and fast-forward; failure → `<promise>FAILED</promise>` and original `GitCommandError` propagates. Tool surface restricted to `Read,Edit,Bash` (ADR 0024) | conflict resolver, merge conflict agent |
| **HITL verdict** | Routing decision in the preflight issue's label: `ready-for-agent` → spawn one Implementer; `ready-for-human` → orchestrator aborts | HITL decision, routing verdict |
| **agent flag profile** | Per-`AgentRole` set of CLI flags selected by the service adapter; hardcoded (not user-tunable via `STAGE_OVERRIDES`) because the flags encode each role's tool/permission contract. Claude profiles are enforced through `claude_service.build_command`: Universal `--disable-slash-commands`, `--exclude-dynamic-system-prompt-sections`, `--strict-mcp-config --mcp-config '{"mcpServers":{}}'`; **tool-restricted** Planner (`--tools "Read,Glob"`) and divergence-resolver (`--tools "Read,Edit,Bash"`); **investigator restriction** (`--disallowedTools "Edit Write NotebookEdit"`) for preflight-issue and improve scan; **full-tool** Implementer, Reviewer, Merger. Codex profiles are coarser: partial roles bypass Codex's inner command sandbox to allow targeted shell/file operations inside pycastle's container boundary; Codex v1 preserves partial-role limits primarily through prompt contract and worktree lifecycle because the CLI does not expose Claude-equivalent per-tool denial (ADR 0024, ADR 0031) | flag matrix, per-role flags |
| **agent tool policy group** | Cross-service categorisation for an agent role's tool/permission contract. **Restricted** roles get the smallest read-oriented tool set needed for planning. **Partial** roles may inspect the worktree and run targeted shell/file operations for issue filing, diagnostics, or textual conflict resolution, but must not do broad implementation work. **Full** roles may freely mutate the worktree and run shell commands as part of implementing, reviewing, or merging code. Service adapters map these policy groups onto their own CLI flags; exact enforcement strength may differ by provider, so partial is a normative role contract even where a provider can only enforce it weakly. Membership: restricted = Planner; partial = divergence-resolver, preflight-issue, failure-report, improve; full = Implementer, Reviewer, Merger | tool tier, permission tier |
| **session-start tool policy** | Rule for any resumable agent session: choose the least restrictive tool/permission profile needed by any phase that may run in that session at Fresh start, then keep that profile stable for every Resume. Later prompts may narrow behaviour by instruction, but service CLI flags must not tighten or loosen mid-session. This is load-bearing for improve's `main` namespace because phase 1 scan and phase 2 PRD share one service transcript, and phase 2 requires shell/`gh` even if phase 1 mostly reads | resume tool policy, session permissions |
| **commit body composition** | Host-side commit body produced by `run_issue()` on clean exit: `f"Implement #{n} - {message_or_title}"` (Implementer) / `f"Review #{n} - {message_or_title}"` (Reviewer); `message` = `<commit_message>` if present, else issue title. No orchestrator-readable stage marker in body — stage-done is filesystem-based (ADR 0007) | commit message format |
| **`<commit_message>` tag** | Optional XML tag from Implementers/Reviewers carrying plain description of changes; orchestrator uses content as commit body when present, synthetic fallback when absent. Absence is **not** a failed run (ADR 0007) | — |

## Iteration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **orchestrator** | Main loop coordinating agent phases across GitHub issues | runner, coordinator |
| **iteration** | One complete plan-bounded loop ending after all planned issues attempted; up to MAX_ITERATIONS per `pycastle run`; exactly one merge phase closes the iteration regardless of `max_parallel` | cycle, round, pipeline |
| **3-phase loop** | One iteration's structure: plan, implement+review, merge | pipeline, workflow |
| **`max_parallel`** | Config integer; **concurrency cap only** — the maximum number of agents running at once. Implemented as two coordinating semaphores in `implement_phase`: agent semaphore (`max_parallel` slots) gates `_bounded_run_agent`, worktree semaphore (`max_parallel + 1` slots) brackets every `managed_worktree` call in `run_issue` (both Implementer and Reviewer opens). Does **not** cap the per-iteration batch — every issue the planner returns runs in the iteration it was planned in, queued through the semaphores; one merge phase closes the iteration. `max_parallel = 1` serialises agents but does not give each issue its own merge phase (ADR 0027) | parallelism, concurrency limit |
| **plan phase** | Phase where Planner analyzes open issues and produces a plan | planning step |
| **implement phase** | Phase where Implementers fix issues in isolated worktrees | coding step |
| **review phase** | Phase where Reviewer checks an Implementer's changes before merge | code review step |
| **merge phase** | Phase where completed branches are integrated and issues close only after their branch is a verified merged branch; conflicting branches are recovered one branch at a time | integration step |
| **safe SHA** | SHA most recently observed at HEAD by `PreflightCache.get_safe_sha()` and confirmed by `PREFLIGHT_CHECKS` in `preflight-sandbox`; single-slot verdict surviving across iterations until HEAD advances. Fetched on demand by `improve_phase`, `planning_phase` (only on the fresh-plan path), `merge_phase`. `implement_phase` consumes the SHA from `PlanReady.sha`; on the in-flight planning path `PlanReady.sha` is `None` because resumed branches already exist locally and `managed_worktree`'s existing-branch path ignores the sha argument — merge-phase preflight remains the safety gate (ADR 0014) | verified SHA, clean SHA |
| **preflight pull** | `git pull --ff-only` (with merge fallback) inside every `get_safe_sha()` call; waits for clean working tree; on textual conflict spawns divergence-resolver in `diverge-sandbox`; on transient remote-reach failures retries per the operator-actionable git retry profile, then escalates as `OperatorActionableGitError` (ADR 0026); on resolver failure raises | — |
| **`PreflightCache`** | Process-scoped object owning the safe-SHA verdict. Constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Single private slot `_verdict: PreflightReady | PreflightAFK | PreflightHITL | None` plus `asyncio.Lock` serialising concurrent callers. `async def get_safe_sha(deps) -> PreflightResult`: under lock, waits for clean tree, runs `pull_with_merge_fallback` (ff-only → clean merge → divergence-resolver), reads HEAD; if slot SHA matches HEAD returns cached verdict; else opens `transient_worktree("preflight-sandbox", sha=HEAD)`, runs `PREFLIGHT_CHECKS`, stores `PreflightReady(sha=HEAD)` on success or runs preflight-issue agent and stores `PreflightAFK | PreflightHITL` on failure. Single-slot suffices because the escalation ladder only moves HEAD forward (ADR 0013) | preflight cache, safe-sha cache |
| **preflight-fix path** | Orchestrator routing when `get_safe_sha()` returns `PreflightAFK`: freshly-filed preflight issue is handed straight to `implement_phase` in the same iteration; `planning_phase` is bypassed (planning skip); normal Reviewer → merge pipeline follows | preflight fast path |
| **programmatic merge path** | Fast-path merge logic running `git merge --no-edit` directly via subprocess without the Merger; used when all branches merge cleanly | fast path, direct merge |
| **clean merge** | `git merge --no-edit` that exits zero with no conflict resolution | conflict-free merge, successful merge |
| **conflicting branch** | Branch whose `git merge --no-edit` exits non-zero; `git merge --abort` runs immediately, branch collected for Merger | failed merge branch |
| **merged branch** | Branch whose tip is a Git ancestor of the target branch's HEAD; content-equivalent commits without ancestry do not count as merged | copied branch, integrated content |
| **in-flight issue** | Open issue carrying preserved work from a previous iteration. Detection requires evidence of real work, not just a leftover ref: in-flight ⇔ (worktree dir exists AND `any_role_dir_present(wt_path)` — i.e., some role session was started) OR (`merge-base(main, branch)..branch` is non-empty — i.e., the branch has commits ahead of where it forked from main). Empty branch refs (Implementer created branch but made zero commits) do **not** count, and are cleaned up by the broadened preservation rule's teardown path | mid-flight issue, resumed issue |
| **in-flight issue selector** | Internal module under `pycastle.iteration` owning selection of ready-for-agent issues whose preserved work should resume. Given the ready-for-agent issues plus local repo state, it returns the exact issue dicts in input order. | **interrupted-work prompt clause** | Conditional role-prompt addition fired when `run_kind == FRESH` and the worktree has uncommitted changes (dirty working tree). Tells the agent to run `git diff` and `git status` to understand current state and continue from where the previous agent left off. Does not inline the diff; the agent inspects via tool calls to keep prompt token cost bounded. Replaces the rejected WIP-commit approach (ADR 0015) | cross-service handoff clause, WIP-aware clause |
| **service registry** | In-process module `ServiceRegistry` under `src/pycastle/services/` wrapping `services: dict[str, AgentService]` with four methods: `resolve(override, now)` (best available service implementing stage fallback chain), `has_available(now)`, `next_wake_time(now)`, `summary_lines()`, plus lookup-by-name. Per-stage `StageOverride.service: str` references entries by name; no `default_service` — every override names its service explicitly | service map, service config |
| **locally configured service** | Service that this machine is actually prepared to run. Presence is credential-backed, not Dockerfile-backed: Claude is locally configured when the required Claude credential is present; Codex is locally configured when host Codex auth exists and the service can be instantiated. A locally configured service may still be temporarily unavailable due to usage limits. Referenced-but-unconfigured services in a stage fallback chain are unresolved entries and are skipped during resolution | configured service, installed service |
| **lazy service registry** | Construction strategy in `main.py`: walk every stage override, collect referenced service names, instantiate exactly those `AgentService` implementations, wrap in `ServiceRegistry`. Claude-only configs never construct `CodexService`; unknown service names caught by fail-fast validation before instantiation | on-demand registry, demand-driven registry |
| **fail-fast service+model+effort validation** | Validation pass at top of `pycastle run` (after config load, before container start): walks every `StageOverride` node (primary and fallback), asserts `service` non-empty and known, and validates `effort` plus non-empty `model` against that service's allowlists even if the service is not locally configured. Local absence is not itself a validation error; only an empty/unknown service name or invalid model/effort is. Collects every violation, reports in single error message, exits non-zero | startup validation, run-time config check |
| **`valid_efforts()`** | Method on `AgentService` returning `frozenset[str]` of effort strings pycastle supports for each service. `ClaudeService` returns `{low, medium, high, xhigh, max}`; `CodexService` returns `{low, medium, high, xhigh}`. Consumed by fail-fast validation; per-service because sets genuinely differ | effort allowlist, service efforts |
| **`valid_models()`** | Method on `AgentService` returning `frozenset[str]` of model shorthands the service accepts. `ClaudeService` returns `{haiku, sonnet, opus}`; `CodexService` returns `{gpt-5.5, gpt-5.4, gpt-5.4-mini, gpt-5.3-codex, gpt-5.3-codex-spark, gpt-5.2}`. Empty string (CLI default) is always valid. Consumed by fail-fast validation | model allowlist, service models |
| **per-stage service dispatch** | At agent invocation time, the resolved `StageOverride.service` selects the `AgentService` instance used for command building, env injection, and output parsing. Each stage may use a different service within the same iteration. Replaces the pre-#923 bug where a single runner-wide service was selected from `default_service` | stage-level service routing |

## Test Anti-Patterns (Red Flags)

| Term | Definition | Why it's a problem |
| --- | --- | --- |
| **parent-env-dependent test** | Test whose expected result depends on terminal or parent-process environment inherited from the runner, such as `TERM`, `NO_COLOR`, `FORCE_COLOR`, `COLORTERM`, or `CI` | Preflight and agent-invoked checks may legitimately have different parents; assertions then report runner shape rather than product behavior |
| **platform-dependent test** | Test whose expected result changes because of host OS shape, such as path separators, shell semantics, line endings, filesystem locking, executable names, or platform-specific defaults | Windows host tests and Linux container checks cover different product boundaries; failures then report OS assumptions rather than behavior |
| **Verifying through external means** | Tests calling subprocess, querying DBs, or checking external state directly instead of testing through the service interface | Requires external tools in test env; failures reflect environment, not code bugs |
| **Mocking internal collaborators** | Tests mocking classes/functions within the same codebase rather than using DI | Brittle tests coupled to implementation; breaks on refactoring |
| **Testing private methods** | Tests calling `_`-prefixed functions or accessing private attributes | Private methods are implementation details; test public behavior |
| **Asserting on call counts/order** | Tests verifying internal function calls happened N times or in specific sequence | Breaks on refactoring; couples to implementation rather than behavior |
| **Test name describes HOW not WHAT** | Test names like `test_calls_git_config` instead of `test_setup_configures_git_identity` | Developers cannot understand intent from the name alone |

## Relationships

- **STAGE_OVERRIDES** has six entries, one per agent role; each must specify `service` (non-empty) and `effort` (non-empty); `model` empty means CLI default. `preflight-issue` and `improve` entries are not tied to a phase in the 3-phase loop.
- **`load_config()`** is pure (no subprocess); `docker_image_name` is derived from sanitised `Path.cwd().name` if unset post-merge; model strings stored as-is (ADR 0002).
- **Layered config merge** applies at every CLI subcommand entry; every subcommand prints the resolved layer summary at startup (ADR 0003).
- **`PreflightCache`** is constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Its `asyncio.Lock` serialises concurrent callers; parallel `run_issue` queue, second-and-later see a populated slot. Iterations with no save-SHA consumer (idle without `--improve`, in-flight-only) never invoke `get_safe_sha()`. Single-slot suffices because HEAD only moves forward.
- **HITL verdict** is read by `get_safe_sha()` from the GitHub issue label after the preflight-issue agent completes; `ready-for-agent` → `PreflightAFK` (triggers preflight-fix path); `ready-for-human` → `PreflightHITL` (aborts iteration with `AbortedHITL`).
- **HOST_CHECKS** and **PREFLIGHT_CHECKS** may diverge because host checks validate the current OS/toolchain while preflight checks validate the agent/container boundary; the default **HOST_CHECKS** run only the pytest host suite.
- `pycastle check` invokes one **host-check issue agent** per failed **HOST_CHECKS** command.
- The **host check run module** is the locality for ADR-0036 behaviour; command code should not duplicate the host check sequence or expose private helper call order as a test surface.
- The **host-check issue agent** owns prompt-level deduplication: repeated runs with the same open host-check failure update/comment on the existing issue rather than filing duplicates.
- `pycastle check` requires a clean working tree, refreshes the local branch using the same pull/update recovery behaviour as the normal run loop, and validates the resulting committed `HEAD`, preventing local uncommitted edits from becoming unreproducible filed issues.
- Host check commands execute on the operator's current OS; the **host-check issue agent** runs in the normal agent environment using captured failure output and minimal host context for diagnosis. The host-check prompt states that the failure was observed outside the agent container; it does not use a separate check-context metadata field.
- Passing **HOST_CHECKS** are report-only; `pycastle check` does not auto-close or relabel issues previously filed by the **host-check issue agent**.
- **Preflight-issue agent** and **host-check issue agent** share the same **check-failure issue filing** policy; the existing preflight prompt remains independent, while the host-check prompt adds only enough context to make the host OS boundary clear.
- A **host check** is separate from the normal run loop: it does not implement or merge filed issues; it stops after diagnosing failures and filing **AFK issue** or **HITL issue** records.
- **Planning skip** has two triggers, both routing to `implement_phase`: in-flight present, or preflight-fix AFK. **Implement skip** and **review skip** are checked inside `run_issue` before any worktree is created; both triggered by stage-done signal (ADR 0007).
- **Merge-time preflight skip** leaves conflict issues open and routes the preflight issue first; conflict issues resume only after the preflight-fix path repairs the broken post-clean-merge baseline.
- **Sequential mode**: after each merge, safe SHA re-pinned to new HEAD; failed issue stays `ready-for-agent` and queue continues; Merger remains a fallback.
- **Collision detection** uses a per-branch `asyncio.Lock` held in `implement_phase` for the full duration of each `run_issue` call.
- **`transient_worktree`** is used by `planning_phase` (plan-sandbox); preserves worktree on `AgentFailedError` so the Failure-Report agent can mount it (ADR 0030). **`managed_worktree`** is used by `merge_phase`, `run_issue` (delete=False), and improve mode (delete=True). Gitdir overlay ownership stays inside `build_volume_spec`; phase code passes only a mount path to `AgentRunner`, and `DockerSession` receives any `auto_overlay` handle for cleanup.
- Host mounts per worktree container: worktree → RW at `/home/agent/workspace`; `<host-repo>/.git` → RW at `/.pycastle-parent-git` when the mount path is a git worktree; gitdir overlay → RO over `/home/agent/workspace/.git` when the worktree `.git` file needs a container-internal parent-git path.
- **Recently-closed filter** lives entirely inside `GithubService`; scoped to `get_open_issues` only. Other list-style reads (`get_open_issue_numbers`, `get_open_sub_issues`) are not filtered (ADR 0011).
- **GitHub authentication** is single-source: `GH_TOKEN` only (ADR 0004). Host code uses `urllib`; agent containers use `gh` (installed in Dockerfile per ADR 0021). `(owner, repo)` derived from `git remote get-url origin` — no `gh repo view`.
- **Repo-scoped GitHub API failures** during `pycastle run` startup or planning are terminal: the orchestrator prints a clear access message and exits non-zero instead of crashing into the pycastle bug reporter.
- **`Deps`** does not carry `env`; credentials are extracted in `main.py` and passed directly to `AgentRunner` at construction.
- Each phase module declares its own per-phase dependency protocol listing only its actual field accesses; `Deps` satisfies all structurally.
- Rich markup must never be embedded in a `StatusDisplay.print` message; colouring is via the `style` parameter only.
- All orchestrator-level terminal output is routed through `StatusDisplay.print()`; bare `print()` is not used while a `StatusDisplay` is active.
- Test setup normalises terminal/colour environment at the Python test boundary for the whole suite, so **parent-env-dependent tests** are treated as invalid regardless of whether pytest is launched by trusted preflight, an agent CLI, CI, cron, or a developer shell.
- **Parent-env-dependent test** prevention is a test-suite invariant, not pycastle's user-facing colour policy; support for honouring `NO_COLOR` in real CLI output is separate product behaviour.
- Windows host tests are first-class for host-side pycastle behaviour; Linux container checks remain canonical for agent/container behaviour. **Platform-dependent tests** need explicit boundary marking rather than leaving either side red.
- During Work, the container runner injects the pre-rendered prompt (built upstream by `AgentRunner._build_prompt`, ADR 0012); passes decoded NDJSON + `on_turn` to `process_stream`, which raises `UsageLimitError(reset_time)` on 429. Setup and Pre-flight produce no console output — activity reflected only in body column.
- A **Setup phase** failure is any failure before role work or preflight checks begin that prevents establishing the container baseline: container start, git identity propagation, gitdir/volume wiring, or dependency installation. Once an agent CLI starts, failures follow the role's agent-failure path; once a configured check command starts, ordinary source/project-level failures follow the preflight check failure path. Exception: a preflight command that fails because a Python-declared check tool is unavailable is still a Setup/toolchain contract failure, even though the command was attempted.
- **Auto bug reporter** is *outside* the iteration layer and does not depend on `Deps`, `Logger`, or `StatusDisplay` — must function during config-load crashes. Cross-repo POST goes through `GithubService.create_issue_in(owner_repo, ...)`. The reporter is the *only* feature that talks to a GitHub repo other than the consumer's `origin`.
- **`AgentRunner`** constructs a `DockerSession` (via `build_volume_spec`) and a `ContainerRunner`, then orchestrates the three lifecycle phases. At session-build time it asks `AccountPool` for the active OAuth token and injects it as `CLAUDE_CODE_OAUTH_TOKEN`. On `UsageLimitError` it calls `pool.mark_exhausted(token, reset_time)` before re-raising.
- **Agent run kind** dispatch is reached only when the agent would otherwise spawn: stage-done signal check in `run_issue` decides Skip first; only not-skipped roles invoke `decide_agent_run_kind`. Skip and Resume share the same role-dir-driven mechanism (ADR 0007).
- A `Fresh` run begins with wipe-before-Fresh then launches `claude --session-id` with the role's normal prompt. `Resume` launches `claude --resume` with continuation prompt alone. Non-typed Resume crashes use the non-typed Resume retry, not relaunch-as-Fresh. Container env carries `CLAUDE_CONFIG_DIR=/home/agent/workspace/.pycastle-session/<role>/` in every case.
- Each `(role, worktree_path)` tuple gets a structurally unique derived session UUID, so parallel agents on different worktrees cannot collide on any Claude-internal session-id-keyed cache.
- **Broadened preservation rule** enforced in `managed_worktree.__aexit__` for every branched call site. When `delete_branch_on_teardown=True`, `teardown_worktree` and `delete_branch` skipped together when predicate holds. **Worktree reuse on enter** in `__aenter__` skips `_create_worktree` when path exists, branch matches, predicate holds. `transient_worktree` preserves on `AgentFailedError` only (ADR 0030); all other exceptions tear down unconditionally.
- On agent success, orchestrator runs **role session cleanup** *inside* the worktree context manager, after commit (Implementer/Reviewer) or merge (Merger) and before `__aexit__`. Issue worktree: cleanup clears session JSONL but leaves role dir as stage-done sentinel — predicate preserves the worktree. Improve-sandbox and merge-sandbox: cleanup calls `RoleSession(...).discard()` so dir is removed entirely, predicate goes False, sandbox torn down. Cleanup failure logged, never raised.
- `AgentTimeoutError` is preservation-worthy: single-agent phases run `timeout_retries` (default 1); on exhaustion, top-level `try/except AgentTimeoutError` (mirror of centralized usage-limit catch) converts to `AbortedTimeout(failed_role, worktree_path)`. Orchestrator continues to next iteration without sleep. Parallel `implement_phase` aggregates timeouts as `ImplementResult.errors`.
- Session resume and skip are two role-dir-driven mechanisms sharing the directory and predicate, differing only in predicate outcome (ADR 0007).
- **Merger preservation** differs from Implementer/Reviewer: merge-sandbox may preserve failure evidence for diagnosis, but the next Merger attempt recreates the sandbox from the current safe SHA instead of resuming an old baseline.
- Session-storage gitignoring uses `.git/info/exclude` entries appended at orchestrator startup. Shared parent `.git` means exclusion applies in every worktree without per-worktree files.
- **`AccountPool`** is an in-memory ordered registry of `(name, token, exhausted_until)` tuples seeded once in `main.py` from `CLAUDE_CODE_OAUTH_TOKEN` (primary) and optional `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY` (secondary, preferred when present). Exposes `pick()`, `mark_exhausted(token, reset_time)` setting per-token wake-time, `has_available()`, `earliest_wake_time()`. Constructed in `main.py`, passed to `AgentRunner.__init__` as separate injectable. Failover at per-agent-run granularity. State in-memory only — process restart re-probes secondary optimistically (ADR 0005). Per ADR 0015: the class moves into `ClaudeService` as a private detail; env-var contract and dual-token semantics preserved byte-for-byte. Claude subscription-access denial can permanently exhaust the active account for the current process. Orchestrator-level failover moves up to the stage fallback chain.
