# Session resume across container teardown

> **Amended 2026-05-10 (#640).** The fail-soft fresh fallback originally chosen here has been removed. Non-typed exceptions on a Resume run are now handled by a single in-call retry against the same session, then converted to `FailedOutput(failure_class="non_typed_crash")` which routes through the existing **Failure-Analysis agent** path. See the *Considered Options* entry "Re-launch as Fresh on any non-typed Resume crash" and the *Consequences* entry "non-typed Resume retry" for the current design.

When an agent is interrupted mid-task the orchestrator preserves the worktree and tears down the container. On the next iteration a fresh agent is spawned with no memory of analysis already done — the work product survives via git and implement-skip / review-skip, but *in-progress reasoning* is dropped on every interruption. We introduce a session-resume primitive: every `claude` invocation runs with `--session-id <SESSION_UUID>` (Fresh) or `--resume <SESSION_UUID>` (Resume), where `SESSION_UUID` is derived per `(role, worktree_path)` and `CLAUDE_CONFIG_DIR` is set to a per-role subdir inside the agent's worktree. A non-empty role session dir is the resume signal. On Resume, only the shared continuation prompt is sent. On any non-typed failure of `--resume`, the call retries once against the same session; on a second non-typed exception returns `FailedOutput(failure_class="non_typed_crash")` routing through the Failure-Analysis path (amended #640).

## Considered Options

- **Status quo (always restart from scratch).** Rejected: implement-skip / review-skip preserves committed work, but in-flight reasoning is exactly what those mechanisms cannot preserve. The motivating case — pool sleep mid-Implementer — discards codebase exploration the agent already paid for.
- **Replace implement-skip / review-skip with session-resume.** Rejected: skip is durable across worktree removal (reads the branch ref); resume is ephemeral with the worktree by design. Orphan sweep or any process touching `pycastle/.worktrees/` would silently demote a completed phase to "redo." Skip and resume are orthogonal.
- **Host-side volume mount for session storage.** Rejected: introduces a second cleanup path (sessions and worktrees can become out of sync) and breaks the symmetry where every artifact lives inside a worktree. The existing worktree-preserve / orphan-sweep machinery transparently handles the lifecycle.
- **Random UUID per agent run, persisted in a file.** Rejected: `CLAUDE_CONFIG_DIR` already isolates session storage per `(worktree, role)`, so within any one config dir at most one session ever exists. A randomly-generated UUID adds a file to maintain while solving no real problem.
- **Three fixed per-role UUID constants or a single fixed UUID for all roles.** Rejected: with parallel agents on different worktrees sharing a constant, any unknown Claude-internal cache keyed on session-id (rather than isolated by `CLAUDE_CONFIG_DIR`) becomes a silent corruption surface. Derivation eliminates the question.
- **Derived UUID per `(role, worktree_path)` — the chosen design.** `uuid.uuid5(uuid.uuid5(NAMESPACE_DNS, "pycastle.<role>"), str(worktree_path.resolve()))`. Structurally unique per agent+worktree; cross-stream leakage impossible; no UUID file needed.
- **Trigger resume only on `UsageLimitError`.** Rejected: filtering by interruption cause requires threading the cause through the cancellation path for no real benefit — fail-soft already bounded the worst case.
- **Localized `<worktree>/.pycastle-session/.gitignore` containing `*`.** Rejected: a `*` rule inside the directory does not hide the directory itself from `git status`; an agent following "keep tree clean" could `git add` session data.
- **Auto-append `/.pycastle-session/` to the consuming project's root `.gitignore`.** Rejected: writes a tracked file in the consuming repo, requires a migration story for existing projects.
- **Idempotent append to `<repo>/.git/info/exclude` — the chosen design.** `.git/info/exclude` is local, never-committed, and shared across all worktrees of the repo. The orchestrator startup appends `.pycastle-session/` and `.claude/` if not already present.
- **Re-launch as Fresh on any non-typed Resume crash (the original fail-soft fresh fallback). Originally chosen, rejected on amendment (#640):** the common-case non-typed crash (transient docker hiccup, malformed stream frame) was triggering a silent history wipe and turning a recoverable interruption into the exact "lose all in-progress reasoning" failure the primitive exists to prevent. The current design retries once against the same session and on a second failure converts to `FailedOutput(failure_class="non_typed_crash")`. The Failure-Analysis agent files a `bug` + `needs-triage` issue with a Recovery section instructing the human to wipe the session dir manually if they suspect transcript corruption.

## Consequences

> **Amended 2026-05-15 (#692).** Claude session files moved from `.pycastle-session/<role>/` into `.pycastle-session/<role>/[<namespace>/]claude/`. `CLAUDE_CONFIG_DIR` and `--resume`/`--session-id` flags are now claude-private: `ClaudeService.state_dir_relpath(role, namespace)` returns the per-service subdirectory path and `ClaudeService.is_resumable(state_dir)` is the per-service resume predicate. The stage-done predicate (empty role dir) is unchanged.

- Every agent container run launches with `CLAUDE_CONFIG_DIR=/home/agent/workspace/.pycastle-session/<role>/[<namespace>/]claude/` and `--session-id <SESSION_UUID>` (Fresh) or `--resume <SESSION_UUID>` (Resume). These are managed by `ClaudeService` via `state_dir_relpath` and `build_env`; the orchestrator learns the path through `AgentService.state_dir_relpath(role, namespace)`.
- `decide_agent_run_kind(role, session_dir_present) → Resume | Fresh` is a pure function. The `Skip` path stays inline in `run_issue` (commit-prefix branching); the dispatch function is only invoked when the agent would otherwise spawn.
- Prompt-shape contract: **Fresh** — `/tmp/.pycastle_prompt` contains only the role prompt. **Resume** — `/tmp/.pycastle_prompt` contains only the continuation prompt (role prompt is in resumed conversation history).
- The worktree-preservation rule broadens to `dirty OR usage_limit OR session_resumable`. The `session_resumable` predicate is `role_dir.is_dir() AND any(role_dir.rglob("*"))`.
- `_agent_worktree.__aenter__` and `branch_worktree.__aenter__` gain a reuse path: if the worktree already exists, the branch matches, and a role under `<wt>/.pycastle-session/` is resumable, the create_worktree call is skipped and the existing worktree is reused as-is.
- `branch_worktree` (merge-sandbox) gains the same preservation symmetry: on `__aexit__`, both `teardown_worktree` and `delete_branch` are skipped together when the predicate holds.
- Cleanup-on-success is load-bearing for teardown: after every successful commit or merge application the orchestrator runs `shutil.rmtree(role_session_dir, ignore_errors=True)` inside the worktree context manager. Without this, the broadened preservation rule would keep the worktree alive forever after success.
- Non-typed Resume retry (amended #640): any exception from `runner.work()` on a Resume run not in `{UsageLimitError, AgentTimeoutError, AgentOutputProtocolError, PreflightFailure}` triggers a single in-call retry with `run_kind=RESUME`. A second non-typed exception converts to `FailedOutput(failure_class="non_typed_crash")`; the orchestrator returns `AbortedAgentFailure` and exits cleanly.
- Session-storage gitignoring via `<repo>/.git/info/exclude`: the orchestrator startup appends `.pycastle-session/` and `.claude/` idempotently. Applies in every worktree without per-worktree files.
- The continuation prompt at `defaults/prompts/_resume-prompt.md` is shared across all roles. After amendment (#640) it assumes conversation history is present: instructs the agent to inspect uncommitted state via `git status`, scan recent messages to identify the last completed step, and continue from there.
- Improve mode (issue #408) builds multi-prompt-within-one-run on top of this primitive without modifying the resume infrastructure.
