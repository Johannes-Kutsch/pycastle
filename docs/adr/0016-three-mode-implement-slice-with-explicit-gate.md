# Three-mode implement slice with explicit gate

Today's single `implement-prompt.md` mixes a long architecture-vocabulary standards block (`{{IMPLEMENTATION_STANDARDS}}`) between §2 Behaviors and §3 Tracer Bullet, and offers no forcing function between Explore and Implement. Under context pressure, the implement agent skips §3 — observed verbatim in the wild as `"Now I have all the context I need. Let me implement the changes:"` followed by a numbered build plan with "Add tests" as the last step (real example: application-pipeline issue #205, where steps 1–2 were a pure module-extraction refactor and steps 3–4 were the behavior change, all bundled into one slice with two pre-baked test-shaped acceptance criteria). The failure has three layered causes: (a) the `to-issues` skill writes acceptance criteria *as tests* ("a test asserts X"), priming the agent to satisfy a verification checklist rather than drive a red/green loop; (b) issues mix refactor-prep steps with behavior steps and the prompt has no story for refactor-shaped work, so the agent defaults to a single build plan covering both; (c) there is no checkpoint between "I understand the problem" and "I'm writing production code" that forces re-entry into TDD. We fix all three: split slice authorship into three mutually-exclusive modes (`refactor-slice`, `behavior-slice`, `docs-slice`) marked by GitHub labels, dispatch to one of three implement prompts per session based on the label, and gate the behavior prompt with a two-phase structure terminated by mandatory `<first_behavior>` and `<failing_test>` tags. The slicing rule is the load-bearing constraint: any step that cannot be verified by a new test of observable behavior goes into a `refactor-slice`, never rides along inside a `behavior-slice`. Multiple refactor steps may be combined into a single `refactor-slice`; the behavior slice that depends on the new shape lists the refactor in `Blocked by`. Refactor slices land first, behavior slices follow.

## Considered Options

- **Fix only the `to-issues` skill (rephrase acceptance criteria as behaviors).** Rejected as the primary lever: the implement prompt's structural problems persist regardless of how well-written the issue is. Under context pressure the agent will still collapse Explore → Implement on a perfectly-written behavior-only issue, because nothing in the prompt makes the phase boundary observable.
- **Fix only the implement prompt (add a gate, leave issues alone).** Rejected as the only change: a gate works against well-formed issues but cannot rescue an issue that bundles refactor-prep with behavior — the agent legitimately has no "first failing test" to emit for steps that don't change observable behavior, so the gate has no answer for that step shape.
- **One implement prompt with conditional sections (`{{#if behavior}}…`).** Rejected: keeps the bloat that caused §3 to get skipped in the first place. The agent still reads two irrelevant disciplines per session, and the "skim past the relevant section" failure recurs. Also introduces template-branching logic the prompt pipeline doesn't currently have.
- **One issue per refactor + one issue per behavior, with mode embedded in the body as a `## Mode` section.** Rejected in favor of GitHub labels: duplicates state already expressible as a label, and the implement agent has to body-parse to dispatch. Labels are the existing mechanism for "what kind of work is this" (per `ready-for-agent` / `ready-for-human`) and are queryable via `gh issue list -l <name>`.
- **One issue per refactor + one issue per behavior, marked by title prefix (`[refactor] …`).** Rejected: title prefixes drift under hand-editing; not queryable as structured data; redundant with labels which are pycastle's existing convention.
- **Two labels: `mode:refactor` and `mode:behavior` (docs unlabeled, default).** Rejected: docs slices left in a silent default reproduce the drift risk `feedback_github_issue_labels.md` warned about. Made all three labels explicit instead.
- **One label `mode:refactor`; absence implies behavior.** Rejected: same silent-default anti-pattern; flips the asymmetry the wrong way, since behavior is the more disciplined mode and should be the explicit one.
- **Namespaced label prefix (`mode:refactor` etc.).** Rejected: introduces a `mode:` namespace convention pycastle's other labels do not use, forcing every future label axis to face "should this be namespaced too?" Flat kebab-case names with `-slice` suffix match the existing `ready-for-agent` style and anchor the labels to the to-issues vocabulary.
- **Slice authors use judgement to decide refactor vs behavior, no stated rule.** Rejected: this is effectively what the skill does today, and is why bundled-issue slices like #205 emerge. Without a stated rule, the default instinct is "make a thin vertical slice that goes end-to-end," which bundles the prep into the behavior cut.
- **Symbol-move rule (only literal symbol moves/renames qualify as refactor-slice).** Rejected as too narrow: real refactor prep often introduces a `Protocol` for dependency injection, rewires imports across packages, or extracts a shared module used by multiple call sites. None are pure symbol moves; all are legitimately pure-refactor with no new observable behavior. The chosen rule ("cannot be verified by a new test of observable behavior") covers these naturally.
- **Touched-file-count rule (count files; multi-module → refactor-slice).** Rejected: file count doesn't track the underlying distinction. A one-file behavior change touching the test and implementation isn't a refactor; a multi-file refactor confined to one package isn't a behavior change.
- **Allow small refactors to ride along inside the behavior-slice when they stay in the files the behavior touches.** Rejected mid-grilling: the implement agent's session mode must be unambiguous (refactor-or-behavior, never both), or the dispatch logic in the prompt has to handle mid-session mode switching — exactly the kind of state machine that erodes under context pressure. The cleaner rule is "if it's not a behavior change, it's a refactor-slice." Multiple refactor steps can be bundled into one `refactor-slice` to avoid issue-count explosion, which addresses the "20 micro-refactor slices" concern without compromising session-mode purity.
- **Three prompts but with refactor and docs sharing a stripped prompt.** Rejected: the discipline overlaps ("no new tests, run feedback commands") but the artifact diverges (refactor preserves behavior under tests; docs touches no code at all). A shared prompt requires hedged language for both, reintroducing the ambiguity three-prompt dispatch was supposed to eliminate. Three separate prompts keep each one short and single-purpose; shared "output rules" boilerplate factors into the existing standards-injection pattern in `prompt_pipeline.py`.
- **Reuse `AgentRole` as the dispatch axis (three implementer roles instead of three templates).** Rejected: `AgentRole` already drives stage tracking (`is_stage_done_for`), log filenames, status display, and worktree session directories. Splitting one implementer role into three forces all those subsystems to learn about slice modes for no benefit — slice mode is a per-issue property, not a per-stage one.
- **Internal renderer-side dispatch (`PromptTemplate.IMPLEMENT` stays singular; `PromptRenderer` peeks at the issue's labels to pick the file).** Rejected: couples `PromptRenderer` to GitHub-issue shape, which it does not currently know about. Threading labels through the renderer is a wider change than dispatching outside it. Explicit dispatch at the call site is grep-able and independently testable.
- **No gate; trust the prompt's TDD framing.** Rejected: this is the status quo, demonstrably insufficient. The example `"Now I have all the context I need. Let me implement the changes:"` shows the agent fluently writing transition sentences past §3; only a mandatory artifact (real pytest output) survives that fluency.
- **Gate as a three-line checklist ("First behavior: X. First test name: Y. Test currently: RED.").** Rejected: checklists get filled in performatively without running the test. The agent can write "Test currently: RED" without it being true.
- **Gate as a required tag without a phase structure.** Rejected: the tag becomes a one-off ritual the agent emits and then resumes its previous flow. The phase structure (Phase A explore + write first failing test; Phase B implement-to-green loop) reframes the *whole prompt* around the gate, so there is no "previous flow" to fall back to.
- **Refactor-implement gate: green-before / green-after (paste test output at session start and end).** Rejected: pycastle's iteration pipeline already pins the implement session to a preflight-verified SHA (per ADR 0014). Re-running the suite at session start is duplicate work the orchestrator has already done. The refactor session's only added discipline beyond standard end-of-session checks is "no new tests" — see Consequences.
- **Refactor-implement gate: characterization test for uncovered symbols before refactoring.** Rejected: pulls the refactor session into writing tests for already-shipped behavior, inverting the rule "refactor slice produces no new tests." `feedback_no_private_method_tests.md` further rules out testing the private helpers a refactor typically produces. Uncovered behavior surfaced by a refactor is filed as a follow-up `behavior-slice`, not patched in-session.
- **Docs slices applied by a pycastle-internal command, bypassing the implement agent.** Rejected: improve mode files dedicated CONTEXT.md / ADR issues that go through the normal implement-and-review pipeline. Bypassing review for docs is the wrong incentive — domain-doc edits are exactly where a second pair of eyes earns its keep.
- **Default the slice-mode label when missing or pick one if ambiguous.** Rejected: silent defaults reintroduce the drift risk that motivated three explicit labels. The picker fails fast on missing/unknown/multiple slice-mode labels and the iteration treats the issue the way it treats other malformed-issue conditions.

## Consequences

- **Three GitHub labels, mandatory per code-or-docs issue.** `refactor-slice`, `behavior-slice`, `docs-slice` — exactly one per issue that produces code or docs. HITL architectural-decision slices stay unlabeled on this axis; their gate is human review (`ready-for-human`), not implement-agent discipline. Names are flat kebab-case to match existing `ready-for-agent` / `ready-for-human` style.
- **Labels are first-class config and provisioned by `pycastle labels`.** `Config` gains three fields (`refactor_slice_label`, `behavior_slice_label`, `docs_slice_label`) defaulting to the literal label names. `src/pycastle/labels.py` gains three entries in the provisioned label set. The canonical label set grows from six to nine.
- **Three new prompt-pipeline placeholders.** `REFACTOR_SLICE_LABEL`, `BEHAVIOR_SLICE_LABEL`, `DOCS_SLICE_LABEL` join the global placeholder pool; they derive from `Config` like the existing `READY_FOR_AGENT_LABEL`.
- **Three implement prompts, one per slice mode.** `behavior-implement.md` (the rewritten descendant of today's `implement-prompt.md`, two-phase TDD with the gate), `refactor-implement.md` (short — change the named symbols, no new tests, run feedback commands at end), `docs-implement.md` (shortest — apply named markdown edits, no code, no tests). `PromptTemplate` gains `IMPLEMENT_BEHAVIOR`, `IMPLEMENT_REFACTOR`, `IMPLEMENT_DOCS`; the singular `IMPLEMENT` is removed (no back-compat shim — this is a single-context repo and the rename lands atomically).
- **Behavior gate.** `behavior-implement.md` is structured as two named phases. Phase A: explore (issue-scoped, not full-repo), name the first behavior from the issue's acceptance criteria, write one failing test, run feedback commands. Phase A terminates with mandatory `<first_behavior>...</first_behavior>` (behavior name + observable surface) and `<failing_test>...</failing_test>` (real pytest output showing the test red). No `Edit` / `Write` on non-test files is permitted before both tags are emitted. Phase B: implement to green, incremental loop for remaining behaviors, refactor, finish. The phase boundary is the load-bearing artifact — the agent cannot transition out of Phase A without real test output.
- **Refactor gate.** No new tests. All existing tests must pass after the change (standard end-of-session feedback-commands run). Uncovered behavior surfaced by the refactor is filed as a follow-up `behavior-slice`, never patched by adding a test in the refactor session. Private-helper testing is forbidden per `feedback_no_private_method_tests.md`.
- **Docs gate.** No code files touched. No new tests. Output is the markdown edits described in `What to build` plus a `<commit_message>` tag.
- **Dispatch site.** A new helper `pick_implement_template(issue, cfg) -> PromptTemplate` lives next to the call site in `src/pycastle/iteration/implement.py`. It reads `issue["labels"]`, expects exactly one slice-mode label, and fails fast with a clear error on missing / unknown / multiple slice-mode labels. The iteration treats the malformed-label case the way it treats other malformed-issue conditions (skip the issue, surface the error).
- **`to-issues` slicing rule.** The skill (and improve `03-issues.md`) state the rule verbatim: "If a step cannot be verified by a new test of observable behavior, file it as its own `refactor-slice`; multiple such steps may be bundled into one `refactor-slice`. Refactor slices land first; the dependent behavior slice lists the refactor in `Blocked by`." Refactor steps never ride along inside a behavior slice.
- **Acceptance-criteria shape per slice mode.** `behavior-slice` criteria use behavior + observable-surface phrasing ("the parser log file contains `http_get_start` for the attempt and does not contain a matching `http_get_ok`"); never test-prescriptive ("a test asserts X"). `refactor-slice` criteria are outcome-shaped ("`current_stage` is imported from `_context`; no behavior change"). `docs-slice` criteria are file-state-shaped ("`CONTEXT.md` contains the term `slice mode` defined as …").
- **Acceptance-criteria banlist.** Both surfaces (`SKILL.md` and `improve/03-issues.md`) explicitly forbid the sentence shapes "a test asserts X", "test verifies X", "unit test X", "the test should X". Authors phrase verification as the system's behavior, not as test-code structure.
- **Surfaces that must move in lockstep.** Three sets of edits per change to slicing rules / criteria shape / banlist / mode-label set: `C:\Users\johan\.claude\skills\to-issues\SKILL.md`, `src/pycastle/defaults/prompts/improve/03-issues.md`, and (for the new labels) `CONTEXT.md` glossary. Three prompt files (`implement/behavior.md`, `implement/refactor.md`, `implement/docs.md`) share their "output rules" boilerplate via the existing standards-injection pattern in `prompt_pipeline.py:150`.
- **Migration of in-flight issues.** Out of scope for this ADR. Existing open issues are unlabeled on the slice-mode axis until manually triaged; the iteration's fail-fast on missing label surfaces them naturally. Improve mode produces correctly-labeled new issues from the first run.
- **No worktree / orchestration changes.** Slice mode is a per-issue label, not a per-stage concept. Stage tracking (`is_stage_done_for`), session resume (ADR-0006, ADR-0015), preflight, planner, reviewer, merger all unchanged.
