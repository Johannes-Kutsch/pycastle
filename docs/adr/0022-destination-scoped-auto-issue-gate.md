# Destination-scoped auto-issue gate, default off

`auto_file_bugs` gates *every* auto-filed issue whose destination is the upstream pycastle repo (`bug_report_repo`, default `Johannes-Kutsch/pycastle`) — regardless of trigger severity. The flag defaults to `False`. A single helper `auto_file_issue(title, body, labels, *, cfg)` owns the gate check, token resolution, API-path attempt via `GithubService.create_issue_in`, and prefilled-URL fallback; every call site delegates to it and otherwise carries no knowledge of repos, tokens, or URL construction. Consuming-project agent-filed issues (e.g. `AgentRole.PREFLIGHT_ISSUE` writing into the user's own repo) are unaffected — the gate is destination-scoped, not severity-scoped, and "destination" specifically means *upstream pycastle*.

The trigger is issue #808: with the existing crash-only auto-filer plus the planned usage-limit-parse-failure reporter (#807), non-developer users running pycastle would file recurring noise into the upstream tracker on each format drift or unhandled exception. The current default `auto_file_bugs=True` was safe only while there was exactly one filing site; a second site forces the question of what `auto_file_bugs` actually denotes.

## Considered Options

- **Status quo: per-site flags (e.g. `auto_file_bugs` for crashes, future `auto_report_parse_failures` for #807).** Rejected. Each new auto-report site adds a config knob; the user-facing mental model degrades from "do I want to talk to upstream?" to a matrix of toggles. The actual axis users care about is destination, not severity.
- **Severity-scoped gate: crashes always file (the user already lost their session); soft reports gated.** Rejected. Conflates two questions — "did the user already crash?" and "should we ping upstream?" — and forces every future report site to be classified on a severity axis the maintainer doesn't actually care about. The reason to suppress filing isn't "this isn't serious enough"; it's "this user isn't the maintainer."
- **Detect dev-checkout at startup (running pycastle from `Johannes-Kutsch/pycastle` clone ⇒ enable).** Rejected. Heuristic. Wrong in forks, CI, contributors-via-fork. A config flag the developer flips once is simpler and explicit.
- **Lint/AST rule that fails CI if a new filing site bypasses the gate.** Rejected: structure rather than behaviour; the helper-funnel achieves the same property by construction.
- **Destination-scoped single gate (chosen).** `auto_file_bugs` means "I want pycastle to file issues into `bug_report_repo` on my behalf." Default `False`. One helper, one mental model, one place to add future filing sites.

## Consequences

- `Config.auto_file_bugs` default flips from `True` to `False` in `src/pycastle/config/loader.py`. Default scaffolded config comment in `src/pycastle/defaults/config.py` documents the developer-opt-in posture.
- `bug_reporter.py` consolidates: `auto_file_issue(title, body, labels, *, cfg) -> None` becomes the single public entry point alongside `report_and_exit`. It owns gate check, token resolution (`_safe_resolve_token`), API attempt (`_try_api_path`), prefilled-URL construction (`build_bug_report_url` with truncation), and the user-visible printed line (`Filed issue #N: <url>` or the bare prefilled URL). `report_and_exit` becomes a thin caller: print traceback to stderr → call helper → `sys.exit(1)`. The previously-public `build_bug_report_url` becomes an internal.
- Body composition stays with the caller (each site composes its own `## Traceback` / `## Failed message` heading); the helper prepends only the shared env block (pycastle/Python/OS).
- `UsageLimitError` gains `raw_message: str | None`; `_check_usage_limit` (Claude) and `_extract_usage_limit` / `_parse_reset_time` (Codex) pass the unparsed line when they identify a usage-limit message they can't fully parse. The `run_iteration` boundary handler (ADR 0008) is the single place that calls `auto_file_issue` for this case and owns a per-process dedupe set keyed by message signature, so a session that repeatedly emits the same broken format files at most once.
- The HITL config-flip is a separate `ready-for-human` issue: maintainer adds `auto_file_bugs = True` to their global `~/.pycastle/config.py`. No CLI helper is provided; the action is one-time per developer machine.
- Consuming-project agent-filed issues (PREFLIGHT_ISSUE, failure-report, improve/02–04 prompts via `{{ISSUE_TRACKER}}` per ADR 0021) are out of scope — they file into the cwd-derived repo, not `bug_report_repo`, and are not gated by this flag.
- A test in `tests/test_bug_reporter.py` asserts that with `auto_file_bugs=False` and a reachable token, the API path is not attempted; the prefilled URL is the only output. A parametrized regression test covers the usage-limit-parse-failure boundary path: injecting an unparseable 429 line yields `UsageLimitError(reset_time=None, raw_message=...)`, and on the first occurrence per process the helper is invoked with the generic title `[pycastle] failed to parse usage-limit reset time (claude|codex)`.
