# Ubiquitous Language

## Package & Distribution

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **pycastle** | The installable Python package that orchestrates Claude Code agents | orchestrator, tool |
| **consuming project** | A project that installs pycastle and customizes it via a local pycastle directory | host project, parent project |
| **pycastle directory** | The `pycastle/` directory inside a consuming project, containing local overrides for config, Dockerfile, .env, and prompts | config dir, override dir |
| **defaults** | Files bundled inside the pycastle package that serve as scaffolding templates and runtime fallbacks | base config, starter files |
| **pycastle init** | CLI command that copies all defaults into the consuming project's pycastle directory, then runs the init wizard | setup, scaffold |
| **init wizard** | Interactive step-by-step flow inside `pycastle init` that collects credentials and optionally creates GitHub labels | setup wizard, onboarding |
| **pycastle labels** | CLI subcommand that creates or resets GitHub labels in a target repo using the canonical label set | label setup, label sync |
| **auto-discovery** | Runtime behavior of looking for a pycastle directory in CWD before falling back to `pycastle home`, then to defaults; project marker is "a `pycastle/` directory exists in CWD" | — |
| **pycastle home** | User-scope global directory holding optional `config.py` and `.env` shared across all consuming projects; resolved via `platformdirs` (`~/.config/pycastle/` Linux/Mac, `%APPDATA%\pycastle\` Windows) and overridable via `PYCASTLE_HOME`; only `config.py` and `.env` are globalizable — `Dockerfile`, `prompts/`, `.gitignore` always live in the consuming project | global config dir, user config dir |
| **PYCASTLE_HOME** | Env var that overrides the `platformdirs` default for `pycastle home`; resolution in `load_config` is explicit `global_dir` arg → `PYCASTLE_HOME` → `platformdirs` default | — |
| **layered config merge** | Precedence stack used by the config loader: `defaults module → pycastle home/config.py → pycastle dir/config.py`, applied field-by-field; `.env` extends to `defaults → global .env → local .env → process env`, applied per-key (process env wins) | config layering, config precedence |
| **global-forbidden fields** | `Config` fields that may not be set in global `config.py` and trigger `ConfigValidationError` at load: `pycastle_dir`, `prompts_dir`, `logs_dir`, `worktrees_dir`, `env_file`, `dockerfile`, `docker_image_name`; inherently project-shaped | global-forbidden path fields, path field guard |
| **scope flag** | The `--global` / `--local` mutually-exclusive CLI flag pair on `pycastle init`; selects whether `config.py` and `.env` are scaffolded into `pycastle home` or local `pycastle/`; absent flag triggers a single interactive prompt; `Dockerfile`, `prompts/`, `.gitignore` always scaffolded locally | global/local flag |
| **layer summary** | One-line message printed at startup by every CLI subcommand listing which config layers were found (e.g. `Config: defaults + ~/.config/pycastle/config.py + pycastle/config.py`); routed via `StatusDisplay.print("", ...)` | config layer line |

## Configuration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **config.py** | Python file in the pycastle directory defining behavioral configuration; overrides defaults module field by field at runtime | settings.py, settings |
| **defaults module** | `src/pycastle/defaults/config.py` bundled in the package; pure default values, no logic | defaults config, fallback config |
| **config loader** | The `loader.py` module inside `config/`; reads defaults, executes global then local `config.py` via importlib (each layer field-by-field overriding), applies programmatic overrides, rejects global-forbidden path fields, validates effort strings, returns immutable `Config`; accepts optional `global_dir: Path | None` (resolved via `PYCASTLE_HOME` then `platformdirs` when `None`); no subprocess calls, no service deps | — |
| **.env** | File holding secrets and credentials only — never committed; may live in local pycastle dir and/or `pycastle home`; per-key layered as `defaults → global .env → local .env → process env` | environment file, config |
| **GH_TOKEN** | The sole GitHub credential: a personal access token in .env, used by host code (`GithubService` HTTP) and propagated into agent containers; validated at orchestrator startup via `check_auth()` (ADR 0004) | github token, gh pat |
| **CLAUDE_CODE_OAUTH_TOKEN** | Long-lived OAuth token generated by `claude setup-token` and stored in `.env`; the **primary account** token in the dual-account model — fallback when secondary exhausted, sole token when only one account configured (ADR 0005) | claude token, oauth token, primary token |
| **CLAUDE_CODE_OAUTH_TOKEN_SECONDARY** | Optional second OAuth token in `.env`; the **secondary account** token — preferred over primary when present and not exhausted (ADR 0005) | secondary token |
| **primary account** | Claude account whose token is `CLAUDE_CODE_OAUTH_TOKEN`; always present, fallback when secondary exhausted | fallback account |
| **secondary account** | Optional Claude account whose token is `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY`; preferred at every spawn when present and not exhausted | preferred account |
| **PREFLIGHT_CHECKS** | Config entry (`list[tuple[str, str]]`) of `(name, command)` pairs run inside the cache's ephemeral `preflight-sandbox` by `ContainerRunner.preflight` whenever `PreflightCache.get_safe_sha()` observes an unseen HEAD; detection-only, no auto-fix (e.g. `ruff check .`) | preflight commands, check list |
| **IMPLEMENT_CHECKS** | Config entry (`list[str]`) rendered into the FEEDBACK LOOPS section of the implement-prompt as agent instructions; may differ from PREFLIGHT_CHECKS (e.g. `ruff check --fix`) | feedback commands, implement commands |
| **field-by-field override** | Config loader strategy: for each non-underscore name in the consuming project's config.py, `setattr` replaces the corresponding name in the loader module; absent names fall back to defaults | full replacement, merge override |
| **STAGE_OVERRIDES** | Config dict with one entry per agent role (`plan`, `implement`, `review`, `merge`, `preflight-issue`, `improve`), each holding model shorthand and effort level | stage config, model config |
| **diagnose_on_failure** | Boolean (default `True`) controlling two behaviours: (1) whether phase 1 of an improve run emitting `<promise>NO-CANDIDATE</promise>` triggers the **no-candidate report** or terminates immediately; (2) whether a `FailedOutput` from any agent spawns the **Failure-Analysis agent** before returning `AbortedAgentFailure` | failure diagnosis flag, no-candidate toggle |
| **stage override** | Per-agent-role `model` + `effort` entry inside STAGE_OVERRIDES | phase config, agent config |
| **model shorthand** | Short family alias (`haiku`, `sonnet`, `opus`) accepted by Claude CLI natively; stored as-is in `Config`, passed through to `claude --model` at stage execution time; not resolved at config load (ADR 0002) | model alias, model name |
| **full model ID** | Versioned Claude model identifier (e.g. `claude-sonnet-4-6`); may be stored directly in a stage override; passed through to `claude --model` unchanged | model ID, model version |
| **effort level** | One of (`low`, `medium`, `high`, `xhigh`, `max`); validated at config load against this fixed set | effort, effort flag |
| **CLI default** | Behavior when no `--model` or `--effort` flag is injected — triggered by an empty string in STAGE_OVERRIDES | default model, unset |
| **ConfigValidationError** | Raised by config loader when an effort level is unrecognised; includes invalid value, closest valid suggestion, and full list of valid options | validation error, config error |
| **auto_push** | Boolean (default `True`) controlling whether `merge_phase` pushes local main to the remote after merges produce commits | push_after_merge, AUTO_PUSH |
| **docker_image_name** | Docker image tag for `pycastle build`, `pycastle run` (implicit build step), and agent containers; global-forbidden; when unset locally, derived at runtime by sanitising `Path.cwd().name` (`re.sub(r"[^a-z0-9]+", "-", name.lower()).strip("-")`); scaffolded local `config.py` ships only a commented-out hint with the derived name | image name, container image |
| **auto_file_bugs** | Boolean (default `True`) gating the API path of the **auto bug reporter**; when `True`, escaped exceptions POSTed via `GH_TOKEN`; when `False`, falls back to prefilled-URL path; orthogonal to whether a report is *produced* | bug toggle, auto report flag |
| **bug_report_repo** | Config entry (default `"Johannes-Kutsch/pycastle"`) holding `owner/repo` target for auto-filed bug reports; validated against `^[^/]+/[^/]+$` (raises `ConfigValidationError`); not global-forbidden | upstream repo, bug repo |

## GitHub Integration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **label** | A GitHub issue classification marker with name, description, and hex color | tag, category |
| **canonical label set** | Ten labels hardcoded in the package: `bug`, `needs-info`, `needs-triage`, `ready-for-agent`, `ready-for-human`, `wontfix`, the three **slice-mode labels** `refactor-slice`, `behavior-slice`, `docs-slice`, and **needs-slice-type** | default labels, label config |
| **label reset** | Option to delete all existing labels in a repo before creating the canonical set | label wipe, clean labels |
| **issue label** | The label value (default: `ready-for-agent`) marking an issue as eligible for agent processing | trigger label, agent label |
| **slice-mode label** | One of `refactor-slice`, `behavior-slice`, `docs-slice`; mandatory on every issue that produces code or docs; selects which implement prompt the iteration dispatches to. Exactly one per issue — missing, multiple, or unknown slice-mode labels are filtered out of the planning-phase candidate list and marked with [[needs-slice-type]]; the implement-phase picker remains as defense-in-depth. Classified for callers by the **slice-mode classifier**. | mode label, slice label |
| **needs-slice-type** | Label applied by `planning_phase` to any `ready-for-agent` issue lacking exactly one slice-mode label (zero or multiple). Synced both directions each iteration: added on the transition into malformed, removed on the transition out. The label coexists with `ready-for-agent` (the issue is not demoted); it just marks the issue as ineligible for this iteration's implement phase. The malformed issue remains in `all_open_issues` so dependent slices can still see it as a blocker | slice-needed label |
| **refactor-slice** | Slice mode for an issue whose steps cannot be verified by a new test of observable behavior (symbol moves, renames, protocol introductions, import rewires, dead-code removal). Multiple such steps may be bundled into one refactor-slice. Refactor slices land first; dependent behavior slices list them in `Blocked by`. Dispatches to `implement/refactor.md` (ADR 0016) | refactor prep, refactor issue |
| **behavior-slice** | Slice mode for an issue that introduces or changes observable behavior verifiable by a new test. Acceptance criteria describe the behavior and its observable surface, never test-code structure. Dispatches to `implement/behavior.md` with the **behavior gate** (ADR 0016) | feature slice, behavior issue |
| **docs-slice** | Slice mode for an issue whose only artifacts are markdown edits (CONTEXT.md additions, ADRs, README updates). No code touched, no tests run. Dispatches to `implement/docs.md` (ADR 0016) | docs issue, documentation slice |
| **slice-mode classifier** | In-process module at `src/pycastle/slice_classifier.py` owning the **slice-mode label** concept. Exposes the `SliceMode` enum (variants for refactor / behavior / docs, each carrying its display name and its `PromptTemplate`; the label string itself is resolved from `Config` at lookup time), the `SliceClassification` discriminated union (a well-formed variant carrying a resolved `SliceMode`, a malformed variant carrying the list of slice-mode labels actually found on the issue), the pure `classify_slice(issue, cfg) -> SliceClassification` function, and the `slice_labels(cfg) -> frozenset[str]` helper. Every call site that reads slice-mode labels off an issue (`_resolve_slice` in `iteration/implement.py`, `partition_by_slice_label` and `_sync_needs_slice_type` in `iteration/planning.py`, `handle_preflight_failure` in `iteration/preflight.py`) routes through this module; no per-caller reconstruction of the slice-label set or the cardinality check (ADR 0016) | slice classifier, mode classifier |

## Agents & Orchestration

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **agent** | A Claude Code instance running inside an isolated Docker container | bot, worker |
| **orchestrator** | The main loop coordinating agent phases across GitHub issues | runner, coordinator |
| **iteration** | One complete plan-bounded loop ending after all planned issues attempted; up to MAX_ITERATIONS per `pycastle run`; in parallel mode a single merge phase closes the iteration, in sequential mode each issue gets its own merge before the next starts | cycle, round, pipeline |
| **3-phase loop** | One iteration's structure: plan, implement+review, merge | pipeline, workflow |
| **sequential mode** | Auto-activates when `max_parallel = 1`; within one iteration each issue is processed individually — implement → review → merge — with safe SHA re-pinned from HEAD after each merge before the next starts; eliminates merge conflicts from parallel branches sharing the same base SHA | serial mode, one-at-a-time mode |
| **plan phase** | Phase where the Planner analyzes open issues and produces a plan | planning step |
| **implement phase** | Phase where Implementers fix individual issues in isolated worktrees | coding step |
| **review phase** | Phase where the Reviewer checks an Implementer's changes before merge | code review step |
| **merge phase** | Phase where completed branches are integrated into main, issues closed, and — when conflicts exist — the Merger spawned | integration step |
| **Planner** | Agent role for plan phase; runs in a clean checkout of the safe SHA so it sees the same committed state Implementers will start from; outputs a plan | planning agent |
| **Implementer** | Agent role for implement phase; one per issue | coding agent, implementation agent |
| **Reviewer** | Agent role running after an Implementer completes; validates changes before merge. Renders a `PER_ISSUE` scope template (`PromptTemplate.REVIEW`); **does not** receive a precomputed diff — the reviewer runs `git diff main...` itself inside the container so branch lag does not pull unrelated content into the prompt | review agent |
| **Merger** | Agent role spawned during merge phase only when at least one conflicting branch exists; integrates only conflicting branches and closes their issues | merge agent, integration agent |
| **preflight-issue agent** | Agent spawned by `PreflightCache.get_safe_sha()` when `PREFLIGHT_CHECKS` reports a failure; explores codebase to find root cause, evaluates HITL, files one structured GitHub issue, outputs the issue number as `<issue>NUMBER</issue>` | bug-report agent, error reporter |
| **HITL verdict** | Routing decision encoded in the preflight issue's label after the preflight-issue agent completes; `ready-for-agent` → spawn one Implementer, `ready-for-human` → orchestrator aborts | HITL decision, routing verdict |
| **safe SHA** | The SHA most recently observed at HEAD by `PreflightCache.get_safe_sha()` and confirmed by `PREFLIGHT_CHECKS` in an ephemeral `preflight-sandbox`; held as a single-slot verdict on `PreflightCache` that survives across iterations until HEAD advances. Fetched on demand by `improve_phase`, `planning_phase`, and `merge_phase` via `deps.preflight_cache.get_safe_sha(deps)`. `implement_phase` does **not** call the cache — it consumes the SHA from `PlanReady.sha` / the preflight-fix verdict, pinned by the upstream caller (ADR 0014) | verified SHA, clean SHA |
| **preflight pull** | The `git pull --ff-only` that runs inside every `PreflightCache.get_safe_sha()` call, before HEAD is read and the cache slot consulted; waits for clean working tree; aborts if remote has diverged or is unreachable | — |
| **`PreflightCache`** | Process-scoped object owning the safe-SHA verdict; constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Single private slot `_verdict: PreflightReady \| PreflightAFK \| PreflightHITL \| None` plus an `asyncio.Lock` serialising concurrent callers. Public method `async def get_safe_sha(deps) -> PreflightResult`: under the lock, waits for clean working tree, runs `git pull --ff-only`, reads HEAD; if the slot's SHA matches HEAD returns the cached verdict; otherwise opens a `transient_worktree("preflight-sandbox", sha=HEAD)`, runs `PREFLIGHT_CHECKS` via `agent_runner.run_preflight`, on success stores `PreflightReady(sha=HEAD)`, on failure runs the **preflight-issue agent** and stores `PreflightAFK \| PreflightHITL` carrying the filed issue number. Single-slot suffices because `git pull --ff-only` only moves HEAD forward | preflight cache, safe-sha cache |
| **preflight-fix path** | Orchestrator routing when `PreflightCache.get_safe_sha()` returns `PreflightAFK`: the freshly-filed preflight issue is handed straight to `implement_phase` in the same iteration — `planning_phase` is bypassed entirely (see **planning skip**); normal Reviewer → merge pipeline follows | preflight fast path |
| **programmatic merge path** | Fast-path merge logic that runs `git merge --no-edit` directly via subprocess without spawning the Merger; used when all branches merge cleanly | fast path, direct merge |
| **clean merge** | A `git merge --no-edit` that exits zero and requires no conflict resolution | conflict-free merge, successful merge |
| **conflicting branch** | A branch whose `git merge --no-edit` exits non-zero; `git merge --abort` is run immediately and the branch is collected for the Merger | failed merge branch |
| **commit body composition** | Host-side commit body produced by `run_issue()` after a clean agent exit: `f"Implement #{n} - {message_or_title}"` for Implementer and `f"Review #{n} - {message_or_title}"` for Reviewer, where `message` is `<commit_message>` content if present, otherwise issue title (synthetic fallback). Body carries no orchestrator-readable stage marker — stage-done is filesystem-based (ADR 0007) | commit message format |
| **`<commit_message>` tag** | Optional XML tag emitted by Implementers and Reviewers carrying the agent's plain description of changes; orchestrator uses its content as the commit body when present, falls back to a synthetic body when absent. Absence is **not** a failed run (ADR 0007) | — |
| **in-flight issue** | Open issue with an existing `pycastle/issue-<n>` branch or worktree from a previous interrupted iteration; signals partial or full completion | mid-flight issue, resumed issue |
| **merge-time preflight skip** | When `merge_phase`'s call to `deps.preflight_cache.get_safe_sha(deps)` (over the post-clean-merge HEAD, before spawning the Merger) returns `PreflightAFK \| PreflightHITL`: log a diagnostic, skip the Merger, return `MergeResult` with conflict issues pending. The cache's single slot now holds the broken-HEAD verdict, so the next iteration's first `get_safe_sha()` call returns the same cached verdict (no duplicate issue filed) and the orchestrator's preflight-fix path routes the existing issue through `implement_phase` | merge preflight abort |
| **planning skip** | Pipeline-level bypass of `planning_phase` triggered in two cases: (1) **in-flight present** — any open issue is in-flight, so `run_iteration` hands the in-flight set straight to `implement_phase`; (2) **preflight-fix AFK** — `PreflightCache.get_safe_sha()` returned `PreflightAFK` carrying a single freshly-filed issue, routed directly to `implement_phase` | plan bypass |
| **implement skip** | Behavior in `run_issue` when `.pycastle-session/implementer/` exists and `RoleSession(...).is_resumable()` is false (Implementer ran to clean exit, session content cleared): Implementer is not spawned and Reviewer runs directly. See **stage-done signal** and ADR 0007 | — |
| **review skip** | Behavior in `run_issue` when `.pycastle-session/reviewer/` exists and `RoleSession(...).is_resumable()` is false: both Implementer and Reviewer are skipped and the issue is counted complete. Works for the empty-Review case (no commit required). See ADR 0007 | — |
| **stage-done signal** | Per-role completion predicate `RoleSession(<wt>, <role>).is_done()`. Role dir created when agent first starts; on clean exit orchestrator clears its session JSONL contents but leaves the dir, flipping the predicate from "in progress" to "done." Replaces the prior commit-prefix scan (ADR 0007). Honoured by `managed_worktree.__aexit__` via the **session role-dir-present predicate** so the sentinel survives teardown between implement and review in the same `run_issue` call | stage completion check |
| **plan** | Structured JSON from Planner listing which issues to work on and the branch name for each; `plan_phase()` sorts issues by ascending issue number so older issues are processed first | plan output, plan JSON |
| **issue** | A GitHub issue labeled for agent processing, representing one unit of work | ticket, task, card |
| **AFK issue** | Issue the Planner assigns to an Implementer because it can be resolved autonomously; labeled `ready-for-agent` | agent issue, auto issue |
| **HITL issue** | Issue requiring human intervention; labeled `ready-for-human` — Planner must never assign it to an Implementer | manual issue, human issue |
| **blocker** | An issue that must be resolved before another can be worked on. Any open non-`wontfix` issue (regardless of label) counts as a hard blocker for the Planner; `wontfix` is treated as effectively closed | dependency, prerequisite |
| **all-open issues list** | Lightweight `[{number, title, labels}]` projection of every open issue (any label) injected into the plan prompt as `ALL_OPEN_ISSUES_JSON`; gives the Planner cross-label visibility so it can detect when an AFK issue is blocked by a `ready-for-human` / `needs-info` / `needs-triage` issue | blocker visibility list |
| **ready-for-agent issues list** | Hydrated `[{number, title, body, labels, comments}]` projection of open issues carrying the configured `issue_label`; injected into the plan prompt as `READY_FOR_AGENT_ISSUES_JSON`; the candidate set the Planner picks from for `<plan>.issues` | AFK issues list, candidate set |
| **AllBlocked** | Outcome returned by `planning_phase` when the Planner produces zero implementable issues (every AFK issue is blocked by something in the **all-open issues list**); `run_iteration` ends the iteration with `Done` — no within-iteration improve dispatch from this path; the next iteration's preflight + `should_dispatch_improve` will reach `DispatchImprove` naturally if conditions still hold | empty plan, all-blocked |
| **dependency graph** | The set of blocker relationships between issues, analyzed by the Planner to determine the safe working set | issue graph, dependency map |
| **worktree** | Isolated git working tree created on the host and bind-mounted into an agent container; one of: named-branch worktree (Implementer/Reviewer/Merger), named-branch merge-sandbox (Merger), named-branch improve-sandbox (improve agent), or detached plan-sandbox (planning phase) | workspace, branch dir |
| **plan-sandbox worktree** | Temporary detached checkout of safe SHA created by `planning_phase` when the Planner is invoked; always removed in `try/finally`; never branch-associated; at `.pycastle/.worktrees/plan-sandbox` | planner worktree |
| **merge-sandbox worktree** | Temporary named-branch worktree (`pycastle/merge-sandbox`) created by `merge_phase` from HEAD after clean merges; Merger runs inside it to resolve conflicts; always removed in `try/finally`; on success `merge_phase` fast-forwards `main` from the branch before cleanup; at `.pycastle/.worktrees/merge-sandbox` | merger worktree, conflict worktree |
| **improve-sandbox worktree** | Named-branch worktree (`pycastle/improve-sandbox`) created by `improve_phase` from safe SHA when the improve agent dispatches; agent scans codebase, writes a PRD, files sub-issues — branch is never committed to but exists so the **broadened preservation rule** keeps the worktree alive across pool sleep with no new policy; preflight skipped inside it; uses `managed_worktree(name="improve-sandbox", branch="pycastle/improve-sandbox", delete_branch_on_teardown=True, ...)`, mirroring merge-sandbox; at `.pycastle/.worktrees/improve-sandbox` | improve worktree, picker worktree |
| **branch** | Git branch name assigned to an issue inside the plan; pattern `pycastle/issue-<n>-<slug>` | feature branch, issue branch |
| **orphan worktree** | Worktree directory under `.pycastle/.worktrees/` no longer registered in git, typically left by a crashed agent | stale worktree, leftover worktree |
| **orphan sweep** | Startup operation that cross-references `.pycastle/.worktrees/` against `git worktree list --porcelain`, deletes unregistered directories, and removes the `.worktrees` parent if no active children remain | worktree cleanup, stale cleanup |
| **collision detection** | Mechanism preventing two parallel agents from simultaneously creating worktrees for the same branch, implemented as a per-branch async lock | — |

## Prompts

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **prompt** | A markdown file in the prompts directory driving an agent's behavior for one phase | instruction, template |
| **prompts directory** | The `prompts/` subdirectory inside the pycastle directory holding all prompt files | templates dir |
| **placeholder** | A `{{VARIABLE}}` token inside a prompt, substituted at render time. Every placeholder belongs to either the **global placeholder pool** (available in any template) or exactly one **scope placeholder pool** (available only in templates of that scope) | template variable, slot |
| **placeholder provider** | Code that produces a placeholder's value. Global providers read run-level state from `Config` (label fields, check tuples) or from fixed files (`coding-standards/*.md`); scope providers receive the scope's runtime context (an issue, a branches list, a session sid) and return the scope's dynamic dict | placeholder source, value source |
| **global placeholder pool** | Placeholders available in every template regardless of scope. Members: `TESTING_STANDARDS`, `MOCKING_STANDARDS`, `INTERFACES_STANDARDS`, `DEEP_MODULES_STANDARDS`, `REFACTORING_STANDARDS`, `LANGUAGE_STANDARDS`, `DEEPENING_STANDARDS`, `ISSUE_TRACKER`, `BUG_LABEL`, `ENHANCEMENT_LABEL`, `READY_FOR_AGENT_LABEL`, `READY_FOR_HUMAN_LABEL`, `NEEDS_TRIAGE_LABEL`, `NEEDS_INFO_LABEL`, `WONTFIX_LABEL`, `REFACTOR_SLICE_LABEL`, `BEHAVIOR_SLICE_LABEL`, `DOCS_SLICE_LABEL`, `FEEDBACK_COMMANDS`, `CHECKS`. All derive from `Config` or fixed files and are computed once at run start | global pool, universal placeholders |
| **scope placeholder pool** | Placeholders specific to a single **prompt scope**. Members vary per scope (see **prompt scope** entry). Always supplied by the callsite at render time, never cached | scope pool, scope-specific placeholders |
| **prompt scope** | The runtime context a template renders against, declared as a property of each `PromptTemplate` enum variant. Determines which scope placeholder pool applies. Seven scopes: `PER_ISSUE` (`ISSUE_NUMBER`, `ISSUE_TITLE`, `ISSUE_BODY`, `ISSUE_COMMENTS`, `BRANCH`), `MERGE` (`BRANCHES`), `PLAN` (`ALL_OPEN_ISSUES_JSON`, `READY_FOR_AGENT_ISSUES_JSON`), `PREFLIGHT` (`CHECK_NAME`, `COMMAND`, `OUTPUT`), `IMPROVE_SCAN` (∅), `IMPROVE_SESSION` (`IMPROVE_SHORT_SID`), `RESUME` (∅) | scope, template scope |
| **`PromptTemplate`** | Enum naming each shipped prompt template; each variant carries its filename and **prompt scope**. The complete registry of templates pycastle knows about — adding a template means adding an enum variant | template registry, prompt enum |
| **`PromptRenderer`** | Single object owning all prompt rendering for a run. Constructed once per `pycastle run` / `pycastle improve` with `cfg`; computes the **global placeholder pool** once and caches it; at construction validates every shipped template's `{{TOKEN}}`s are in `(global pool ∪ template scope pool)` and aborts the run on any unknown or out-of-scope token. Exposes `render(template, scope_args)` which requires `scope_args.keys() == template.scope.placeholders` exactly (fail-loud on missing/extra/typo'd keys), preprocesses `` !`shell` `` expressions on the raw template, merges scope_args with cached globals, and substitutes `{{placeholders}}`. Substituted content stays inert and is never re-scanned for shell expressions. Owned by `agent_runner` | prompt pipeline, renderer |
| **shell expression** | A `` !`command` `` token inside a prompt, replaced by the command's stdout at preprocess time; expanded only against the raw template, never against substituted content | shell expansion |
| **CODING_STANDARDS.md** | A reference document placed in the prompts directory and treated as a prompt for discovery and scaffolding purposes | standards file |
| **EXPLORATION section** | Section of the implement prompt instructing the Implementer to read files before coding; scoped to files mentioned in the issue body — not a full repository survey | explore section, discovery section |
| **FEEDBACK LOOPS section** | Section of the implement prompt instructing the Implementer to run IMPLEMENT_CHECKS before committing | feedback section, pre-commit checks |
| **implement/behavior.md** | Implement prompt for `behavior-slice` issues (`PromptTemplate.IMPLEMENT_BEHAVIOR`, `PER_ISSUE` scope); two-phase structure terminated by the **behavior gate**. Descended from the pre-ADR-0016 `implement-prompt.md` | behavior prompt, tdd prompt |
| **implement/refactor.md** | Implement prompt for `refactor-slice` issues (`PromptTemplate.IMPLEMENT_REFACTOR`, `PER_ISSUE` scope); short — change the named symbols, no new tests, standard end-of-session feedback-commands run. | **prompt-level dedup** | Resume-safety mechanism for the multi-issue-filing phases (phase 03 sub-issues, phase 04 no-candidate report): every such phase opens with a `GET /search/issues?q=[improve-{{IMPROVE_SHORT_SID}}]+in:title+repo:OWNER/REPO` REST call (recipe in `_issue-tracker.md`) and the prompt instructs the agent to skip filings already present; orchestrator does not enforce — trust placed in the prompt; duplicate filings remain a possible-but-rare outcome if the agent ignores the dedup instruction. Phase 02 (parent PRD) carries no dedup check — under **per-phase session namespacing** the orphan-reset path wipes `_phase_progress` on the rare crash window between phase 02 completion and phase 03 start, so phase 02 never re-runs against an existing PRD; an orphan PRD on GitHub is the documented trade-off | filing dedup, idempotency check |
| **vertical-slice rule** | Phase 3 size constraint replacing a numeric cap: each sub-issue must (a) cut end-to-end through every layer it touches, (b) be independently mergeable, (c) be entirely AFK-implementable; preference for many thin slices over few thick; no upper bound on slice count — a numeric cap was rejected because it pushes the agent to bundle work into thicker slices | slice constraint, tracer-bullet rule |
| **session namespace** | String partition folded into both the **derived session UUID** and the **role session dir** path so one agent role can run multiple isolated Claude conversations within a single worktree; default empty for every role except `improve` (which uses `main` for phases 01/02/04 and `issues` for phase 03); `RunRequest.session_namespace` is the carrier; designed so phase 03's vertical-slicing work does not pay the cache-warming cost of phase 01's full codebase scan and phase 02's full PRD draft sitting in the conversation transcript | session group, conversation namespace |
| **parent PRD handoff** | Mechanism by which the PRD issue number crosses the phase 02 → phase 03 **session namespace** boundary: phase 02's prompt emits the parent issue as JSON-form `<issue>{"number": N, "labels": []}</issue>`, the IMPROVE-role parser in `process_stream` recognises it (alongside the bare-integer `<issue>N</issue>` tags phase 03 emits per filed sub-issue) and surfaces it as `IssueOutput(number=N, labels=[])`; `ImprovePhaseDriver` stashes `output.number` in memory via `record_outcome`; `improve_phase` then calls `GithubService.get_issue` and `get_issue_comments` once per phase 03 entry to re-fetch live PRD title/body/comments, and inlines them into phase 03's prompt as `{{ISSUE_NUMBER}}` / `{{ISSUE_TITLE}}` / `{{ISSUE_BODY}}` / `{{ISSUE_COMMENTS}}` placeholders (driver remains free of network I/O). Re-fetched every entry rather than disk-cached so a human edit to the PRD between phase 02 and phase 03 is honoured; not persisted to disk so a crash in the gap between phase 02 progress write and phase 03 first run triggers **orphan-after-02 reset** | PRD passthrough, ISSUE_* injection |
| **orphan-after-02 reset** | Recovery branch in `ImprovePhaseDriver.start()` for the rare crash window where the orchestrator dies after phase 02 writes `_phase_progress = "02-prd"` but before phase 03's `_phase_in_flight` is written: at loop entry, if `last_id == "02-prd"` AND `in_flight_id != "03-issues"`, the in-memory PRD number is gone and there is no recovery path (no disk persistence by design); orchestrator unlinks `_phase_progress`, the loop restarts from phase 01, picks a new candidate, files a new PRD. The previous PRD on GitHub is left open with `[improve-<old-sid>]` magic title prefix and no triage label — invisible to `cfg.issue_label` queries, manual `gh issue close` to clean up. A crash *during* phase 03 (after PRD body has reached Claude turn 1 in the `issues` namespace) is recoverable with no orphan via standard Resume | orphan PRD reset, lost-prd-number recovery |

## Agent Service Abstraction (ADR 0015 — forward-looking)

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **AgentService** | Protocol implemented by every agent backend (claude, codex, future local-GPU). Sits at the **streaming-execution seam**: owns command construction, env injection, wire-format parsing, and its own resume contract. Container/worktree/log/timeout machinery stays shared in `ContainerRunner`. Methods (names TBD): `name`, `is_available(now)`, `next_wake_time()`, `mark_exhausted(reset_time)`, `state_dir_relpath(role, namespace) -> str \| None`, `is_resumable(state_dir) -> bool`, `build_env(state_dir_container_path) -> dict[str,str]`, async `run(...) -> Iterator[ParsedTurn]` | agent backend, model service, agent provider |
| **streaming-execution seam** | The layer where `AgentService` plugs in: between `ContainerRunner` (owns prompt-file write, `session.exec_stream`, log persistence, idle-timeout via `queue.Queue`, `on_turn`/`on_tokens` callbacks) and the per-service command/parser. Rejected alternatives: narrower CLI-builder seam (forces every service to mimic claude's stream-json) and wider full-agent-run seam (duplicates docker/session machinery per service) — see ADR 0015 | service seam |
| **ParsedTurn** | The 4-event vocabulary that crosses the streaming-execution seam: `AssistantTurn(text)`, `Tokens(count)`, `Result(text)`, `UsageLimit(reset_time)`. Service yields these from its native wire format; the shared coordinator routes them into existing per-role handlers in `_HANDLERS`. Deliberately minimal — `ToolUse`/`Reasoning`/`ToolResult` events are not in the vocabulary until a real consumer needs them | turn event, normalized event |
| **service resume state** | Per-`(worktree, role, namespace, service)` state directory at `.pycastle-session/<role>/[<namespace>/]<service>/`; opaque to the orchestrator, owned by the service. `AgentService.is_resumable(state_dir)` is the per-service resume predicate, polymorphic over services (claude: directory non-empty; a stateless service: always False). Separate axis from **stage-done signal**, which stays service-agnostic at the role-dir level | per-service resume dir |
| **stage fallback chain** | Per-stage failover configured via nested `StageOverride.fallback: StageOverride \| None`; primary and fallback each carry a full `(service, model, effort)` triple so the fallback can be tuned independently (e.g. codex's best implement model differs from its best review model). Dispatch loop: ask primary `service.is_available()` → use if yes; else ask `fallback.service.is_available()` → use if yes; else sleep until `min(next_wake_time across involved services)`. Snap-back is automatic — every new dispatch re-evaluates primary first. Replaces the rejected single global service-priority list | fallback override, per-stage failover |
| **WIP commit** | Host-driven commit produced by the orchestrator on `UsageLimitError` / `AgentTimeoutError` before container teardown: `git add -A && git commit -m "WIP: <role> #N - interrupted"` with the agent's last assistant turn (from `logs/<agent>.log`) as commit body. Carries cross-service mid-stage handoff state durably — when a different service picks up next iteration, it sees the WIP commit on the branch. Squashed away at stage completion via `git reset --soft <anchor>` walking back from HEAD skipping `^WIP: <role> #N -` subjects. **`AgentFailedError`** does not WIP-commit (failed-agent dirty trees are not handoff state) | interrupt commit, handoff commit |
| **WIP-aware prompt clause** | Conditional addition to the role prompt fired only when both (a) WIP commits exist on the branch for this role+issue and (b) the picked service has no resume state in this worktree (i.e. cross-service pickup, not same-service `--resume`). Instructs the agent to inspect `git log` / `git diff` of the WIP commits and continue the stage's goal from there. Same-service resume gets neither the clause nor the WIP narrative — claude already has its conversation in `--resume` history | cross-service handoff clause |
| **FRESH-start hygiene** | Idempotent `git restore --staged --worktree . && git clean -fd` against the worktree before every FRESH agent dispatch; protects against leftover dirty state from a crashed prior attempt that never produced a WIP commit. No-op in the happy path | fresh tree wipe |
| **service registry** | Top-level config field `services: dict[str, AgentService]` plus `default_service: str` declaring all available services by name. Per-stage `StageOverride.service: str` references entries by key. Empty `service=""` resolves to `default_service`. Separate from **stage fallback chain** — the registry defines what's available, the fallback chain decides preference order per stage | service map, service config |
| **lazy service registry** | Construction strategy in `main.py` for the service registry: walk every stage override and the top-level `default_service` / `default_fallback`, collect the set of service names referenced, instantiate exactly those `AgentService` implementations. Claude-only configs never construct `CodexService`; unknown service names are caught by the **fail-fast service+effort validation pass** before instantiation is attempted | on-demand registry, demand-driven registry |
| **fail-fast service+effort validation** | Validation pass at the top of `pycastle run` (after config load, before any container starts): walks every effective `(stage, service, effort)` pair including each `StageOverride.fallback`'s triple, asserts the service name is in the registry and `effort in service.valid_efforts()`. Collects every violation and reports them in a single error message — naming each `(stage, service, effort, valid_set)` tuple — then exits non-zero. Model strings remain unvalidated (both Claude CLI and codex CLI accept free-form model strings) | startup validation, run-time config check |
| **`valid_efforts()`** | Method on the `AgentService` protocol returning the `frozenset[str]` of effort strings the service's CLI accepts. `ClaudeService` returns `{low, medium, high, xhigh, max}`; `CodexService` returns `{none, minimal, low, medium, high, xhigh}`. Consumed by the fail-fast validation pass; per-service because the sets genuinely differ | effort allowlist, service efforts |

## Codex Service (ADR 0020 — forward-looking)

| Term | Definition | Aliases to avoid |
| --- | --- | --- |
| **CodexService** | Forward-looking (ADR 0020): second implementation of the **AgentService** protocol from ADR 0015, driving the headless `codex exec` CLI. Single-account for v1 (mirrors `ClaudeService` safety margins: `+2 min` on top of any parsed reset time, next-hour-boundary `+2 min` when reset time is unknown). Owns codex-specific `build_command` argv branching, `build_env` (sets `TZ=UTC` + `CODEX_HOME=/home/agent/.codex`), per-role state-dir layout, JSONL wire-format parser, and codex-flavoured usage-limit regex | Codex wrapper, codex provider |
| **`~/.codex/auth.json`** | Codex CLI's OAuth credential file on the host, produced by `codex login` and rotated in place on every refresh (codex-rs `auth/manager.rs:792-930`). Holds an access token, refresh token, account id, and the resolved `AuthMode` — opaque to pycastle. Required on the host before `pycastle init` is run with codex selected; pycastle does not invoke `codex login` itself | codex creds, codex token file |
| **codex auth bind-mount** | Per-role bind-mount of `<role-state-dir>/codex/` (containing the seeded `auth.json`) to `/home/agent/.codex/` inside the agent's container — codex's default `$CODEX_HOME`. Each role gets its own `auth.json` lineage that diverges after first refresh; concurrent refreshes across roles do not collide because each role uses its own refresh-token chain. Replaces the rejected single-shared-mount design which would race on `refresh_token_reused` | codex mount, codex creds mount |
| **codex auth seeding** | Init-time copy of host `~/.codex/auth.json` into every role's `<role-state-dir>/codex/auth.json`. Eager — happens once per `pycastle init` invocation per role, idempotent across re-runs (re-init seeds newly-added roles without disturbing existing role dirs). Only `auth.json` is copied — never `config.toml`, never `sessions/` — because pycastle owns the runtime contract for model/effort/sandbox/approval and inheriting host config invites footguns (host MCP servers referencing host-only paths). Rejected alternatives: lazy seeding on first run, copying the full directory, single shared host mount | codex creds seed, auth.json fan-out |
| **host-token invalidation** | Consequence of **codex auth seeding**: after any per-role copy of `auth.json` performs its first refresh, the host's `~/.codex/auth.json` is effectively orphaned from codex's POV (its refresh token is one rotation behind any role's). Adding a new role later therefore requires another `codex login` on the host plus re-running `pycastle init`. Acceptable because role addition is an init-time event and `codex login` is a one-browser-click flow | host creds staleness, init re-seed requirement |
| **`thread_id`** | Codex's per-conversation identifier emitted on the `thread.started` JSONL event (`codex-rs/exec/src/exec_events.rs`). Captured by `CodexService.run` on first emission, persisted via the existing `RoleSession` mechanism, and passed back as the positional argument of `codex exec resume <thread_id>` on the next dispatch. Diverges from claude's session-resume model where pycastle generates the UUID and feeds it via `--session-id` — codex generates the id server-side | codex session id, codex thread |
| **codex resume contract** | Per-service resume contract for `CodexService`: `build_command` branches argv on `session_uuid`. Fresh: `codex exec [flags] <prompt>`. Resume: `codex exec resume <thread_id> [flags] <prompt>`. `is_resumable(state_dir) := state_dir/sessions/ contains at least one rollout-*.jsonl` — the pre-seeded `auth.json` does NOT count as a resume signal | codex --resume, codex thread resume |
| **codex JSONL envelope** | Codex's `--json` output format: line-delimited `ThreadEvent`s tagged by `"type"`. Events consumed by `CodexService.run`: `thread.started {thread_id}` (side-effect: persist via `RoleSession`), `item.completed type=agent_message` (→ `AssistantTurn`), `turn.completed.usage` (→ `Tokens(input+cached+output+reasoning)` then return), `turn.failed.error.message` / `error.message` (→ `UsageLimit(reset_time)` when usage-limit pattern matches). Events dropped: `turn.started`, `item.started`/`item.updated`, `item.completed type=reasoning`/`command_execution`/`file_change`/`mcp_tool_call`. No `Result` yield — codex has no equivalent of claude's `type:"result"` event; the existing `process_stream_from_events` fallback terminates correctly on `turn.completed` | codex stream-json, codex event stream |
| **codex usage-limit regex** | Detection + reset-time extraction for codex's `turn.failed.error.message`. Trigger substring: `You've hit your usage limit`. Reset-time extraction: trailing `(?:or )?try again at <time>` clause in two shapes — same-day `H:MM AM/PM` and cross-day `Month Dth, YYYY H:MM AM/PM`. With container `TZ=UTC` (enforced by `CodexService.build_env`), the extracted time is UTC by construction. No match (degraded `Try again later.` variant) → `UsageLimit(reset_time=None)`, which falls through to the next-hour `+2 min` default. Brittle by necessity: codex's `--json` flattens the structured `RateLimitSnapshot` to a human-readable string (tracked upstream in #778) | codex limit parser, codex reset regex |
| **codex sandbox + approval pins** | `CodexService.build_command` unconditionally emits `--sandbox danger-full-access` and `-c approval_policy=never`. Pycastle's container is the isolation boundary; an inner codex sandbox would only fight the existing container layout, and `--ask-for-approval` does not exist on `codex exec` anyway (approval policy must be set via `-c`). Not user-tunable in v1 | codex hardening flags, codex auto-approve |
| **codex effort flag** | Reasoning-effort selection for codex. No CLI flag exists on `codex exec`; the value is passed via `-c model_reasoning_effort=<value>`. Accepted values: `none | minimal | low | medium (default) | high | xhigh`. Overlaps with **claude effort levels** on `{low, medium, high, xhigh}` only — codex-only `{none, minimal}`, claude-only `{max}` — which is why **`valid_efforts()`** is per-service | codex --effort, reasoning effort |
| **codex auth verify** | Step in the init wizard's codex branch: assert host `~/.codex/auth.json` exists; if not, print `"No codex credentials found at ~/.codex/auth.json. Run 'codex login' and re-run 'pycastle init'."` and exit non-zero. Matches the `claude setup-token` precedent — pycastle does not own the codex login UX. Failure does not roll back claude's `.env` write from a prior step (init is idempotent; re-running after `codex login` completes the codex branch with the claude branch as a no-op) | codex login check, codex creds preflight |
| **service-selection prompt** | First prompt in the init wizard: `"Which agent services do you want to use? [claude/codex/both]"` (default `claude` for back-compat — existing users re-running init see no surprise codex flow). Drives both the per-service env-collection branches (`CLAUDE_CODE_OAUTH_TOKEN` prompt for claude, **codex auth verify** + seeding for codex) and the **Dockerfile template selection** | init service prompt, agent selector |
| **Dockerfile template selection** | Init-time choice between two static templates in `src/pycastle/defaults/`: `Dockerfile.claude` (current behaviour) and `Dockerfile.claude-codex` (adds Node.js runtime + `npm install -g @openai/codex`). Init picks based on the **service-selection prompt**'s answer and copies into `pycastle/Dockerfile` via existing skip-if-exists semantics — user-owned thereafter. `pycastle init --refresh` re-picks the template by walking the config (no re-prompt) and overwrites. ## Test Anti-Patterns (Red Flags)

| Term | Definition | Why it's a problem |
| --- | --- | --- |
| **Verifying through external means** | Tests that call subprocess, query databases, or check external state directly instead of testing through the service interface | Requires external tools in test environment; failures reflect environment problems, not code bugs |
| **Mocking internal collaborators** | Tests that mock classes or functions within the same codebase rather than using dependency injection | Creates brittle tests coupled to implementation; breaks on refactoring |
| **Testing private methods** | Tests that call functions prefixed with `_` or access private attributes | Private methods are implementation details; test public behavior instead |
| **Asserting on call counts/order** | Tests that verify internal function calls happened N times or in a specific sequence | Breaks on refactoring; couples tests to implementation rather than behavior |
| **Test name describes HOW not WHAT** | Test names like `test_calls_git_config` instead of `test_setup_configures_git_identity` | Developers cannot understand test intent from the name alone |

## Relationships

- **STAGE_OVERRIDES** has exactly six entries, one per agent role; empty string for `model` or `effort` means CLI default. The `preflight-issue` and `improve` entries are not tied to a phase in the 3-phase loop.
- **`load_config()`** is pure — no subprocess; if `docker_image_name` is unset post-merge it is derived from sanitised `Path.cwd().name`; model strings stored as-is and resolved by Claude CLI at stage execution (ADR 0002).
- **Layered config merge** applies at every CLI subcommand entry (`run`, `labels`, `init`); every subcommand prints the resolved layer summary at startup (ADR 0003).
- **`PreflightCache`** is constructed once in `orchestrator.run()` outside the iteration loop and threaded into every `IterationDeps`. Its `asyncio.Lock` serialises concurrent callers — parallel `run_issue` instances all calling `get_safe_sha()` queue, second-and-later callers see a populated slot and return without re-running preflight. Iterations with no save-SHA consumer (idle without `--improve`, in-flight-only) never invoke `get_safe_sha()` and pay no preflight cost. Cache survives across iteration boundaries; HEAD only moves forward (`git pull --ff-only`), so single-slot suffices.
- The **HITL verdict** is read by `PreflightCache.get_safe_sha()` from the GitHub issue label after the preflight-issue agent completes; `ready-for-agent` produces `PreflightAFK` (caller returns it up to `run_iteration`, triggering the preflight-fix path), `ready-for-human` produces `PreflightHITL` (aborts iteration with `AbortedHITL`).
- The **planning skip** has two triggers, both routing directly to `implement_phase`: in-flight branches present, or preflight-fix AFK. **implement skip** and **review skip** are checked inside `run_issue` before any worktree is created; both triggered by **stage-done signal** — role-dir presence with cleared session content (ADR 0007).
- A **merge-time preflight skip** leaves conflict issues open; they become in-flight on the next iteration. The cache's single slot holds the broken-HEAD verdict, so the next iteration's first `get_safe_sha()` call returns the same `PreflightAFK | PreflightHITL` without re-running checks or filing a duplicate issue.
- In **sequential mode**, after each merge safe SHA is re-pinned to new HEAD; failed issue is skipped (remains `ready-for-agent`) and queue continues; Merger remains a fallback for unexpected conflicts.
- **Collision detection** uses a per-branch `asyncio.Lock` held in `implement_phase` for the full duration of each `run_issue` call.
- **`transient_worktree`** is used by `planning_phase` (plan-sandbox) — always-teardown scratch. **`managed_worktree`** is used by `merge_phase`, `run_issue` (Implementer/Reviewer, delete=False), and improve mode (delete=True). Gitdir overlay stays at the `run_issue` call site, inlined around `managed_worktree`.
- Host mounts per container: host repo → RO at `/home/agent/repo`; worktree → RW at `/home/agent/workspace`; `<host-repo>/.git` → RW at `/.pycastle-parent-git`; on Windows, gitdir overlay → RO over `/home/agent/workspace/.git`.
- The **recently-closed filter** lives entirely inside `GithubService`; iteration layer gets no new parameters or state. Scoped to `get_open_issues` only — `get_issue` is consistent and untouched. Other list-style reads (`get_open_issue_numbers`, `get_open_sub_issues`) are not currently filtered; new list callers must decide whether their use case cares about the close→list lag (ADR 0011).
- **GitHub authentication** is single-source: `GH_TOKEN` only (ADR 0004). The `gh` CLI is not a runtime dependency. Host code and agent containers use the same token. The `(owner, repo)` pair is derived from `git remote get-url origin` via `GitService` — no `gh repo view`.
- **`Deps`** does not carry `env`; credentials are extracted in `main.py`, passed directly to `AgentRunner` at construction, not accessible to any iteration-layer phase.
- Each phase module declares its own per-phase dependency protocol listing only its actual field accesses; `Deps` satisfies all structurally; tests construct minimal inline dataclasses.
- Rich markup (e.g. `[red]...[/red]`) must never be embedded in a `StatusDisplay.print` message string; colouring is expressed exclusively via the `style` parameter.
- All orchestrator-level terminal output is routed through `StatusDisplay.print()` so `rich` can coordinate it with the live panel; bare `print()` is not used while a `StatusDisplay` is active.
- During Work the container runner injects the pre-rendered prompt (built upstream by `AgentRunner._build_prompt`, ADR 0012); passes the decoded NDJSON line stream and an `on_turn` callback to `process_stream`, which raises `UsageLimitError(reset_time)` immediately on 429 (where `reset_time: datetime | None` is parsed from the Claude message and converted to local time). Setup and Pre-flight produce no console output — activity reflected only in body column of the agent status row.
- The **auto bug reporter** is *outside* the iteration layer and does not depend on `Deps`, `Logger`, or `StatusDisplay` — must function even when those have not been constructed (e.g. crash during config load). Cross-repo POST goes through `GithubService.create_issue_in(owner_repo, title, body, labels)`; the public-repo issues-write permission granted to all authenticated GitHub users is what makes the upstream POST work without any extra token. The reporter is the *only* feature that talks to a GitHub repo other than the consumer's `origin`.
- **`AgentRunner`** constructs a `DockerSession` (calling `build_volume_spec`) and a `ContainerRunner`, then orchestrates the three lifecycle phases; only caller of `build_volume_spec`. At session-build time it asks **`AccountPool`** for the active OAuth token and injects it as `CLAUDE_CODE_OAUTH_TOKEN` into `container_env`. On `UsageLimitError` it calls `pool.mark_exhausted(token, reset_time)` before re-raising.
- The **agent run kind** dispatch is reached only when the agent would otherwise be spawned: stage-done signal check in `run_issue` decides Skip first; only not-skipped roles invoke `decide_agent_run_kind`. Skip and Resume share the same role-dir-driven mechanism (ADR 0007).
- A `Fresh` run begins with **wipe-before-Fresh** then launches `claude --session-id <SESSION_UUID>` with the role's normal prompt. `Resume` launches `claude --resume <SESSION_UUID>` with continuation prompt alone (role prompt already in resumed conversation). Resume runs that crash with a non-typed exception are handled by the **non-typed Resume retry** (one re-attempt against the same session, then `FailedOutput`), not by relaunching as Fresh. Container env carries `CLAUDE_CONFIG_DIR=/home/agent/workspace/.pycastle-session/<role>/` in every case.
- Each `(role, worktree_path)` tuple gets a structurally unique **derived session UUID**, so parallel agents on different worktrees cannot collide on any Claude-internal session-id-keyed cache.
- The **broadened preservation rule** is enforced in `managed_worktree.__aexit__` for every branched call site. When `delete_branch_on_teardown=True`, `teardown_worktree` and `delete_branch` are skipped together when the predicate holds. **Worktree reuse on enter** in `managed_worktree.__aenter__` skips `_create_worktree` when path exists, branch matches, and session predicate holds. `transient_worktree` does not participate in either rule.
- On agent success the orchestrator runs **role session cleanup** *inside* the worktree context manager, after the commit (Implementer/Reviewer) or merge (Merger) and before `__aexit__`. For the issue worktree, cleanup clears session JSONL contents but leaves the role dir as the **stage-done signal** (ADR 0007); the **session role-dir-present predicate** preserves the worktree across `__aexit__`. For improve-sandbox and merge-sandbox, cleanup calls `RoleSession(...).discard()` so the role dir is removed entirely, the predicate goes False, and the sandbox is torn down. Cleanup failure logged but never raised.
- `AgentTimeoutError` is preservation-worthy at the iteration boundary: in single-agent phases the `timeout_retries` in-call retry runs (default 1), and on exhaustion the `try/except AgentTimeoutError` wrapping `run_iteration` (mirror of the **centralized usage-limit catch**) converts the exception to `AbortedTimeout(failed_role, worktree_path)`. The orchestrator continues to the next iteration without sleep or shutdown. Parallel `implement_phase` aggregates timeouts as per-issue entries in `ImplementResult.errors` instead.
- Session resume and implement-skip / review-skip are the two role-dir-driven mechanisms: skip fires when role dir present and `has_resumable_session` false; resume fires when both true. They share the directory and predicate, differ only in predicate outcome (ADR 0007).
- **Merger preservation** matches Implementer/Reviewer: merge-sandbox preserved whenever broadened predicate holds. Real merge conflicts touch ~1–2 branches and resuming partial conflict-resolution thinking is the exact savings the primitive is built for.
- Session-storage gitignoring uses `.git/info/exclude` entries appended at orchestrator startup. Worktrees share the parent `.git`, so exclusion applies in every worktree without per-worktree files. No tracked file touched, no migration required.
- **`AccountPool`** is an in-memory ordered registry of `(name, token, exhausted_until)` tuples seeded once in `main.py` from `CLAUDE_CODE_OAUTH_TOKEN` (primary) and optional `CLAUDE_CODE_OAUTH_TOKEN_SECONDARY` (secondary, preferred when present). Exposes `pick()`, `mark_exhausted(token, reset_time)` setting per-token wake-time (`reset_time + 2 min` parsed, or next-hour-boundary fallback), `has_available()` for orchestrator's pre-sleep gate, `earliest_wake_time()` for sleep duration. Constructed in `main.py`, passed to `AgentRunner.__init__` as separate injectable; never threaded through `Deps`. Failover is per-agent-run granularity. State is in-memory only — process restart re-probes secondary optimistically (ADR 0005). Forward-looking (ADR 0015): the class moves into **`ClaudeService`** as a private detail; the user-visible env-var contract and dual-token semantics are preserved byte-for-byte. Orchestrator-level failover moves up to the **stage fallback chain** (each stage's `StageOverride.fallback` carries a full `(service, model, effort)` triple).
