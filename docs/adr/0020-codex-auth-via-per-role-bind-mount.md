# Codex authentication via per-role bind-mount of `~/.codex/auth.json`

ADR 0015 introduced the `AgentService` seam and promised that the second service (Codex) would land as a separate change. This ADR is that change's architectural backbone: how `CodexService` authenticates inside per-role agent containers without an OPENAI API key. The OpenAI Codex CLI cannot route ChatGPT Plus/Pro/Team/Business/Enterprise subscription billing via `OPENAI_API_KEY` — subscription requests need a `chatgpt-account-id` header which is only populated by `AuthMode::Chatgpt` (`codex-rs/login/src/auth/manager.rs:339-343`, `codex-rs/core/src/connectors.rs:362,457`). The only path to subscription billing is the OAuth flow: `codex login` on the host writes `~/.codex/auth.json`, a mutable JSON file holding an access token, refresh token, and account id, which codex rotates on every refresh (`codex-rs/login/src/auth/manager.rs:792-930`). Pycastle therefore models codex auth as a **bind-mounted file**, not an env-var-shaped token, and pays for that by introducing per-role seeding at `pycastle init`. Per-role copies diverge after first refresh — each role becomes its own refresh-token lineage — which is fine because codex's `refresh_token_reused` error only fires when the *same* refresh token is reused, not when two independently-rotated descendants exist.

## Considered Options

- **`OPENAI_API_KEY` env var, sandcastle-style.** Rejected: cannot route subscription billing (verified at codex-rs `manager.rs:339-343`); falls through to per-token API billing from the platform.openai.com account, defeating the entire reason a user with a ChatGPT subscription would want this integration. Sandcastle (https://github.com/mattpocock/sandcastle) chose this path and tracks subscription support as open issue #191 — not a precedent.
- **`CODEX_ACCESS_TOKEN` env var (codex's AgentIdentity mode).** Rejected: this is codex's enterprise/service-account JWT path (`codex-rs/login/src/auth/manager.rs:467,480`), separate from consumer ChatGPT OAuth. Not what subscription users have.
- **Single shared bind-mount of host `~/.codex/` across all role containers.** Rejected: N containers writing to the same `auth.json` concurrently would race on refresh — the first refresh invalidates the file's refresh token from codex's POV, and the other N-1 containers would hit `refresh_token_reused` (a hard failure). Coordinating with a host-side file lock spanning the refresh window is invasive and fragile.
- **Read-only mount of just `auth.json`.** Rejected: codex needs to write the rotated token back; a read-only mount breaks refresh as soon as the access token expires.
- **Per-role bind-mount of `~/.codex/` seeded once at `pycastle init` (chosen).** Each role gets its own `auth.json` lineage at `.pycastle-session/<role>/[<namespace>/]codex/auth.json`, bind-mounted to `/home/agent/.codex/` inside the container (codex's default `$CODEX_HOME`). After first refresh in any role, the host's `~/.codex/auth.json` is effectively orphaned — a re-seed (via re-running `pycastle init` after another `codex login`) is required to add new roles. The trade: more init-time machinery in exchange for concurrent-safe refresh semantics and per-role failure isolation.
- **Lazy seeding (copy host `auth.json` into a role's state dir on first run if absent).** Considered and rejected: hidden side effect; failure mode if the host isn't logged in surfaces deep inside the dispatch loop rather than at init. Eager seeding at init makes the failure visible at setup time and fits pycastle's existing "check credentials at init, then run" pattern (`claude setup-token` + `.env` write).
- **Pycastle wrapping `codex login` as a subcommand.** Rejected: matches the precedent set by `claude setup-token` (user runs the CLI's own login command on the host; pycastle does not own the login UX). Avoids taking on responsibility for codex's interactive browser-OAuth flow which could change between codex versions.
- **Copying the entire `~/.codex/` directory at init (`auth.json` + `config.toml` + `sessions/`).** Rejected: pycastle owns the runtime contract — model, effort, sandbox, approval policy come from `pycastle.toml` and `build_command`, not inherited from the user's host `config.toml`. Avoids host-config-references-host-paths footguns (e.g. host MCP servers pointing at tools that do not exist in the container). Only `auth.json` is copied.
- **One fat default Dockerfile that always installs both claude and codex.** Rejected: codex requires Node.js runtime (`npm install -g @openai/codex`), which costs ~80–100 MB and is irrelevant for claude-only users. Two static templates — `Dockerfile.claude` and `Dockerfile.claude-codex` — are shipped in `defaults/`, picked by the init wizard's service-selection answer.

## Consequences

- **`CodexService` lives at `src/pycastle/services/codex_service.py`** implementing the `AgentService` protocol from ADR 0015. Single-account for v1 (`_exhausted_until: datetime | None` mirroring `ClaudeService`'s safety margins from ADR 0005).
- **`build_command` branches on `session_uuid`.** Fresh: `codex exec [flags] <prompt>`. Resume: `codex exec resume <thread_id> [flags] <prompt>`. The `thread_id` comes from codex's `thread.started` JSONL event on the first dispatch and is persisted via the existing `RoleSession` mechanism — diverges from claude's `--session-id` flow where pycastle generates the UUID.
- **`build_env` sets `TZ=UTC` and `CODEX_HOME=/home/agent/.codex`.** `TZ=UTC` is load-bearing: codex's `UsageLimitReachedError` Display impl (`codex-rs/protocol/src/error.rs:537-548`) renders reset timestamps in the *process's local timezone* using `%-I:%M %p` (or `Month Dth, YYYY %-I:%M %p` cross-day) — never ISO, never with an explicit zone. Pinning the container TZ to UTC makes the `_check_usage_limit` regex deterministic across host environments.
- **Per-role state-dir layout.** `state_dir_relpath(role, namespace) → .pycastle-session/<role>/[<namespace>/]codex/` matches ADR 0015's per-service worktree marker scheme. `is_resumable(state_dir)` returns True iff `state_dir/sessions/` contains at least one `rollout-*.jsonl` — `auth.json` is pre-seeded and must not count as a resume signal.
- **Sandbox + approval are pinned by pycastle.** `--sandbox danger-full-access` and `-c approval_policy=never` flags are emitted by `build_command` and not user-tunable. Pycastle's container is the isolation boundary; an inner codex sandbox would only fight the existing container layout. `--ask-for-approval` does not exist on `codex exec` anyway (`codex-rs/exec/src/cli.rs`) — approval policy must be set via `-c`.
- **`pycastle init` gains a service-selection prompt** as the first wizard question: `"Which agent services do you want to use? [claude/codex/both]"` (default `claude` for back-compat). The codex branch (a) verifies `~/.codex/auth.json` exists and exits non-zero with an actionable instruction if not (matching the `claude setup-token` precedent — pycastle does not invoke `codex login`), and (b) walks every role × namespace in the resolved config and copies the host `auth.json` into each role's state dir. Only `auth.json` is copied — not `config.toml`, not `sessions/`. Re-running init seeds newly-added roles without touching existing role directories.
- **Two static Dockerfile templates ship in `src/pycastle/defaults/`.** `Dockerfile.claude` (current behaviour, no change) and `Dockerfile.claude-codex` (adds Node.js runtime + `npm install -g @openai/codex`). The init wizard picks one based on the service-selection answer; existing skip-if-exists semantics make the Dockerfile user-owned after first init. `pycastle init --refresh` picks the template by walking the config (no re-prompt) and overwrites `pycastle/Dockerfile` per existing refresh semantics. Refresh does *not* re-seed per-role codex credentials — credentials are init's responsibility.
- **Service registry construction in `main.py` becomes lazy.** The hardcoded `{"claude": ClaudeService(...)}` is replaced by a collector that walks every stage override + top-level default to determine which service names are referenced, then instantiates exactly those. Claude-only users pay zero startup cost for codex. Unknown service names surface as fail-fast validation errors before any container starts.
- **Fail-fast `(service, effort)` validation lives at the top of `pycastle run`.** A new pass after config load walks every effective `(stage, service, effort)` pair (including each `StageOverride.fallback`'s triple), asserts the service name is known and the effort is in `service.valid_efforts()`. All violations are collected and reported in a single error before any container starts. `valid_efforts()` is added to the `AgentService` protocol: `ClaudeService` returns `{low, medium, high, xhigh, max}`; `CodexService` returns `{none, minimal, low, medium, high, xhigh}` — overlapping but genuinely different sets.
- **Wire-format parser.** `CodexService.run(lines)` consumes codex's JSONL `ThreadEvent` envelope. `item.completed type=agent_message` → `AssistantTurn`; `turn.completed.usage` → `Tokens` then return (stream terminator); `turn.failed | error` whose message matches the usage-limit regex → `UsageLimit(reset_time)`. `reasoning`, `command_execution`, `file_change`, `mcp_tool_call` items are dropped (mirrors claude's existing skip of `thinking` blocks; tool-call visibility is a separate feature). No `Result` yield — codex has no equivalent of claude's `type:"result"` event, and the existing `process_stream_from_events` fallback ("join collected turns when no Result arrives") terminates correctly on `turn.completed`.
- **Host-token invalidation is a documented gotcha.** After any role's first refresh, the host's `~/.codex/auth.json` is effectively orphaned: re-seeding new roles requires another `codex login` on the host. This is acceptable because new roles are an init-time event, not a per-run event, and `codex login` is a one-browser-click flow. Tracked as user-facing documentation work in the implementing PR.
- **No multi-account codex pool in v1.** Branch 10 of the grilling locked single-account. Codex's mutable-file auth model would make multi-account meaningfully harder than claude's static-token pool (each account needs its own host-side seed dir, no clean account-switch CLI today). Cross-service fallback via ADR 0015's `StageOverride.fallback` partially substitutes — if a user has both claude and codex configured, they already get cross-service failover at the stage level.
- **Upstream limitations tracked as separate issues.** Codex's `--json` output flattens the structured `RateLimitSnapshot` to a human-readable string and drops the typed `resets_at` (#778, to file upstream). Codex's reasoning items may swallow the agent output protocol's DONE/HITL/AFK markers if the model emits them inside reasoning rather than the final `agent_message` (#779, verify during integration testing).
- **Related cleanup.** `_KNOWN_MODELS` / `_list_models` / `ClaudeService.list_models` are dead code (no production consumer; tests-only). `CodexService` does not implement `list_models`, and the `list_models` method on the `AgentService` protocol should be removed in #780 before or after this work lands (not in parallel).
