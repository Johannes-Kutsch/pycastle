# Fixed project-local layout

Pycastle no longer exposes most project-local directory locations as configuration. `pycastle/` is the fixed project-local root, with fixed children for local config, `.env`, prompt overrides, worktrees, setup scaffold, and the optional universal Dockerfile override. `logs_dir` remains configurable because log placement is operational configuration rather than ownership of prompt/runtime defaults: local `logs_dir` is used directly, while global `logs_dir` is a parent and the effective directory is `<logs_dir>/<sanitised project name>/`. Global `config.py` and global `.env` in `pycastle home` remain part of the configuration model. `pycastle init` does not create the optional local Dockerfile override; `pycastle build` falls back to the bundled universal Dockerfile unless the user creates `pycastle/Dockerfile`.

Init refresh ownership is explicit rather than discovered by walking bundled defaults: the copied scaffold allowlist is `.gitignore` plus the cron helpers under `setup/`, while `config.py.example` is rendered separately. Files outside that allowlist, including Python cache artifacts, are not project scaffold.

This amends ADR 0003: path fields were previously allowed in local `config.py` and forbidden only globally. The cleanup is intentionally breaking for prompt/worktree/env/Dockerfile path relocation, but tolerant for stale config files: removed path keys are ignored rather than treated as unknown keys. Configurable project-local paths made ownership ambiguous, let prompt and runtime artifacts drift away from the standard project shape, and conflicted with the simpler rule that bundled defaults are runtime fallbacks while local files are targeted overrides.
