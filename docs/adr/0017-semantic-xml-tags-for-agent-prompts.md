# Semantic XML tags for agent prompts

Agent prompt files have grown to mix instructions, data, and injected standards fragments in plain markdown, making it hard to signal to the model which part is structural instruction versus payload data. Anthropic's prompt-engineering guidance recommends semantic XML tags to separate instructions from data and to give each section a distinct, named identity. We adopt a fixed four-tag vocabulary for top-level structure in all agent prompts: `<task>`, `<context>`, `<workflow>`, `<output>`. These four tags are the only top-level structural peers; no additional top-level section tags are introduced. Data sub-sections that already carry fine-grained inner tags — issue JSON (`<issue>`), comment lists (`<comments>`), open-issue snapshots (`<all-open-issues-json>`) — collapse into the `<context>` block and keep their existing inner tags unchanged. `##` sub-headings inside a tagged section remain markdown; tags are for top-level structure only, not for every heading level. Standards fragments loaded via `PromptRenderer._STANDARDS_FILES` self-wrap inside the fragment file itself using three named tags: `<design-standards>`, `<implementation-standards>`, `<output-rules>`. Each fragment's top-level `#` heading is removed; `##` sub-headings stay as markdown inside the tag. This keeps the renderer's injection logic unchanged — fragments are still injected at their placeholder positions — while giving each injected block a distinct semantic identity at the point of injection.

## Considered Options

- **Generic `<Block>` wrapper for every top-level section.** Rejected: a single generic tag loses the per-section semantic signal that makes XML tags valuable for prompt engineering. The model cannot distinguish a task block from a context block; the whole-prompt structure reads as undifferentiated payload.
- **Per-section sub-tags (e.g. `<background>`, `<constraints>`, `<examples>` inside `<context>`).** Rejected: pushes the vocabulary into unbounded growth — every new prompt would motivate new inner tags, and there is no natural stopping criterion. Sub-heading structure inside a section is already served by markdown `##` headings, which are readable and editable without expanding the tag vocabulary.
- **Separate top-level tag per standards fragment (`<implementation-standards>` as a top-level peer alongside `<task>` etc.).** Rejected: the number of top-level tags would grow with every new standards file, defeating the fixed-vocabulary goal. Standards fragments are injected into the `<context>` or `<workflow>` section at well-defined placeholder positions; keeping them inside those sections preserves the four-tag invariant.
- **No XML tags; rely on markdown headings alone.** Rejected: this is the status quo being replaced. Markdown headings are not parsed as structural separators by the model in the same way XML tags are. Anthropic's documented guidance specifically recommends XML tags for this purpose.

## Consequences

- **Fixed four-tag vocabulary.** All top-level agent prompts use exactly `<task>`, `<context>`, `<workflow>`, `<output>` as their structural section tags. No other top-level peers. Authoring a new prompt section that does not fit one of the four tags is a signal that the section needs to be merged or the vocabulary needs a deliberate revision via a new ADR.
- **Data sub-sections nest inside `<context>`.** Existing fine-grained inner tags (`<issue>`, `<comments>`, `<all-open-issues-json>`, etc.) are retained unchanged inside `<context>`. No renaming or restructuring of inner tags is required.
- **Markdown sub-headings remain markdown.** `##` and deeper headings inside a tagged section stay as markdown. Tags are not used below the top-level structural boundary.
- **Standards fragments self-wrap.** Each file under `PromptRenderer._STANDARDS_FILES` wraps its own content in one of `<design-standards>`, `<implementation-standards>`, `<output-rules>`. The file's top-level `#` heading is removed; `##` sub-headings are kept. The renderer's injection mechanism is unchanged.
- **Reduced standalone readability for standards fragments.** A standards file without its top-level heading reads less naturally in isolation. This trade-off is accepted: these files are prompt fragments, not standalone documentation. Their primary reading context is inside the rendered prompt.
