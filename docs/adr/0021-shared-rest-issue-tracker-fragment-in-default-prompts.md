# Shared REST issue-tracker fragment in default prompts

Pycastle's default prompts instructed agents to use the `gh` CLI for GitHub issue operations (`gh issue create --body-file`, `gh api repos/.../sub_issues …`, `gh repo view`, `gh issue edit --add-label`, `gh issue list --search`), with the recipe copy-pasted across six prompts: `preflight-issue.md`, `failure-report.md`, `improve/02-prd.md`, `improve/03-issues.md`, and `improve/04-no-candidate-report.md`. This contradicted ADR 0004's "PAT-only — removes the `gh` install requirement; Dockerfile install instructions drop the `gh` step": the agent container ships without `gh` accordingly, so every agent that followed the prompt hit `gh: command not found`. The slice-agent observed on 2026-05-18 against the `application-pipeline` repo failed after two `gh` retries and emitted `<promise>COMPLETE</promise>` with fabricated `<issue>351</issue> <issue>352</issue>` numbers; sibling runs improvised a `curl` fallback at runtime, inconsistently. The fix is to replace the `gh` commands in default prompts with REST calls against `https://api.github.com` using `$GH_TOKEN`, factored into a single shared include `_issue-tracker.md` rendered via a new `{{ISSUE_TRACKER}}` placeholder in the global pool. Each affected prompt drops in the placeholder; the recipe lives in one file.

## Considered Options

- **Install `gh` in the default Dockerfile.** Rejected: directly contradicts ADR 0004's "no `gh` install requirement" and reintroduces a system dependency the host code (PAT + `urllib` via `GithubService`) does not need. Leaves the six-way duplication brittleness intact — the next time prompt syntax drifts from reality, the same hallucination loop replays.
- **Shell-wrapper baked into the image (`/usr/local/bin/issue`).** Rejected as premature. The slice-agent's hallucination came from the prompt explicitly instructing `gh`; making `issue` the literal-prompt token instead of `gh` does not structurally prevent fabricated `<issue>N</issue>` emissions on non-zero exit — the 1053 agent observed exit 127 and fabricated anyway. The wrapper's stronger value (uniform error envelopes) is only meaningful once orchestrator-side issue-existence verification consumes them, which is separate work. With Codex's Dockerfile pending per ADR 0015, a new system binary also doubles the maintenance surface on day one.
- **Inline curl in each of six prompts.** Rejected: reproduces the exact six-way copy-paste brittleness that produced the original bug. The next change to authentication header, label flags, or sub-issue endpoint shape would require touching six prompts and one inevitable copy would be missed.
- **Defer to the consumer-owned `docs/agents/issue-tracker.md`.** Rejected: pycastle default prompts cannot reference a path that does not exist or carries arbitrary content per consuming repo; the doc lives outside pycastle's release boundary.
- **Shared include in pycastle defaults (chosen).** Single fragment at `defaults/prompts/_issue-tracker.md` loaded by `PromptRenderer` and exposed in the **global placeholder pool** as `{{ISSUE_TRACKER}}`. Same mechanism `coding-standards/*.md` already uses. Six prompts each replace their `gh` paragraph with the placeholder.
- **Orchestrator-side issue-existence verification of `<issue>N</issue>` emissions.** Out of scope for this ADR but identified as the actual structural fix to the hallucination class — filed as a follow-up. Without verification, the prompt rewrite makes the failure mode less likely (correct instructions, correct credentials in `$GH_TOKEN`) but does not close it; with verification, even a hallucinated `<issue>351</issue>` is caught when the API confirms no such issue exists or no such sub-issue link exists.

## Consequences

- `defaults/prompts/_issue-tracker.md` is the canonical recipe set: create-issue, view-with-comments, list-by-search, add-label / remove-label, comment, close, and link-as-sub-issue. All recipes call `https://api.github.com` with `Authorization: Bearer $GH_TOKEN`; `OWNER/REPO` is derived once from `git remote get-url origin`.
- `PromptRenderer._STANDARDS_FILES` in `src/pycastle/prompts/pipeline.py` (or a sibling loader) gains `_issue-tracker.md → ISSUE_TRACKER`. The validation pass at construction already enforces that referenced placeholders exist, so a missing file aborts the run loud.
- Six prompts are updated: `preflight-issue.md`, `failure-report.md`, `improve/02-prd.md`, `improve/03-issues.md`, `improve/04-no-candidate-report.md`. Each replaces its `gh`-specific paragraph with `{{ISSUE_TRACKER}}`. `failure-report.md` gains the include where today it just says "file an issue" with no `how`.
- Pycastle's own `docs/agents/issue-tracker.md` is rewritten to match (REST recipes, `$GH_TOKEN`). Consumer-owned copies (e.g. `application-pipeline`'s) need a one-time mirror; this is not automatic and not enforced.
- ADR 0018's prose ("Improve-mode prompts write `gh issue create --body-file` drafts to …") is superseded with respect to the `gh` reference; the drafts-in-role-session-dir decision itself is unaffected — the file is still written, just passed to `curl -d @body.json` instead of `gh --body-file`.
- ADR 0010's reference to manual `gh issue close` for orphan PRD cleanup is unchanged — that is a *user* step on the host (where `gh` may or may not be installed), not an agent operation.
- The `[improve-<short-sid>]` magic title prefix and **prompt-level dedup** glossary entries in `CONTEXT.md` cite `gh issue list --search …` as the dedup query; they update to the REST equivalent.
- ADR 0004 is honored end-to-end for the first time: `gh` is now genuinely not a runtime dependency anywhere in the stack — host code uses `GithubService` (urllib + PAT) and agent prompts use curl + `$GH_TOKEN`.
- Hallucination of `<issue>N</issue>` emissions remains structurally possible until orchestrator-side verification lands. The 2026-05-18 incident is mitigated (correct instructions, working credentials, all six prompts consistent) but not closed; a follow-up issue tracks the verification layer.
