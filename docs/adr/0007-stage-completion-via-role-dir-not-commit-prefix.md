# Stage completion signaled by role-session-dir state, not commit-message prefix

The orchestrator decides whether the Implementer or Reviewer stage of an issue is already done by inspecting the worktree's `.pycastle-session/<role>/` directory rather than scanning the branch's commit subjects for `RALPH: Implement -` / `RALPH: Review -` prefixes. The agent's `<commit_message>` tag becomes optional input to the host-side commit body — its absence triggers a synthetic fallback (`Implement #<n> - <title>`), not a failed run. The trigger for the change was issue #514, where the Implementer agent self-committed without the prefix and emitted no `<commit_message>` tag, putting the orchestrator into a 14-iteration retry loop because the prefix scan kept reporting "implementation not done" while the agent kept seeing its own prior commit and producing only a status summary. Conflating "describe the work" (the commit prefix) with "mark the stage done" (the resume idempotency token) had no recovery path once the agent broke either side of the convention.

## Considered Options

- **Status quo (commit-prefix scan + mandatory `<commit_message>` tag).** Rejected: the failure mode in #514 is structurally unrecoverable. Once the agent commits without the prefix, no further agent run can restore the prefix without the agent itself realising the convention exists; once the agent fails to emit `<commit_message>`, the orchestrator treats the run as failed even when the work is on the branch. Both signals are agent-controlled and the orchestrator has no fallback when the agent breaks them.
- **Add a separate `<role>.done` marker file alongside `<role>/`.** Rejected: introduces a second piece of per-stage state that must agree with the in-progress dir; doubles the number of file operations on the success path; and creates a new "marker present but no session content" / "session content but no marker" pair of inconsistent states that resume logic would have to reason about. Reusing `<role>/` for both signals collapses those states into one.
- **Single `stages.json` per worktree carrying `{implementer: done, reviewer: done}`.** Rejected: write contention if roles ever run in parallel within a single issue, and one corrupt JSON file wedges both stages. Per-role directories already isolate per-role state on the filesystem.
- **Drop the `<commit_message>` tag entirely; commit with a fully synthetic message.** Rejected: agent-authored commit bodies are useful for `git log` archaeology, and keeping the tag as *optional* costs one `if message else default` line over making it mandatory. The failure-mode reduction comes from making it non-load-bearing for stage detection, not from removing it.
- **Let the agent run `git commit` and have the host write the marker on clean exit.** Rejected: the agent self-committing is exactly what went wrong in #514. Treating the agent as the authority on its own commits gives up too much control — it can commit nothing, partial work, or multiple times with confusing messages, and the orchestrator has no say. Host-owned commits keep the body format stable.
- **Commit-counting (presence of any commit on the branch since safe SHA = stage done).** Rejected: structurally cannot distinguish "Reviewer ran and approved with no changes" from "Reviewer never ran" — both produce zero new commits. Counting also breaks if a future stage legitimately produces multiple commits, or if a human-pushed commit lands on the branch.
- **Backwards-compatibility shim recognising either marker presence or `RALPH:` prefix during a transition window.** Rejected: in-flight `pycastle/issue-*` branches are few and local; accepting a one-time re-run of stages on those branches is cheaper than keeping the deprecated prefix-scan code alive. A shim's existence also signals that the prefix is still meaningful, undermining the conceptual move.
- **Role-dir presence + `has_resumable_session` content check** — the chosen design. The directory `.pycastle-session/<role>/` is created when the agent first starts and survives across orchestrator iterations. After clean agent exit, the orchestrator clears the session JSONL contents but leaves the dir in place. The decision per role then becomes: dir absent → never started, run Fresh; dir present and `has_resumable_session` true → in progress, RESUME; dir present and `has_resumable_session` false → done, skip. The same predicate already governs Resume-vs-Fresh dispatch and worktree preservation, so stage-done becomes a third reading of an existing signal rather than a new one.

## Consequences

- `IMPLEMENT_COMMIT_PREFIX` / `REVIEW_COMMIT_PREFIX` constants and the `get_branch_commit_subjects` call in `iteration/implement.py` are removed. The branch's git history no longer carries an orchestrator-readable stage marker; the worktree's filesystem does.
- The skip decision in `run_issue` reads `<wt>/.pycastle-session/implementer/` and `<wt>/.pycastle-session/reviewer/`. `implement_done := implementer_dir.is_dir() and not has_resumable_session(implementer_dir)`; `review_done` likewise. The check requires the worktree to exist; on a worktree-less branch (rare, deliberate cleanup) the next run treats both stages as not-yet-started — the rerun cost accepted in ADR 0006 for the resume primitive applies identically here.
- The success-path `shutil.rmtree(<wt>/.pycastle-session/<role>)` calls in `iteration/implement.py` and `iteration/merge.py` are replaced by content-clearing: the dir survives, but its contents (the session JSONL files) are wiped so `has_resumable_session` returns false and the next iteration sees "stage done." A separate Merger marker is not needed — the Merger has no skip path today and its session-dir lifecycle is unchanged.
- `<commit_message>` becomes optional. `process_stream` for IMPLEMENTER and REVIEWER returns `CommitMessageOutput(message=...)` when the tag is present and `CommitMessageOutput(message=None)` when absent — `CommitMessageParseError` is deleted. The host commit body is `f"{title_prefix}{message}"` if `message` is set, otherwise a synthetic `f"Implement #{issue_number} - {issue_title}"` / `f"Review #{issue_number} - {issue_title}"`.
- Host-side commit timing is "on clean agent exit" — unchanged in shape but now decoupled from the tag-presence check. Any exception out of the agent run path leaves the worktree dirty for the next iteration to either complete or discard via fail-soft; no commit is produced.
- The `CommitMessageParseError` entry in the fail-soft allowlist (`agent_runner.py`) is removed. The remaining preservation-worthy exceptions are `UsageLimitError`, `AgentTimeoutError`, and `PreflightFailure`. A non-typed mid-Resume crash continues to trigger the single-shot fail-soft retry as in ADR 0006.
- The `<commit_message>` examples in `defaults/prompts/implement-prompt.md` and `defaults/prompts/review-prompt.md` are kept as a recommendation, not a requirement. The "Do not stage files or run `git commit`" instruction stays — the host-owned commit invariant is unchanged.
- The session-dir-presence signal is now load-bearing for three things: Resume-vs-Fresh dispatch (existing), worktree preservation (existing), and stage-done detection (new). The same `has_resumable_session` predicate gates all three, so they cannot diverge.
- One-shot migration cost: any branch that already has a `RALPH: Implement -` or `RALPH: Review -` commit but no `.pycastle-session/<role>/` dir on its worktree (or no worktree) will be treated as "stage not yet started" on first run after this change. Implementer re-running on already-implemented work no-ops cleanly (agent sees prior commit, exits without changes); Reviewer re-running on already-reviewed work likewise. No data loss; one-time token cost.
