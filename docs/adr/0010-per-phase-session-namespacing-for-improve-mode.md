# Per-phase session namespacing for improve mode

The improve agent's four phases share a single Claude conversation via a derived session UUID keyed on `(role, worktree_path)`. Phase 02 reuses phase 01's codebase exploration without re-scanning, and phase 03 reuses phase 02's PRD draft — but in practice the conversation grows monotonically and phase 03's vertical-slice work touches none of phase 02's drafting prose. We split phase 03 onto its own Claude session via a **session namespace** dimension — a small string folded into both `derived_session_uuid` and the per-role session signal directory path. Phases 01/02/04 share the `main` namespace; phase 03 uses the `issues` namespace and starts fresh with no prior conversation. Phase 03 receives the PRD's number, title, and body inlined into its prompt via `{{ISSUE_NUMBER}}` / `{{ISSUE_TITLE}}` / `{{ISSUE_BODY}}` / `{{ISSUE_COMMENTS}}` placeholders (mirroring `implement-prompt.md`'s `# CONTEXT` block).

The PRD issue number crosses the namespace boundary via the existing agent-output protocol: phase 02's prompt emits the parent issue as JSON-form `<issue>{"number": N, "labels": []}</issue>`, which `process_stream` surfaces as `IssueOutput(number=N, labels=[])`. `improve_phase` captures `output.number` and drives a fresh `GithubService.get_issue` fetch when assembling phase 03's prompt args. No `_prd_issue` persistence file is added — the non-persistence trades a rare crash window for plumbing simplicity (crash in that window → orphan-reset → one dead PRD on GitHub, requiring manual cleanup).

## Considered Options

- **Status quo (single shared Claude conversation).** Rejected: phase 03 pays cache-warming cost on every turn against a transcript containing the full scan and full PRD body; it actively wants to re-scan the area against fresh `CONTEXT.md` vocabulary rather than inheriting shorthand from phases 01–02.
- **`force_fresh: bool` flag on `RunRequest`.** Rejected: short-circuits `decide_agent_run_kind` contract and still requires moving tracking files out of the role session dir to avoid wipe by `wipe-before-Fresh`.
- **Expand `derived_session_uuid` to fold a phase id without partitioning the role session dir.** Rejected: a different UUID alone starts fresh, but `has_resumable_session(role_dir)` would still see the dir non-empty (from phase 01–02 signals) and pick Resume → continuation prompt to a session Claude has never seen → deterministic agent confusion. UUID partitioning and signal-dir partitioning must move together.
- **Per-phase session subdir (every phase its own namespace).** Rejected: phases 01→02 benefit from a shared transcript — phase 02 reuses phase 01's candidate identity, AFK-safety reasoning, and design-tree grilling for PRD authoring.
- **Per-group session subdir — the chosen design.** Two groups: `main` (phases 01/02/04) and `issues` (phase 03). `RunRequest` gains `session_namespace: str = ""`, threaded into both `derived_session_uuid` and `role_session_dir`. Default empty namespace preserves existing behaviour for all other roles byte-for-byte.
- **Persist the PRD issue number to disk.** Rejected: the only recoverable state it adds covers a sub-second window of orchestrator code with no I/O. The orphan-reset alternative costs nothing in normal operation.
- **Re-fetch PRD body via the agent inside phase 03.** Rejected: deterministic extra agent turn to re-derive a value the host already has; also fails if `gh` is mis-authenticated inside the container.

## Consequences

- `RunRequest` gains `session_namespace: str = ""`. `agent_runner.run` computes `role_session_dir = mount_path / ".pycastle-session" / role.value / namespace`. `derived_session_uuid(role, worktree_path, namespace="")` folds the namespace into the UUID chain only when non-empty — empty produces byte-identical UUIDs to today's two-arg call.
- `improve_phase` selects `namespace = "issues"` for `03-issues.md` and `namespace = "main"` for every other phase. Phase 04 stays in `main` because it fires only on the NO-CANDIDATE branch from phase 01 and needs access to phase 01's shortlist-rejection reasoning.
- `_phase_progress` and `_phase_in_flight` continue to live at `.pycastle-session/improve/` (role-level), sibling to `main/` and `issues/` subdirs. `shutil.rmtree(role_session_dir)` on successful cycle wipes every namespace and tracking file together. `wipe-before-Fresh` operates only on the per-namespace path, leaving sibling namespaces and tracking files intact.
- `IssueOutput(labels, number)` is now produced by the IMPROVE-role parser on JSON-form `<issue>{"number": N, "labels": [...]}</issue>` alongside `<promise>COMPLETE</promise>`. Bare-integer `<issue>N</issue>` tags (phase 03's per-sub-issue listings) continue to be ignored.
- `_ImproveDeps` gains `gh_svc: GithubService`. `GithubService` gains `get_issue(number: int) -> dict[str, str]`. `improve_phase` calls `gh_svc.get_issue(prd_number)` and `gh_svc.get_issue_comments(prd_number)` once per phase 03 entry.
- Phase 02's prompt no longer carries the `## Dedup check` section (deduplication had no remaining caller after orphan-reset semantics). Phase 03's prompt gains a `# CONTEXT` block mirroring `implement-prompt.md`.
- Orphan-reset: if `last_id == "02-prd"` AND `in_flight_id != "03-issues"`, the in-memory PRD number is lost. The orchestrator unlinks `_phase_progress` and restarts from phase 01; the orphan PRD requires manual `gh issue close`.
- ADR 0006's closing consequence — "improve mode builds multi-prompt-within-one-run on top of this primitive without modifying the resume infrastructure" — is superseded by this ADR. The resume primitive is unchanged; the namespace dimension is additive.
