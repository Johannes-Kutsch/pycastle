# Persist agent draft files inside the role session dir

The improve-mode prompts (`02-prd.md`, `03-issues.md`, `04-no-candidate-report.md`) instruct the agent to write each issue body to a file and then call `gh issue create --body-file <path>`. Until now the prompts left the path unspecified and agents in practice chose `/tmp/sliceN.md`, which is container-scoped: on `UsageLimitError` the container is torn down (ADR 0008 unwinds to the iteration boundary; ADR 0005 requires a fresh container on account failover; ADR 0006 designs resume around ephemeral containers plus persisted on-disk session JSONL), and the draft file is destroyed. On Claude `--resume`, the conversation history still shows the `Write(/tmp/sliceN.md, …)` tool call succeeded, so the agent may re-issue `gh issue create --body-file /tmp/sliceN.md` against a path that no longer exists, or correctly re-derive the body — behaviour is non-deterministic and a drafted-but-unfiled slice can be silently lost. We change the prompt convention to write drafts to `<worktree>/.pycastle-session/improve/drafts/`, a sibling of `_phase_progress` and `_phase_in_flight` inside the **role session dir**. That path is preserved across container teardown by the existing broadened preservation rule, and removed by **role session cleanup** on terminal-state success. The fix is prompt-only — no orchestrator code changes.

## Considered Options

- **Heredoc body inline (`gh issue create --body "$(cat <<'EOF' …EOF)"`).** Rejected: large PRD bodies break shell quoting (already called out in phase 02's existing prompt text) and the convenience of re-reading the drafted body across turns is lost. Single-shot inline bodies also remove a useful debugging artefact on the worktree.
- **Per-slice promise markers (`<filed>N</filed>`) plus orchestrator parser.** Rejected: a real contract change — phase 03 today is one prompt → one `<promise>COMPLETE</promise>` terminal token, and per-slice parsing would materially extend the orchestrator's surface area and its coupling to phase 03's mid-stream output. It also does not solve the headline failure mode addressed here: it would make already-filed slices skippable on resume, but the lost slice was never filed in the first place, so no marker exists for the orchestrator to read. This option is a Risk-2 (dedup) fix masquerading as a Risk-1 (lost-draft) fix.
- **Keep `/tmp`; skip container teardown on `UsageLimitError`.** Rejected: fights ADR 0005 (account failover needs a fresh container to swap creds), ADR 0006 (resume is designed around ephemeral containers reading persisted worktree state), and ADR 0008 (centralised catch unwinds to the iteration boundary). Reset windows are typically hours; pinning a container idle that long is wasteful regardless.

## Consequences

- **Risk-2 (dedup) explicitly not addressed.** The prompt-level dedup query at the top of phases 03 and 04 still runs once per phase, not per filing. Duplicate filings remain a documented possible-but-rare outcome (see `prompt-level dedup` in `CONTEXT.md`). Scope is bounded to the more-costly failure mode (lost slice corrupts PRD decomposition silently; duplicate is loud and recoverable).
- **`drafts/` participates in role-session-dir invariants.** Any future refactor of the role-session-dir layout, the broadened preservation rule, or `role session cleanup` must preserve `<worktree>/.pycastle-session/improve/drafts/` across container teardown and remove it on terminal success — same contract as `_phase_progress` and `_phase_in_flight`.
- **No new orchestrator term required beyond the glossary entry.** The drafts dir is not parsed, listed, or consulted by the orchestrator. It is purely a prompt-side persistence convention that piggybacks on existing worktree-preservation machinery.
