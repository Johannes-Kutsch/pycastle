# Universal agent image and stage priority chains

Pycastle keeps the nested `StageOverride(..., fallback=StageOverride(...))` public shape, but interprets each chain as an ordered priority list: resolution skips services that are not locally configured, uses the first configured and available service, sleeps only when configured candidates are exhausted, and fails when a stage has no configured candidate. Model and effort remain validated against every known service named in the chain, even if that service is not configured on the current machine, so enabling a service later cannot reveal stale invalid defaults.

This amends the stricter per-stage fallback semantics of the earlier pluggable-service model and supersedes the earlier per-service image split. `pycastle build` now builds one universal agent image containing all supported service CLIs, and runtime image selection no longer depends on `StageOverride.service`. Service/model/effort choice is owned by stage chains at agent dispatch time; preflight checks use the universal image directly and are not a service-dispatched agent role.

The concrete bundled default chains (specific service/model/effort per stage) are an easily-revised config surface and live in CONTEXT.md's **default stage priority chains** entry. OpenCode participates in bundled defaults only as the first Planner candidate; other stages keep Codex/Claude chains until OpenCode proves suitable. OpenCode uses bare OpenCode Go model ids in config, and `OpenCodeService` maps each to the `opencode-go/<id>` CLI ref. Machines without `OPENCODE_GO_API_KEY` skip the OpenCode node and continue through the configured Codex/Claude fallbacks.
