# Universal agent image and stage priority chains

Pycastle keeps the nested `StageOverride(..., fallback=StageOverride(...))` public shape, but interprets each chain as an ordered priority list: resolution skips services that are not locally configured, uses the first configured and available service, sleeps only when configured candidates are exhausted, and fails when a stage has no configured candidate. Model and effort remain validated against every known service named in the chain, even if that service is not configured on the current machine, so enabling a service later cannot reveal stale invalid defaults.

This amends ADR 0015's stricter fallback semantics and supersedes the earlier per-service image split. `pycastle build` now builds one universal agent image containing all supported service CLIs, and runtime image selection no longer depends on `StageOverride.service`. Service/model/effort choice is owned by stage chains at agent dispatch time; preflight checks use the universal image directly and are not a service-dispatched agent role.

The bundled default stage priority chains are: plan `opencode/kimi-k2.6/medium -> codex/gpt-5.4-mini/low -> claude/haiku/low`; implement `codex/gpt-5.4/medium -> claude/sonnet/medium`; review `claude/sonnet/medium -> codex/gpt-5.4/medium`; merge `codex/gpt-5.5/medium -> claude/opus/high`; preflight-issue `codex/gpt-5.5/medium -> claude/opus/high`; improve `codex/gpt-5.5/high -> claude/opus/high`.

OpenCode participates in bundled defaults only as the first Planner candidate. `kimi-k2.6` is chosen as the stronger unattended-loop planning default; `deepseek-v4-flash` remains the lower-cost alternative when a user chooses to override the stage. These are bare OpenCode Go model ids in pycastle config; `OpenCodeService` injects an inline OpenCode config for the `opencode-go` provider and passes `opencode-go/<id>` to `opencode run --model`, matching the OpenCode CLI's provider/model contract. The implementation, review, merge, preflight-issue, and improve stages keep their existing Codex/Claude chains until OpenCode proves suitable across those role contracts. Machines without `OPENCODE_GO_API_KEY` skip the OpenCode node and continue through the configured Codex/Claude plan fallbacks.
