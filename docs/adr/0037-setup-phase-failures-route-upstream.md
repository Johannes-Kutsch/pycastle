# Setup phase failures route upstream through auto_file_issue

Setup phase failures are handled operator-actionable failures, not preflight check failures and not unhandled crashes. When container startup, git identity propagation, gitdir/volume wiring, or dependency installation fails before role work or preflight checks begin, `run_iteration()` returns a setup-specific abort outcome: pycastle prints the local error, exits non-zero, and routes an upstream pycastle report through `auto_file_issue`, using API filing when `auto_file_bugs` is enabled and the prefilled URL fallback otherwise.

This is deliberately separate from two nearby paths. `PREFLIGHT_CHECKS` failures still go through the preflight-issue agent into the consuming project's tracker, because they are diagnosable check failures after the container baseline exists. `OperatorActionableGitError` still files on the consuming project's `origin`, because remote git reachability is an operator/environment issue rather than a pycastle defect. Setup phase failures sit between those categories: they are locally actionable enough to abort clearly, but upstream-relevant enough to use the pycastle bug-report path instead of the consuming repo.

The same upstream routing applies when a preflight command fails because a check tool declared in the consuming project's Python dependency metadata (`pyproject.toml` or `requirements.txt`) is unavailable at command time. Those consuming-project development tools are still installed by the Setup phase, not baked into the Docker image, so an unavailable declared tool means the Setup/toolchain contract failed even if dependency installation appeared to complete. Routing through the normal preflight-issue agent would misclassify a pycastle-owned environment failure as a source-quality failure in the consuming project.

This remains distinct from an ordinary preflight check failure. If the declared command starts and then reports lint, test, or other project-level failures, that stays a consuming-project preflight result and follows the existing preflight-issue path. Only the "tool was declared, pycastle should have made it available, but the command is missing" case is reclassified as an upstream Setup/toolchain failure.

Non-Python package managers are out of scope for this classification until pycastle owns a Setup-phase contract for installing and verifying those toolchains.

Observable consequence: a missing declared tool such as `ruff` must not spawn the preflight-issue agent or file a consuming-project `ready-for-human` issue. Pycastle should abort through the upstream setup/toolchain failure path and name both the missing tool and the dependency metadata source that made the tool expected.
