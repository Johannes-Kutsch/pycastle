# Shared `gh` issue-tracker fragment in default prompts

Pycastle's default prompts instructed agents to use the `gh` CLI for GitHub issue operations (`gh issue create --body-file`, `gh api repos/.../sub_issues …`, `gh repo view`, `gh issue edit --add-label`, `gh issue list --search`), with the recipe copy-pasted across five prompts: `preflight-issue.md`, `failure-report.md`, `improve/02-prd.md`, `improve/03-issues.md`, and `improve/04-no-candidate-report.md`. The agent container shipped without `gh` per ADR 0004's "PAT-only — removes the `gh` install requirement" clause, so every agent that followed the prompt hit `gh: command not found`. The slice-agent observed on 2026-05-18 against the `application-pipeline` repo failed after two `gh` retries and emitted `<promise>COMPLETE</promise>` with fabricated `<issue>351</issue> <issue>352</issue>` numbers; sibling runs improvised a `curl` fallback at runtime, inconsistently. The fix is to factor the recipe into a single shared include `_issue-tracker.md` rendered via a new `{{ISSUE_TRACKER}}` placeholder in the global pool, and to install `gh` in both agent Dockerfiles so the placeholder is unconditionally executable. Each affected prompt drops in the placeholder; the recipe lives in one file.

## Considered Options

- **Inline `gh` in each of five prompts.** Rejected: reproduces the six-way copy-paste brittleness that produced the original bug. The next change to label flags or sub-issue endpoint shape would require touching five prompts and one inevitable copy would be missed.
- **Shared include with REST + curl recipes (no `gh` install).** Rejected. The shared-fragment mechanism is the load-bearing improvement and is preserved here, but the tool choice flips from curl-against-`https://api.github.com` to `gh`. Curl recipes are markedly more expensive in tool-call tokens (auth header + `OWNER/REPO` derivation + JSON body file + jq post-processing per call), and agents reach for `gh` from training-data familiarity — instructing them in curl fights the prior. The original rejection of `gh` rested on ADR 0004's "no `gh` install requirement" sub-claim; that sub-claim does not survive the agent-prompt context (see below).
- **Shell-wrapper baked into the image (`/usr/local/bin/issue`).** Rejected as premature. The slice-agent's hallucination came from the prompt explicitly instructing `gh`; renaming the literal-prompt token does not structurally prevent fabricated `<issue>N</issue>` emissions on non-zero exit. The wrapper's stronger value (uniform error envelopes consumable by orchestrator-side verification) is only meaningful once the verification layer exists, which is separate work.
- **Defer to the consumer-owned `docs/agents/issue-tracker.md`.** Rejected: pycastle default prompts cannot reference a path that does not exist or carries arbitrary content per consuming repo; the doc lives outside pycastle's release boundary.
- **Shared include with `gh` recipes + `gh` installed in both agent Dockerfiles (chosen).** Single fragment at `defaults/prompts/_issue-tracker.md` loaded by `PromptRenderer` and exposed in the **global placeholder pool** as `{{ISSUE_TRACKER}}`. Same mechanism `coding-standards/*.md` already uses. Five prompts each replace their `gh` paragraph with the placeholder. Both `src/pycastle/defaults/Dockerfile.claude` and `src/pycastle/defaults/Dockerfile.claude-codex` install `gh` via apt — installing in only one would make `{{ISSUE_TRACKER}}` silently container-conditional and replay the 2026-05-18 incident shape in the other container. Authentication is via `$GH_TOKEN` (already provided by ADR 0004); repo derivation is `gh`'s cwd-origin autodetect.
- **Orchestrator-side issue-existence verification of `<issue>N</issue>` emissions.** Out of scope for this ADR but identified as the actual structural fix to the hallucination class — filed as a follow-up. Without verification, the prompt rewrite makes the failure mode less likely (correct instructions, working credentials, all five prompts consistent) but does not close it.

## Consequences

- `defaults/prompts/_issue-tracker.md` is the canonical recipe set: create-issue (`gh issue create --title --body-file`), view-with-comments (`gh issue view --comments`), list-by-search (`gh issue list --search`), add-label / remove-label (`gh issue edit --add-label / --remove-label`), comment (`gh issue comment`), close (`gh issue close`), link-as-sub-issue (`gh api repos/OWNER/REPO/issues/N/sub_issues --method POST --field sub_issue_id=M`). The sub-issue verb is the one recipe that escapes to `gh api` because the sub-issues endpoint has no native `gh` verb. All recipes authenticate via `$GH_TOKEN`; `OWNER/REPO` is autodetected by `gh` from the cwd's origin.
- `PromptRenderer._STANDARDS_FILES` in `src/pycastle/prompts/pipeline.py` carries `_issue-tracker.md → ISSUE_TRACKER`. The validation pass at construction enforces that referenced placeholders exist, so a missing file aborts the run loud.
- Five prompts are updated: `preflight-issue.md`, `failure-report.md`, `improve/02-prd.md`, `improve/03-issues.md`, `improve/04-no-candidate-report.md`. Each replaces its `gh`-specific paragraph with `{{ISSUE_TRACKER}}`. `failure-report.md` gains the include where today it just says "file an issue" with no `how`.
- Both agent Dockerfiles (`Dockerfile.claude`, `Dockerfile.claude-codex`) install `gh` via apt. ADR 0015's pluggable-agent-services contract requires every container to honor the same prompt-level commands; installing `gh` in both keeps that contract intact.
- Pycastle's own `docs/agents/issue-tracker.md` is rewritten to match (`gh` recipes referencing `$GH_TOKEN`). Consumer-owned copies (e.g. `application-pipeline`'s) need a one-time mirror; this is not automatic and not enforced.
- ADR 0018's prose ("Improve-mode prompts write `gh issue create --body-file` drafts to …") stands as-is — the `gh issue create --body-file` reference is once again accurate.
- ADR 0010's reference to manual `gh issue close` for orphan PRD cleanup is unchanged — a *user* step on the host (where `gh` may or may not be installed), not an agent operation.
- The `[improve-<short-sid>]` magic title prefix and **prompt-level dedup** glossary entries in `CONTEXT.md` cite `gh issue list --search …` as the dedup query.
- ADR 0004's PAT-as-sole-credential decision stands; its "no `gh` install requirement" sub-claim does not survive this ADR. The redirect rests on the agent-prompt context — host code (PAT + `urllib` via `GithubService`) genuinely does not need `gh`, but agent prompts do, and the token-cost gap between `gh` and curl recipes is non-trivial.
- Hallucination of `<issue>N</issue>` emissions remains structurally possible until orchestrator-side verification lands. The 2026-05-18 incident is mitigated (correct instructions, working credentials, all five prompts consistent) but not closed; a follow-up issue tracks the verification layer.
