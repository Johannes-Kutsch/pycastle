# Pi Setup Runbook

Brings a fresh Pi to the point where the cron wrapper (`pycastle/setup/cron.sh`) takes over a consuming project.
Target completion time: under one hour for an attentive operator with no prior knowledge of this project.

For an overview of what pycastle does and how the pipeline works, read the project README first.

---

## 1. Hardware + OS prerequisites

1. Confirm hardware: **Raspberry Pi with ≥ 8 GB RAM** (Pi 5 8 GB recommended). Pycastle dispatches agents inside Docker containers that run the consuming project's `[dev]` toolchain (ruff, mypy, pytest, the Claude CLI, etc.); smaller boards thrash.

2. Flash **Raspberry Pi OS Lite, 64-bit, Bookworm (Debian 12)**. Use Raspberry Pi Imager ≥ 1.8.
   - Enable SSH in Imager's advanced options and set a username (e.g. `pi`).

3. Boot and confirm SSH access from your laptop:
   ```bash
   ssh pi@<pi-ip-address>
   ```
   Expected: shell prompt, no errors.

4. Confirm the Pi can reach GitHub and PyPI:
   ```bash
   ping -c 3 github.com
   ping -c 3 pypi.org
   ```
   Expected: 3 replies each, 0% packet loss.

---

## 2. System packages

5. Update package lists and install required system packages:
   ```bash
   sudo apt-get update
   sudo apt-get install -y git python3 python3-venv curl cron
   ```
   `flock` is part of the `util-linux` base package and is already installed; it is not a separate apt package.

6. Verify versions:
   ```bash
   git --version
   python3 --version
   flock --version
   systemctl is-active cron
   ```
   Expected: git ≥ 2.39, python3 ≥ 3.11, flock present, cron `active`.

---

## 3. Docker

Pycastle runs every agent inside an ephemeral Docker container built from `pycastle/Dockerfile`. The host never executes agent code directly.

7. Install Docker via the official convenience script:
   ```bash
   curl -fsSL https://get.docker.com | sh
   sudo usermod -aG docker "$USER"
   ```
   Log out and back in (or `exec su - $USER`) so the new group membership takes effect.

8. Confirm Docker is running and reachable without sudo:
   ```bash
   systemctl is-active docker
   docker run --rm hello-world
   ```
   Expected: `active`, then `Hello from Docker!` banner.

---

## 4. SSH key for GitHub

9. Generate an SSH key on the Pi:
   ```bash
   ssh-keygen -t ed25519 -C "pycastle-pi@$(hostname)" -f ~/.ssh/id_ed25519 -N ""
   ```
   Expected: keypair created at `~/.ssh/id_ed25519{,.pub}`.

10. Print the public key and add it to GitHub:
    ```bash
    cat ~/.ssh/id_ed25519.pub
    ```
    Paste the line into <https://github.com/settings/keys> as a new SSH key.

11. Verify auth:
    ```bash
    ssh -T git@github.com
    ```
    Expected: `Hi <username>! You've successfully authenticated...` (it'll prompt to trust the host key on first connect — accept).

---

## 5. Tokens

Pycastle needs a GitHub PAT and a Claude OAuth token. Have these ready before running `pycastle init`:

- **`GH_TOKEN`**: a GitHub PAT (classic or fine-grained) with `repo` scope for the consuming project. Generate at <https://github.com/settings/tokens>.
- **`CLAUDE_CODE_OAUTH_TOKEN`**: generated by running `claude setup-token` on a machine where the Claude CLI is already authenticated. Copy the output line.

---

## 6. Consuming project setup

12. Clone the consuming project via SSH:
    ```bash
    git clone git@github.com:<owner>/<repo>.git
    cd <repo>
    ```
    Expected: clone completes; you're inside the repo root.

13. Create the host venv:
    ```bash
    python3 -m venv .venv
    ```
    Expected: `.venv/` directory exists with `bin/python` inside.

14. Install pycastle and the consuming project (same fallback chain `cron.sh` uses on every tick):
    ```bash
    .venv/bin/python -m pip install --upgrade pip
    .venv/bin/python -m pip install --upgrade pycastle
    .venv/bin/python -m pip install -e ".[dev]" || .venv/bin/python -m pip install -r requirements.txt
    ```
    Expected: pycastle installed, consuming project installed editable (or via `requirements.txt` fallback).

15. Confirm the CLI is reachable:
    ```bash
    .venv/bin/pycastle --version
    ```
    Expected: version string printed.

---

## 7. Pycastle scaffold

16. Scaffold the `pycastle/` directory in the consuming project. Choose local scope when prompted (global scope is for sharing config across many repos on one host):
    ```bash
    .venv/bin/pycastle init
    ```
    Paste your `GH_TOKEN` and `CLAUDE_CODE_OAUTH_TOKEN` when prompted.
    Expected: `pycastle/` directory created containing `Dockerfile`, `config.py`, `.env`, `setup/cron.sh`, `setup/cron-install.sh`, `setup/cron-uninstall.sh`, `setup/pi-setup.md`, and `prompts/`.

17. Build the agent image:
    ```bash
    .venv/bin/pycastle build
    ```
    Expected: Docker build completes; `docker images` lists the new tag.

---

## 8. Cron entry

18. Install the daily cron entry. The schedule is hardcoded to `0 1 * * *` (01:00 daily); the line is tagged `# pycastle:<absolute-repo-path>` so multiple repos coexist:
    ```bash
    bash pycastle/setup/cron-install.sh
    ```
    Expected: `crontab -l` shows a new line with the repo's marker.

19. Confirm the entry:
    ```bash
    crontab -l | grep pycastle
    ```
    Expected: the installed line is present.

All repos installed via `cron-install.sh` fire at 01:00 and serialize through a global flock at `$PYCASTLE_HOME/.cron.lock` (up to 6 hours wait). No extra setup needed for additional repos — repeat sections 6–8 per repo.

---

## 9. Optional: enable improve mode

For repos you're actively developing, edit `pycastle/config.py`:

```python
improve_mode = "until_sleep"  # or "endless"
improve_max = 1               # cap improve dispatches per run
```

The daily cron tick will then dispatch the improve agent automatically when no issues are ready. The `--improve` CLI flag still works for ad-hoc manual runs and overrides this default.

---

## 10. Smoke test

20. Run the wrapper manually once to verify the full pipeline before waiting for cron:
    ```bash
    bash pycastle/setup/cron.sh
    ```
    Expected: pip upgrade ×2 completes, project reinstalled, `pycastle init --refresh` and `pycastle build` complete, `pycastle run` starts and exits cleanly (immediately if the issue tracker has nothing ready).

---

## Detaching a repo

```bash
bash pycastle/setup/cron-uninstall.sh
```

Removes only the line bearing this repo's marker. No-op if no matching line exists.

---

## Logging

`cron.sh` writes nothing to disk itself — handle output at the crontab level. To capture output, run `crontab -e` and append `>> /path/to/log 2>&1` to the installed line. The header comment in `pycastle/setup/cron.sh` shows the canonical snippet.
