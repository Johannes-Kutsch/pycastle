# Cron Setup Runbook

Brings a fresh Unix host (Linux or macOS) to a working unattended pycastle deployment.
Target completion time: under one hour for an attentive operator with no prior knowledge of this project.

For an overview of what pycastle does and how the pipeline works, read the project README first.

**Windows users:** run this runbook inside WSL (Windows Subsystem for Linux).

---

## 1. Prerequisites

Install the following via your system package manager before continuing:

- `git` ≥ 2.39
- Python ≥ 3.11
- Docker (daemon running)
- `cron` (`crond` on some distributions)

`flock` ships with `util-linux` on Linux and is available on macOS via Homebrew. Verify the tools are available:

```bash
git --version
python3 --version
flock --version
docker run --rm hello-world
```

Expected: version strings printed, then `Hello from Docker!` banner.

---

## 2. SSH key for GitHub

1. Generate an SSH key on the host:
   ```bash
   ssh-keygen -t ed25519 -C "pycastle@$(hostname)" -f ~/.ssh/id_ed25519 -N ""
   ```
   Expected: keypair created at `~/.ssh/id_ed25519{,.pub}`.

2. Print the public key and add it to GitHub:
   ```bash
   cat ~/.ssh/id_ed25519.pub
   ```
   Paste the line into <https://github.com/settings/keys> as a new SSH key.

3. Verify auth:
   ```bash
   ssh -T git@github.com
   ```
   Expected: `Hi <username>! You've successfully authenticated...` (it'll prompt to trust the host key on first connect — accept).

---

## 3. Tokens

Pycastle needs a GitHub PAT and a Claude OAuth token. Have these ready before running `pycastle init`:

- **`GH_TOKEN`**: a GitHub PAT (classic or fine-grained) with `repo` scope for the consuming project. Generate at <https://github.com/settings/tokens>.
- **`CLAUDE_CODE_OAUTH_TOKEN`**: generated by running `claude setup-token` on a machine where the Claude CLI is already authenticated. Copy the output line.

---

## 4. Consuming project setup

4. Clone the consuming project via SSH:
   ```bash
   git clone git@github.com:<owner>/<repo>.git
   cd <repo>
   ```
   Expected: clone completes; you're inside the repo root.

5. Create the host venv:
   ```bash
   python3 -m venv .venv
   ```
   Expected: `.venv/` directory exists with `bin/python` inside.

6. Install pycastle and the consuming project (same fallback chain `cron.sh` uses on every tick):
   ```bash
   .venv/bin/python -m pip install --upgrade pip
   .venv/bin/python -m pip install --upgrade pycastle
   .venv/bin/python -m pip install -e ".[dev]" || .venv/bin/python -m pip install -r requirements.txt
   ```
   Expected: pycastle installed, consuming project installed editable (or via `requirements.txt` fallback).

7. Confirm the CLI is reachable:
   ```bash
   .venv/bin/pycastle --version
   ```
   Expected: version string printed.

---

## 5. Pycastle scaffold

8. Scaffold the `pycastle/` directory in the consuming project. Choose local scope when prompted (global scope is for sharing config across many repos on one host):
   ```bash
   .venv/bin/pycastle init
   ```
   Paste your `GH_TOKEN` and `CLAUDE_CODE_OAUTH_TOKEN` when prompted.
   Expected: `pycastle/` directory created containing `Dockerfile`, `config.py`, `.env`, `setup/cron.sh`, `setup/cron-install.sh`, `setup/cron-uninstall.sh`, and `prompts/`.

9. Build the agent image:
   ```bash
   .venv/bin/pycastle build
   ```
   Expected: Docker build completes; `docker images` lists the new tag.

---

## 6. Cron entry

The cron helpers scaffolded into `pycastle/setup/` manage unattended operation:

- **`setup/cron.sh`** — bootstrap-and-run wrapper. Acquires a global flock at `$PYCASTLE_HOME/.cron.lock` (6-hour timeout) so multiple repos on the same host serialize cleanly, asserts `.venv/` exists, upgrades pycastle, reinstalls the consuming project, runs `pycastle init --refresh`, rebuilds the Docker image, invokes `pycastle run`, then trims `cron.log` to the last 10000 lines so it cannot grow unbounded.
- **`setup/cron-install.sh`** — idempotently installs a daily entry (`0 1 * * *`) into your user crontab, tagged `# pycastle:<absolute-repo-path>` so multiple repos coexist. The crontab line redirects stdout+stderr into `<logs_dir>/cron.log` (resolved from the project's `pycastle/config.py` at install time), so cron output lands alongside the per-agent logs and is captured from second 0 — including bootstrap failures like a missing `.venv/` or pip errors.
- **`setup/cron-uninstall.sh`** — removes only the line bearing this repo's marker.

10. Install the daily cron entry:
    ```bash
    bash pycastle/setup/cron-install.sh
    ```
    Expected: `crontab -l` shows a new line with the repo's marker.

11. Confirm the entry:
    ```bash
    crontab -l | grep pycastle
    ```
    Expected: the installed line is present.

All repos installed via `cron-install.sh` fire at 01:00 and serialize through a global flock at `$PYCASTLE_HOME/.cron.lock` (up to 6 hours wait). No extra setup needed for additional repos — repeat sections 4–6 per repo.

---

## 7. Smoke test

12. Run the wrapper manually once to verify the full pipeline before waiting for cron:
    ```bash
    bash pycastle/setup/cron.sh
    ```
    Expected: pip upgrade ×2 completes, project reinstalled, `pycastle init --refresh` and `pycastle build` complete, `pycastle run` starts and exits cleanly (immediately if the issue tracker has nothing ready).

---

## Detaching a repo

```bash
bash pycastle/setup/cron-uninstall.sh
```

Removes only the line bearing this repo's marker. No-op if no matching line exists.

---

## Optional: enable improve mode

For repos you're actively developing, edit `pycastle/config.py`:

```python
improve_mode = "until_sleep"  # or "endless"
improve_max = 1               # cap improve dispatches per run
```

The daily cron tick will then dispatch the improve agent automatically when no issues are ready. The `--improve` CLI flag still works for ad-hoc manual runs and overrides this default.

## Upgrading pycastle on a deployed host

Run these steps inside each consuming project on the host:

1. **Enter the project and activate its venv**

   ```bash
   cd <project-dir>
   source .venv/bin/activate
   ```

2. **Update pycastle**

   ```bash
   pip install --upgrade pycastle
   # or pin: pip install --upgrade 'pycastle==<version>'
   ```

3. **Refresh the bundled defaults**

   ```bash
   pycastle init --refresh
   ```

   This re-copies the bundled `setup/` scripts, prompts, and Dockerfile, leaving your `config.py` and `.env` untouched.

4. **Remove the existing cronjob**

   The uninstall script keys off a marker derived from the project's absolute path, so it only touches that project's line:

   ```bash
   bash pycastle/setup/cron-uninstall.sh
   ```

5. **Install the new cronjob**

   ```bash
   bash pycastle/setup/cron-install.sh
   ```

6. **Verify**

   ```bash
   crontab -l | grep pycastle
   ```

   You should see one line per project, each ending with `# pycastle:<absolute-project-path>`.
