# Prompt rendering lives in `AgentRunner`, not `ContainerRunner`

`ContainerRunner.work()` previously accepted nine parameters spanning two concerns: prompt rendering (template, scope_args, renderer, `send_role_prompt_on_resume`) and execution context (role, run_kind, session_uuid, `is_failsoft_recovery`). Inside `work()`, a three-way branch over `(run_kind, send_role_prompt_on_resume, is_failsoft_recovery)` decided which template to render and whether to prepend the continuation prompt — meaning the runner had to understand improve-mode session namespace semantics (ADR 0010) to decide what to write to `/tmp/.pycastle_prompt`. We move all prompt-rendering responsibility upstream into `AgentRunner.run()`: `AgentRunner` owns a private `_build_prompt()` helper that resolves the **prompt-shape contract** (CONTEXT.md) into a single string per attempt, and `ContainerRunner.work()` now accepts `(prompt: str, *, role, run_kind, session_uuid)` — three keyword-only arguments, no struct, no renderer dependency, no flag for fail-soft.

## Considered Options

- **Status quo (nine parameters, rendering inside `ContainerRunner`).** Rejected: the cost is the coupling we are trying to remove. The runner cannot be tested without constructing a `PromptRenderer`, and prompt-shape policy lives behind a Docker substrate it does not need.
- **Two cohesive structs (`PromptSpec` + `WorkContext`) passed to a still-renders-internally `work()`.** Rejected: this is the smaller diff but preserves the runner→ADR-0010 coupling and the three-way branch in the place that's hardest to test. The naming question becomes load-bearing for no architectural gain.
- **One composite struct absorbing all nine fields.** Rejected: it groups together two genuinely separate concerns and makes the rendering test surface no easier to isolate.
- **Extend `RunRequest` to absorb the grouping.** Rejected: `RunRequest` is the phase→`AgentRunner` contract, not the `AgentRunner`→`ContainerRunner` contract. Reusing it for the inner seam couples the inner refactor to every phase caller and conflicts with the out-of-scope rule in the originating issue.
- **Render inside `AgentRunner.run()` (chosen).** `AgentRunner.run()` already owns `run_kind` derivation (`decide_agent_run_kind` from session-dir state) and `is_failsoft_recovery` (a local that flips inside the retry loop's except handler). Rendering is a pure function of those locals plus `RunRequest` fields, so co-locating the render with the state that drives it removes the leakage without pushing policy further up.
- **Render in the phase (one level above `AgentRunner`).** Rejected: phases cannot know in advance whether fail-soft will trigger, and would have to either pre-render both prompt variants or expose a re-render callback. Both options leak runner mechanics into every phase, which is the opposite of the decoupling the change targets.
- **Introduce a `WorkContext` struct (role, run_kind, session_uuid).** Rejected in favour of three keyword-only kwargs on `work()`. Three-field structs for fields that are always passed together can pay off, but the issue's user-stories did not surface a future-extension pressure on the execution side strong enough to justify a new named type today; if a fourth execution flag appears, the struct can be added then.

## Consequences

- `ContainerRunner.work()` signature becomes `work(self, prompt: str, *, role: AgentRole, run_kind: RunKind = RunKind.FRESH, session_uuid: str | None = None) -> AgentOutput`. The `template`, `scope_args`, `renderer`, `send_role_prompt_on_resume`, and `is_failsoft_recovery` parameters are removed.
- `AgentRunner` gains a private `_build_prompt(template, scope_args, container_exec, *, run_kind, send_role_prompt_on_resume, is_failsoft_recovery) -> str` method holding the three-way branch previously in `container_runner.py:113-125`. It is called once per attempt inside the retry loop, because both `run_kind` (on fail-soft fallback) and `is_failsoft_recovery` flip between attempts.
- `is_failsoft_recovery` no longer crosses the `AgentRunner`→`ContainerRunner` seam. It remains a local in `AgentRunner.run()` and an input to `_build_prompt`. CONTEXT.md's `fail-soft fresh fallback` definition continues to describe the flag's role; only its location moves.
- `RunRequest` shape is unchanged for phase callers — `template`, `scope_args`, and `send_role_prompt_on_resume` continue to be passed in by phases as today. The prompt-shape contract still routes through `send_role_prompt_on_resume`, but resolution now happens in `_build_prompt`, not `work()`.
- `ContainerRunner` no longer holds a `PromptRenderer` reference at all. Tests for `ContainerRunner` no longer require a renderer fake. Tests for prompt-shape behaviour stay at the `AgentRunner.run()` level (with a fake `ContainerRunner`), as the originating issue's testing decisions specify.
- The `Work phase` and `container runner` definitions in CONTEXT.md are updated to reflect that prompt rendering is no longer an internal step of `ContainerRunner.work()` — the runner receives a pre-rendered string and writes it to `/tmp/.pycastle_prompt`.
- ADR 0009 (`PromptRenderer` with scoped placeholder tiers) is unaffected in shape; only its sole caller moves from `ContainerRunner.work()` to `AgentRunner._build_prompt()`. ADR 0010's session-namespace semantics no longer need to be understood by `ContainerRunner`; namespace selection happens in the phase (via `RunRequest.session_namespace`) and namespace-aware prompt-shape resolution happens in `AgentRunner._build_prompt`.
